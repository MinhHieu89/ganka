# Phase 07.1: Clinical-Billing Integration - Context

**Gathered:** 2026-03-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Connect all department modules (Clinical, Pharmacy, Optical, Treatment) to the Billing module via real-time domain events so invoice line items auto-populate as services are performed during a visit. Includes a configurable service price catalog owned by Billing, SignalR push for cashier dashboard, and auto-invoice lifecycle management (create on visit, void on cancellation).

This phase does NOT include: new billing features, payment gateway integration, VIP pricing, or reporting. It wires up the existing billing infrastructure (Phase 7) to the existing clinical modules.

</domain>

<decisions>
## Implementation Decisions

### Charge Flow Trigger
- Real-time domain events: each source module publishes an integration event when a chargeable action occurs, Billing subscribes and auto-adds line items
- Follows the established Treatment→Pharmacy pattern (domain event → integration event → handler)
- Cashier can edit auto-populated charges (adjust prices, add misc charges, remove items) before finalizing — charges are not locked
- When a chargeable event fires and no invoice exists for that visit, auto-create a Draft invoice and add the line item
- OTC pharmacy sales follow the same event pattern — OTC sale event auto-creates a standalone invoice with nullable VisitId

### Invoice Creation Timing
- Invoice auto-created on visit creation — reception creates visit → VisitCreatedEvent → Billing creates Draft invoice + adds consultation fee line item from service catalog
- Visit cancellation auto-voids the invoice — VisitCancelledEvent → Billing sets invoice status to Voided
- SignalR pushes charge updates to cashier dashboard in real-time (cashier sees charges accumulate as services happen)
- SignalR scope: cashier dashboard only — other roles/pages do not receive real-time billing notifications

### Service Pricing Source
- Billing module owns a configurable service price catalog (ServiceCatalogItem entity) for clinical exam fees
- Admin UI page for full CRUD management of service catalog (add/edit/deactivate services, update prices)
- Service catalog covers: consultation (khám bệnh), follow-up (tái khám), and any other medical services the clinic defines
- Pharmacy prices come from DrugCatalogItem.SellingPrice (existing), Optical prices from Frame/Lens entities (existing), Treatment prices from TreatmentPackage pricing (existing)
- Price changes logged per FIN-09 (audit interceptor already captures field-level changes)

### Charge Coverage — All 4 Departments
- **Medical (Clinical):** Visit created → consultation fee only. Refraction, dry eye assessment, and imaging are included in the consultation fee — no separate charges
- **Pharmacy:** Drug dispensed → per-drug line items (drug name, quantity, unit price from DrugCatalogItem.SellingPrice). OTC sale → standalone invoice with per-item lines
- **Optical:** Glasses order created → frame + lens line items with prices from order. Combo packages use combo price
- **Treatment:** Session recorded → session fee from TreatmentPackage pricing. TreatmentSessionCompletedEvent already exists — extend with billing handler

### Integration Events to Create
- `VisitCreatedIntegrationEvent` — Clinical → Billing (create invoice + consultation line)
- `VisitCancelledIntegrationEvent` — Clinical → Billing (void invoice)
- `DrugDispensedIntegrationEvent` — Pharmacy → Billing (add per-drug line items)
- `OtcSaleCompletedIntegrationEvent` — Pharmacy → Billing (create standalone invoice)
- `GlassesOrderCreatedIntegrationEvent` — Optical → Billing (add frame/lens line items)
- `TreatmentSessionCompletedIntegrationEvent` — Treatment → Billing (add session line item, event already exists)

### Claude's Discretion
- Service catalog price tier approach (one price per service vs multiple tiers for new/follow-up — pick what's cleanest)
- SignalR hub implementation details (connection groups, message format)
- Invoice voiding implementation (soft delete vs status enum)
- How OTC sale invoice handles patient info (nullable PatientId or anonymous)
- Service catalog seeder data (default services and prices)
- Error handling for event processing failures (retry, dead letter, etc.)

</decisions>

<specifics>
## Specific Ideas

- Vietnamese clinic convention: invoices group charges by department (Khám bệnh, Dược phẩm, Kính, Điều trị)
- Consultation fee is the only medical charge — other clinical services (refraction, dry eye, imaging) are bundled into the visit fee
- The cashier should see charges appearing in real-time via SignalR as the patient progresses through the workflow stages
- OTC pharmacy sales (no visit) should feel identical to visit-based invoices from the cashier's perspective

</specifics>

## Existing Code Insights

### Reusable Assets
- **TreatmentSessionCompletedEvent + IntegrationEvent pattern**: Established in Treatment→Pharmacy, reuse same pattern for all billing integrations
- **PublishSessionCompletedIntegrationEventHandler**: Template for creating similar publish handlers in other modules
- **InvoiceLineItem.SourceId/SourceType**: Already exists — populate with source entity IDs and types
- **GetVisitChargesQuery contract**: Defined in Billing.Contracts but unimplemented — may be replaced by event-driven approach
- **QuestPDF infrastructure**: Already integrated for document generation
- **DataTable component**: Reuse for service catalog admin page
- **PermissionModule.Billing**: Already defined — add service catalog permissions

### Established Patterns
- **Domain event → Integration event → Handler**: Treatment module demonstrates this with Wolverine FX
- **Per-feature vertical slices**: Command/Handler with FluentValidation, Result<T> return
- **Cross-module contracts**: Via .Contracts project for DTOs and integration events
- **Audit interceptor**: Captures field-level changes — covers price change logging automatically
- **React Hook Form + Zod + TanStack Query**: Frontend form and data patterns

### Integration Points
- **Clinical module**: Publish VisitCreatedEvent and VisitCancelledEvent (may need to add these domain events)
- **Pharmacy module**: Publish DrugDispensedEvent and OtcSaleCompletedEvent from existing dispensing/OTC handlers
- **Optical module**: Publish GlassesOrderCreatedEvent from order creation handler
- **Treatment module**: Extend existing TreatmentSessionCompletedIntegrationEvent consumer in Billing
- **Billing module**: New event handlers that create/update invoices and add line items
- **SignalR**: New BillingHub for pushing charge updates to cashier dashboard
- **Frontend**: Cashier dashboard needs SignalR connection for real-time updates

</code_context>
