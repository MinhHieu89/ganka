# Phase 07.1: Clinical-Billing Integration - Research

**Researched:** 2026-03-13
**Domain:** Cross-module event-driven integration, SignalR real-time push, service catalog management
**Confidence:** HIGH

## Summary

Phase 07.1 connects four existing department modules (Clinical, Pharmacy, Optical, Treatment) to the Billing module via domain events that auto-populate invoice line items as services are performed. The codebase already has an established pattern for this: Treatment module publishes `TreatmentSessionCompletedEvent` (domain event), which a cascading Wolverine handler converts to `TreatmentSessionCompletedIntegrationEvent` (integration event) for cross-module consumption. This exact pattern must be replicated for Clinical, Pharmacy, and Optical modules.

Key infrastructure already exists: Invoice aggregate with `AddLineItem()` supporting SourceId/SourceType tracking, `Department` enum with all four categories (Medical, Pharmacy, Optical, Treatment), invoice `Void()` method, nullable `VisitId` for OTC sales, and Wolverine with SQL Server transactional outbox and durable local queues. What is missing: domain events in Clinical/Pharmacy/Optical modules, integration event contracts, billing event handlers, a ServiceCatalogItem entity, SignalR hub for real-time cashier updates, and visit cancellation in the Clinical module.

The project uses .NET 10, Wolverine FX for messaging/HTTP, EF Core with schema-per-module isolation, and a React 19 frontend with TanStack Router/Query, shadcn/ui, and i18next. No SignalR infrastructure exists yet -- both `@microsoft/signalr` (frontend) and `Microsoft.AspNetCore.SignalR` (backend, included in ASP.NET Core) need to be added.

**Primary recommendation:** Follow the Treatment module's proven domain-event-to-integration-event cascading handler pattern for all four department integrations, add a BillingHub for SignalR, and build the ServiceCatalogItem as a simple CRUD entity in the Billing domain.

### Pattern 1: Domain Event to Integration Event (Cascading Handler)
**What:** Source module raises domain event on aggregate, Wolverine auto-dispatches it, cascading handler converts to integration event
**When to use:** Every cross-module integration point
**Example:**
```csharp
// Step 1: Domain event (Clinical.Domain/Events/)
public sealed record VisitCreatedEvent(
    Guid VisitId,
    Guid PatientId,
    string PatientName,
    Guid DoctorId) : IDomainEvent
{
    public Guid EventId { get; } = Guid.NewGuid();
    public DateTime OccurredAt { get; } = DateTime.UtcNow;
}

// Step 2: Integration event (Clinical.Contracts/IntegrationEvents/)
public sealed record VisitCreatedIntegrationEvent(
    Guid VisitId,
    Guid PatientId,
    string PatientName);

// Step 3: Cascading handler in source module (Clinical.Application/)
public static class PublishVisitCreatedIntegrationEventHandler
{
    public static VisitCreatedIntegrationEvent Handle(
        VisitCreatedEvent domainEvent)
    {
        return new VisitCreatedIntegrationEvent(
            VisitId: domainEvent.VisitId,
            PatientId: domainEvent.PatientId,
            PatientName: domainEvent.PatientName);
    }
}

// Step 4: Consumer handler in Billing module (Billing.Application/)
public static class HandleVisitCreatedHandler
{
    public static async Task Handle(
        VisitCreatedIntegrationEvent @event,
        IInvoiceRepository invoiceRepository,
        IServiceCatalogRepository serviceCatalogRepository,
        IUnitOfWork unitOfWork,
        ICurrentUser currentUser,
        IBillingNotificationService notificationService,
        CancellationToken ct)
    {
        var consultationFee = await serviceCatalogRepository
            .GetActiveByCodeAsync("CONSULTATION", ct);

        var invoiceNumber = await invoiceRepository
            .GetNextInvoiceNumberAsync(DateTime.UtcNow.Year, ct);

        var invoice = Invoice.Create(
            invoiceNumber, @event.PatientId, @event.PatientName,
            @event.VisitId, new BranchId(currentUser.BranchId));

        if (consultationFee is not null)
        {
            invoice.AddLineItem(
                consultationFee.Name, consultationFee.NameVi,
                consultationFee.Price, 1, Department.Medical,
                @event.VisitId, "Visit");
        }

        invoiceRepository.Add(invoice);
        await unitOfWork.SaveChangesAsync(ct);

        // Push to cashier dashboard via SignalR
        await notificationService.NotifyInvoiceCreatedAsync(invoice.Id, ct);
    }
}
```

### Pattern 2: SignalR Hub with JWT Authentication
**What:** Authenticated SignalR hub with group-based messaging for cashier dashboard
**When to use:** Real-time push to cashier dashboard
**Example:**
```csharp
// Backend Hub
[Authorize]
public class BillingHub : Hub
{
    public async Task JoinCashierDashboard()
    {
        await Groups.AddToGroupAsync(Context.ConnectionId, "cashier-dashboard");
    }

    public async Task LeaveCashierDashboard()
    {
        await Groups.RemoveFromGroupAsync(Context.ConnectionId, "cashier-dashboard");
    }
}

// Notification service (called from event handlers)
public class BillingNotificationService : IBillingNotificationService
{
    private readonly IHubContext<BillingHub> _hubContext;

    public async Task NotifyLineItemAddedAsync(
        Guid invoiceId, string invoiceNumber, string description,
        decimal amount, CancellationToken ct)
    {
        await _hubContext.Clients.Group("cashier-dashboard")
            .SendAsync("LineItemAdded", new
            {
                InvoiceId = invoiceId,
                InvoiceNumber = invoiceNumber,
                Description = description,
                Amount = amount
            }, ct);
    }
}
```

```typescript
// Frontend hook
import { HubConnectionBuilder, LogLevel } from "@microsoft/signalr";

export function useBillingHub() {
  const queryClient = useQueryClient();

  useEffect(() => {
    const connection = new HubConnectionBuilder()
      .withUrl("/api/hubs/billing", {
        accessTokenFactory: () => getAccessToken(),
      })
      .withAutomaticReconnect()
      .configureLogging(LogLevel.Warning)
      .build();

    connection.on("LineItemAdded", (data) => {
      // Invalidate pending invoices query to refresh dashboard
      queryClient.invalidateQueries({ queryKey: ["pending-invoices"] });
    });

    connection.start().then(() => {
      connection.invoke("JoinCashierDashboard");
    });

    return () => { connection.stop(); };
  }, []);
}
```

### Pattern 3: Service Catalog Entity
**What:** Simple CRUD entity in Billing domain for configurable service prices
**When to use:** Clinical exam fee lookup during visit creation

**Recommendation (Claude's Discretion -- price tiers):** Use a single price per service with a `ServiceType` enum (`Consultation`, `FollowUp`, `Custom`). This is cleaner than multi-tier pricing. The service code uniquely identifies the service; the admin can create separate catalog entries for "Consultation" (150,000 VND) and "Follow-up" (100,000 VND) rather than encoding tiers into one entity.

```csharp
public class ServiceCatalogItem : AggregateRoot, IAuditable
{
    public string Code { get; private set; } = string.Empty;        // e.g., "CONSULTATION", "FOLLOWUP"
    public string Name { get; private set; } = string.Empty;        // English name
    public string NameVi { get; private set; } = string.Empty;      // Vietnamese name
    public decimal Price { get; private set; }                       // Price in VND
    public bool IsActive { get; private set; } = true;              // Soft deactivation
    public string? Description { get; private set; }                 // Optional notes
}
```

### Anti-Patterns to Avoid
- **Polling from Billing to source modules:** Do not use `GetVisitChargesQuery` request-response pattern. The event-driven approach is the locked decision. The `GetVisitChargesQuery` contract exists but should NOT be implemented.
- **Direct cross-module DbContext access:** Never query another module's database directly. Always use integration events or Wolverine message bus queries.
- **Fat integration events:** Keep integration events minimal -- only the data the consumer needs. Do not include the entire source entity.
- **Synchronous SignalR in event handlers:** Use fire-and-forget for SignalR notifications. If SignalR push fails, the invoice data is still correct in the database.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Message bus / event dispatch | Custom event dispatcher | Wolverine FX (already configured) | Transactional outbox, durable queues, retry built-in |
| WebSocket management | Custom WebSocket server | ASP.NET Core SignalR | Automatic reconnection, group management, protocol negotiation |
| Invoice number generation | Custom sequence | `IInvoiceRepository.GetNextInvoiceNumberAsync()` | Already exists with year-based sequencing |
| Audit logging for price changes | Custom price change logger | EF Core AuditInterceptor (already configured) | Automatic field-level change tracking per FIN-09 |
| Retry/dead-letter for failed events | Custom retry loop | Wolverine durable queues + error handling policies | Built into Wolverine's transactional outbox |

**Key insight:** Almost all infrastructure for this phase already exists. The work is wiring domain events in source modules and adding event handlers in Billing. Do not build new infrastructure.

## Common Pitfalls

### Pitfall 1: Visit Entity Has No Domain Events or Cancellation
**What goes wrong:** The Visit entity currently does NOT raise any domain events on creation, and has no `Cancel()` method. VisitStatus enum only has Draft/Signed/Amended.
**Why it happens:** Visit was built in Phase 3 before billing integration was planned.
**How to avoid:** Add `VisitCreatedEvent` in Visit.Create() factory, add `Cancel()` method with `VisitCancelledEvent`, add `Cancelled = 3` to VisitStatus enum.
**Warning signs:** If CreateVisitHandler returns without the integration event being published, no invoice will be created.

### Pitfall 2: Domain Events Not Dispatched Without Wolverine EF Core Integration
**What goes wrong:** Domain events added to aggregate roots via `AddDomainEvent()` may not be automatically dispatched by Wolverine.
**Why it happens:** The DbContexts do not use `IntegrateWithWolverine()`. Wolverine's `UseEntityFrameworkCoreTransactions()` middleware provides automatic domain event dispatch at the handler level, but only when Wolverine manages the transaction.
**How to avoid:** Verify that domain events from SaveChanges are being dispatched. The Treatment module's pattern works because Wolverine manages the handler pipeline. If domain events are not auto-dispatched, the cascading handler approach (return integration event from handler method) may need to be used directly instead.
**Warning signs:** Integration event handlers never fire despite domain events being added to aggregates.

### Pitfall 3: Race Condition on Invoice Creation
**What goes wrong:** Two events for the same visit arrive simultaneously (e.g., drug dispensed + glasses order), both try to create an invoice because neither sees the other's.
**Why it happens:** Concurrent event processing with Wolverine durable queues.
**How to avoid:** Use a "get-or-create" pattern: try to find existing invoice by VisitId first, create only if not found. Use database unique constraint on (VisitId, Status=Draft) or handle concurrency exception with retry.
**Warning signs:** Duplicate invoices for the same visit.

### Pitfall 4: SignalR Connection and JWT Token Expiry
**What goes wrong:** SignalR connections drop when JWT expires, cashier stops receiving real-time updates silently.
**Why it happens:** SignalR WebSocket connections are long-lived but JWT tokens are short-lived.
**How to avoid:** Use `withAutomaticReconnect()` on the client. On reconnection, re-join the "cashier-dashboard" group. Implement `accessTokenFactory` that returns a fresh token.
**Warning signs:** Cashier sees stale data after being idle for a while.

### Pitfall 5: OTC Sale Invoice Without Patient
**What goes wrong:** OTC sales may have nullable PatientId, but Invoice.Create requires PatientId (non-nullable Guid).
**Why it happens:** OtcSale.PatientId is nullable (anonymous walk-in customers).

**Recommendation (Claude's Discretion):** Use nullable PatientId on Invoice for OTC sales. The Invoice entity already has `Guid? VisitId` -- add similar pattern. For anonymous sales, use PatientId = null and CustomerName from OtcSale.CustomerName as PatientName.
**How to avoid:** Modify Invoice to accept nullable PatientId, or always require PatientId (register walk-in as PAT-02 lightweight record).

### Pitfall 6: Migration Ordering Across Modules
**What goes wrong:** Adding new entities (ServiceCatalogItem) or modifying existing ones (Visit adding Cancelled status) requires migrations in multiple modules.
**Why it happens:** Schema-per-module isolation means each module has its own migration timeline.
**How to avoid:** Run migrations for each module independently. Order: Clinical first (new events need domain changes), then Billing (new ServiceCatalogItem + handlers).

## Claude's Discretion Recommendations

### Invoice Voiding Implementation
**Recommendation:** Use the existing `InvoiceStatus.Voided` enum value and `Invoice.Void()` method. Already implemented. No changes needed.

### OTC Sale Patient Handling
**Recommendation:** Allow nullable PatientId on Invoice. OTC sales with PatientId = null use CustomerName as display name. This matches the existing OtcSale entity pattern where PatientId is nullable.

### Service Catalog Seeder Data
**Recommendation:** Seed two default services:
- Code: "CONSULTATION", Name: "Consultation", NameVi: "Kham benh", Price: 150000 (VND)
- Code: "FOLLOWUP", Name: "Follow-up Visit", NameVi: "Tai kham", Price: 100000 (VND)

Admin can add more via the CRUD UI.

### Error Handling for Event Processing Failures
**Recommendation:** Rely on Wolverine's built-in retry and dead-letter queue. Wolverine durable local queues already configured with `opts.Policies.UseDurableLocalQueues()`. Configure retry policy for billing handlers: 3 retries with exponential backoff. Failed messages go to Wolverine's dead-letter queue for manual inspection. Log failures with Serilog.

### SignalR Hub Implementation
**Recommendation:** Single hub (`BillingHub`) with group "cashier-dashboard". Messages:
- `InvoiceCreated` -- new invoice created for a visit
- `LineItemAdded` -- charge added to existing invoice
- `InvoiceVoided` -- invoice voided due to visit cancellation

The cashier dashboard page connects on mount, joins group, disconnects on unmount. Use TanStack Query invalidation on SignalR messages rather than direct state updates (simpler, leverages existing data fetching).

## Open Questions

1. **Wolverine Domain Event Auto-Dispatch Mechanism**
   - What we know: Treatment module uses cascading handler (domain event in -> integration event out). Wolverine's `UseEntityFrameworkCoreTransactions()` is configured.
   - What's unclear: Whether domain events on aggregates are auto-dispatched by Wolverine on SaveChanges, or if the cascading handler pattern relies on Wolverine HTTP handler pipeline specifically. The DbContexts do NOT call `IntegrateWithWolverine()`.
   - Recommendation: Test with Treatment module first to verify the mechanism. If domain events from `AddDomainEvent()` are not auto-dispatched, use explicit `IMessageBus.PublishAsync()` in the handler after SaveChanges.

2. **Invoice PatientId Nullability**
   - What we know: Invoice currently has `Guid PatientId` (non-nullable). OTC sales need nullable PatientId.
   - What's unclear: Whether changing PatientId to nullable breaks existing invoice queries/reports.
   - Recommendation: Add migration to make PatientId nullable. Review existing queries for null-safety.

3. **Visit Cancellation Workflow**
   - What we know: Visit has no Cancel method or Cancelled status. VisitStatus enum is Draft/Signed/Amended.
   - What's unclear: Business rules for when a visit can be cancelled (only Draft? Signed too?).
   - Recommendation: Allow cancellation of Draft visits only. Signed visits cannot be cancelled (medical record integrity). Add `Cancelled = 3` to VisitStatus.
