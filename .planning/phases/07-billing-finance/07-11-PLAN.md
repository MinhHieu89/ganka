  <behavior>
    ApplyDiscount:
    - Given Draft invoice, DiscountType=Percentage, Value=10 -> creates pending discount with CalculatedAmount = 10% of subtotal
    - Given Draft invoice, DiscountType=FixedAmount, Value=50000 -> creates pending discount with CalculatedAmount = 50000
    - Given line item ID -> applies discount to specific line item
    - Given Finalized invoice -> returns error

    ApproveDiscount:
    - Given valid manager PIN hash match -> approves discount, recalculates invoice totals
    - Given invalid PIN -> returns error "Invalid manager PIN"
    - Given already approved discount -> returns error

    RequestRefund:
    - Given Finalized invoice, amount, reason -> creates refund request in Requested status
    - Given Draft invoice -> returns error (can only refund finalized)
    - Given amount exceeding invoice total -> returns error
  </behavior>
  <implementation>
    Write tests first, then implement. Manager PIN verification queries Auth module via Contracts (or accepts pre-validated PIN hash).
    NOTE: RejectDiscount, ApproveRefund, and ProcessRefund handlers are created in Plan 26 (supplementary discount/refund handlers).
  </implementation>
</feature>

    1. **ApplyDiscount.cs**:
       - ApplyDiscountCommand: sealed record(Guid InvoiceId, Guid? InvoiceLineItemId, int DiscountType, decimal Value, string Reason)
       - Validator: Value > 0, Reason NotEmpty, if Percentage then Value <= 100
       - Handler: Load invoice, create Discount entity, calculate CalculatedAmount based on type (percentage of SubTotal or LineTotal, or fixed amount). Discount starts as Pending. Save. Return Result<DiscountDto>.

    2. **ApproveDiscount.cs**:
       - ApproveDiscountCommand: sealed record(Guid InvoiceId, Guid DiscountId, string ManagerPin)
       - Handler: Load invoice, find discount by DiscountId, verify manager PIN by querying Auth module (IMessageBus.InvokeAsync with a VerifyManagerPinQuery). On valid PIN: call discount.Approve(managerId), recalculate invoice totals (DiscountTotal updates). Save. Return Result.
       - Note: For PIN verification, add VerifyManagerPinQuery to Auth.Contracts if it doesn't exist, or use a simpler approach where the handler receives the manager's userId and validates permissions directly.

    3. **RequestRefund.cs**:
       - RequestRefundCommand: sealed record(Guid InvoiceId, Guid? InvoiceLineItemId, decimal Amount, string Reason)
       - Validator: Amount > 0, Reason NotEmpty
       - Handler: Validate invoice is Finalized. Validate Amount <= TotalAmount (or line item total for partial). Create Refund entity in Requested status. Save. Return Result<RefundDto>.

    Add Billing.Domain reference to Billing.Application.csproj if not present. Add WolverineFx reference for IMessageBus cross-module queries.
