## Summary

This phase creates Vietnamese user stories documentation for all features implemented in Phases 1, 2, and 3 (main phases only). The output is a set of markdown files in `docs/user-stories/` organized by workflow area, using standard user story format in Vietnamese. This is a pure documentation phase -- no code changes, no tests, no infrastructure.

The completed requirements span 5 categories: Authentication & Security (AUTH-01 through AUTH-05), Patient Management (PAT-01 through PAT-05), Appointment Scheduling (SCH-01 through SCH-06), Clinical Workflow (CLN-01 through CLN-04), Refraction & Examination (REF-01 through REF-03), Diagnosis (DX-01, DX-02), Audit & Compliance (AUD-01 through AUD-04), User Interface (UI-01, UI-02), and Architecture (ARC-01 through ARC-06). Total: 35 completed requirements to document as user stories.

**Primary recommendation:** Organize into 5-6 workflow files grouped by how clinic staff actually uses the system, with each story traced back to requirement IDs from REQUIREMENTS.md, written in formal Vietnamese with proper diacritics and technical terms preserved in their original form.

### Story Format Template

Each file should follow this structure:

```markdown
# [Workflow Name] - User Stories

**Pham vi:** [Brief description of what this workflow covers]
**Yeu cau lien quan:** [Requirement IDs]
**So luong user stories:** [Count]

---

## US-[PREFIX]-001: [Story Title in Vietnamese]

**La mot** [vai tro], **Toi muon** [hanh dong], **De** [loi ich].

**Yeu cau lien quan:** [REQ-ID]

### Tieu chi chap nhan

**Luong chinh (Happy Path):**
1. Nguoi dung [action] -> He thong [response]
2. Nguoi dung [action] -> He thong [response]
3. ...

**Truong hop ngoai le:**
1. Neu [condition] -> He thong [response]
2. Neu [condition] -> He thong [response]

**Truong hop loi:**
1. Khi [error condition] -> He thong [error response]

### Ghi chu ky thuat
- [Technical detail relevant for dev/QA, e.g., API endpoint, field names, validation rules]
```

**Important:** The above template shows unaccented Vietnamese for markdown readability in this research doc. The actual user story files MUST use proper Vietnamese diacritics throughout (per CONTEXT.md locked decision and Phase 01.2-07 decision).

### Anti-Patterns to Avoid
- **Module-based organization:** Do NOT organize files by backend module (Auth, Patient, Clinical). Organize by workflow as staff experiences it.
- **Phase-based organization:** Do NOT create one file per phase. Group by workflow area.
- **English story text:** Do NOT write stories in English then translate. Write directly in Vietnamese.
- **Missing diacritics:** Do NOT use unaccented Vietnamese (e.g., "Nguoi dung" instead of "Nguoi dung" with proper marks). This was a Phase 01.2-07 decision -- proper diacritics are mandatory.
- **Overly abstract stories:** Do NOT write stories that don't reference the actual system behavior. Stories should mirror the implementation -- reference real field names, real UI elements.
- **Missing error scenarios:** Do NOT only document happy paths. Each story should include edge cases and error handling.

## Common Pitfalls

### Pitfall 1: Incomplete Requirement Coverage
**What goes wrong:** Some completed requirements get missed, leaving gaps in documentation.
**Why it happens:** Requirements are spread across 9 categories; easy to miss one.
**How to avoid:** Use the REQUIREMENTS.md traceability table to systematically check off each completed requirement (35 total from Phases 1, 2, 3). Every `[x]` requirement must have at least one user story.
**Warning signs:** Story count seems too low; requirement IDs missing from any story file.

### Pitfall 2: Incorrect Vietnamese Diacritics
**What goes wrong:** Vietnamese text has wrong or missing diacritical marks.
**Why it happens:** Automated generation or copy-paste can lose diacritics.
**How to avoid:** Use the existing vi/*.json files as reference for all UI-related terms. Verify diacritics on key medical/clinical terms.
**Warning signs:** Text looks "flat" without accent marks; words like "benh nhan" instead of "benh nhan" (with proper marks).

### Pitfall 3: Stories Describe Planned Features, Not Implemented Ones
**What goes wrong:** Stories describe behavior that hasn't been implemented yet (e.g., features from Phase 4+).
**Why it happens:** REQUIREMENTS.md lists all v1 requirements; easy to accidentally include unchecked ones.
**How to avoid:** Only include requirements marked `[x]` in REQUIREMENTS.md. Cross-reference with ROADMAP.md phase completion status.
**Warning signs:** Stories reference DRY-*, IMG-*, RX-*, TRT-*, PHR-*, OPT-*, FIN-*, PRT-* requirements (all Phase 4+, not yet implemented).

### Pitfall 4: Inconsistent Story ID Scheme
**What goes wrong:** Story IDs don't follow a consistent pattern, making traceability difficult.
**Why it happens:** Different files use different numbering conventions.
**How to avoid:** Establish clear prefix convention mapped to workflow files:
- US-AUTH-NNN for authentication
- US-PAT-NNN for patient management
- US-SCH-NNN for scheduling
- US-CLN-NNN for clinical workflow
- US-AUD-NNN for audit
- US-SYS-NNN for UI/system
**Warning signs:** Duplicate IDs, inconsistent prefixes, gaps in numbering.

### Pitfall 5: Too Technical or Too Abstract
**What goes wrong:** Stories either read like API specs (too technical) or business requirements (too abstract).
**Why it happens:** Audience is dev team + QA, requiring a specific balance.
**How to avoid:** Story description should be business-level ("La mot bac si, toi muon..."). Acceptance criteria should mix behavior description with technical details (field names, API references, validation rules).
**Warning signs:** No mention of specific fields/forms in acceptance criteria; or conversely, stories that read like Swagger docs.

## Code Examples

### Example: Vietnamese User Story (fully formed)

```markdown
## US-AUTH-001: Dang nhap he thong

**La mot** nhan vien phong kham, **Toi muon** dang nhap vao he thong bang email va mat khau, **De** truy cap cac chuc nang theo vai tro cua toi.

**Yeu cau lien quan:** AUTH-01

### Tieu chi chap nhan

**Luong chinh (Happy Path):**
1. Nguoi dung truy cap trang dang nhap (/login) -> He thong hien thi form voi cac truong: Email, Mat khau, Ghi nho dang nhap
2. Nguoi dung nhap email va mat khau hop le -> He thong xac thuc thanh cong
3. He thong tao JWT token chua role-based claims -> Chuyen huong den Dashboard
4. Neu chon "Ghi nho dang nhap" -> Phien dang nhap duoc luu qua HTTP-only cookie voi thoi han dai hon

**Truong hop loi:**
1. Khi nhap sai email hoac mat khau -> He thong hien thi thong bao "Ten dang nhap hoac mat khau khong dung"
2. Khi tai khoan bi vo hieu hoa -> He thong tu choi dang nhap

### Ghi chu ky thuat
- JWT token chua: userId, email, roles[], permissions[]
- Refresh token luu trong HTTP-only cookie (khong luu trong JavaScript)
- API endpoint: POST /api/auth/login
```

**Note:** This example shows the structure. Actual stories MUST have proper Vietnamese diacritics on every word.

## Open Questions

1. **Story count per file**
   - What we know: 35 requirements to cover, estimated 30-40 stories total
   - What's unclear: Exact 1:1 vs 1:many mapping between requirements and stories
   - Recommendation: Allow 1:many where a requirement is naturally experienced as multiple user actions (e.g., SCH-02 self-booking has submit + status check). Allow many:1 where requirements are closely related (e.g., REF-01 + REF-02 + REF-03 can combine into 1-2 stories about refraction recording)

2. **Architecture requirements (ARC-01 through ARC-06) as user stories**
   - What we know: These are internal/infrastructure requirements, not user-facing workflows
   - What's unclear: How to write meaningful user stories for "ACL adapter pattern" or "Azure SQL backup"
   - Recommendation: Frame as system-level stories from the Admin/Manager perspective: "La mot quan tri vien, toi muon he thong tu dong sao luu du lieu hang ngay, de dam bao khong mat du lieu khi co su co." Keep acceptance criteria technical since the audience is dev/QA.

3. **Decimal phase features**
   - What we know: Context says skip decimal phases (1.1, 1.2, 2.1)
   - What's unclear: Some decimal phases fixed bugs or changed behavior that affects how features work now
   - Recommendation: Document the **current behavior** (which includes all fixes), not the behavior as originally shipped in integer phases. The "skip decimal phases" instruction means don't create separate stories for refactoring work -- but do describe the system as it currently works.
