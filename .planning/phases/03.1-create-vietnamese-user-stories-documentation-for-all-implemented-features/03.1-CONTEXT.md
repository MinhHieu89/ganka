# Phase 03.1: Create Vietnamese User Stories Documentation - Context

**Gathered:** 2026-03-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Create comprehensive Vietnamese user stories documentation in BDD format for all user-facing features implemented in Phases 1, 2, and 3. Documentation organized by workflow, stored in `docs/user-stories/`, targeting dev team and QA audience.

</domain>

<decisions>
## Implementation Decisions

### Document Structure
- One file per workflow area in `docs/user-stories/`
- Organized by role/workflow (how staff actually use the system), not by module or phase
- Workflow files: e.g., authentication, patient-registration, scheduling, clinical-examination, etc.

### Story Format
- BDD scenario format: Cho rằng / Khi / Thì (Vietnamese keywords)
- Comprehensive coverage: happy path + edge cases + error scenarios for each feature
- Mix of business-level story descriptions with technical details in acceptance criteria
- Each scenario gets a unique ID for traceability (e.g., US-AUTH-001, US-PAT-003)

### Coverage Scope
- Cover Phases 1, 2, 3 (main phases only — skip decimal phases 1.1, 1.2, 2.1 which are internal refactoring/bug fixes)
- Include admin features (user management, role configuration, audit logs)
- Include public-facing features (patient self-booking page, booking status check)

### Language & Tone
- Fully Vietnamese with proper diacritics throughout
- Formal/professional tone ("Người dùng nhập thông tin bệnh nhân...")
- Technical terms kept as-is (ICD-10, SPH, CYL, AXIS, ADD, PD, VA, IOP, JWT, etc.)
- BDD keywords in Vietnamese: Cho rằng / Khi / Thì (not Given/When/Then)

### Claude's Discretion
- Exact workflow file breakdown (how many files, what groupings)
- Scenario numbering scheme within each file
- Level of technical detail in acceptance criteria
- How to handle cross-workflow features (e.g., allergy alerts span patient + clinical)

</decisions>
