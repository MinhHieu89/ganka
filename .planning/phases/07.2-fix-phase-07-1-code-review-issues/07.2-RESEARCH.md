### Pattern 1: Wolverine Static Handler (existing pattern)
**What:** All handlers are static classes with a static `Handle` method. Wolverine resolves dependencies via method injection.
**When to use:** All command/query/event handlers.
**Example:**
```csharp
public static class HandleSomethingHandler
{
    public static async Task Handle(
        SomeEvent @event,
        ISomeRepository repository,
        IUnitOfWork unitOfWork,
        ILogger logger,
        CancellationToken ct)
    {
        // handler logic
    }
}
```

### Pattern 2: Domain Event -> Integration Event Cascading (existing pattern)
**What:** Domain events stay internal to a module. A cascading handler maps them to integration events in the Contracts layer for cross-module consumption.
**Example:**
```csharp

{
    public static TreatmentSessionCompletedIntegrationEvent Handle(
        TreatmentSessionCompletedEvent domainEvent)
    {

            /* map fields */);
    }

```

### Pattern 3: Get-or-Create Invoice Pattern (existing pattern)
**What:** Billing handlers check for existing invoice by VisitId, create if missing.
**Idempotency extension:** Before adding line items, check for existing items with same SourceId + SourceType.

### Anti-Patterns to Avoid
- **Creating new DI scope in singleton interceptor:** The current DomainEventDispatcherInterceptor creates `_serviceProvider.CreateScope()` which bypasses the ambient Wolverine scope and its transactional outbox. Fix: resolve IMessageBus from the DbContext's service provider or restructure to use Wolverine's built-in event publishing.
- **Checking only one status in guard clauses:** HandleVisitCancelled only checks `!= Voided` but should guard against all non-Draft statuses.
- **Using `as string` casts on typed router props:** Bypasses TanStack Router type safety.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Invoice number uniqueness | Application-level locking | SQL SEQUENCE or SELECT WITH (UPDLOCK, HOLDLOCK) + unique index | Race conditions impossible to solve at app layer alone |
| Endpoint authorization | Custom middleware | ASP.NET Core `RequireAuthorization("PolicyName")` with `AddAuthorizationBuilder()` | Battle-tested, declarative, auditable |
| SignalR connection lifecycle | Manual reconnect logic | `withAutomaticReconnect()` (already used) + stable useRef pattern | Framework handles reconnection; just need stable React references |
| i18n in Zod schemas | Inline translation | Schema factory function `createSchema(t)` that takes translation function | Keeps schemas pure, translations swappable |

## Common Pitfalls

### Pitfall 1: DomainEventDispatcherInterceptor Scope Isolation
**What goes wrong:** The singleton interceptor creates a new DI scope to resolve IMessageBus. This new scope is NOT the same scope Wolverine uses, so published messages bypass the transactional outbox and could be sent even if the outer transaction rolls back.
**Why it happens:** EF Core interceptors are typically registered as singletons and don't have access to the request-scoped container.
**How to avoid:** Two approaches:
1. Resolve IMessageBus from the DbContext's internal service provider (if available via `context.GetInfrastructure<IServiceProvider>()`)
2. Register the interceptor as scoped instead of singleton and inject IMessageBus directly
3. Use Wolverine's `MessageContext` which is already scoped to the handler pipeline

**Recommended approach:** Change to scoped registration and inject IMessageBus via constructor. EF Core supports scoped interceptors when using `AddDbContext` with the interceptor registered in DI.
**Warning signs:** Messages published during a failed transaction still being processed.

### Pitfall 2: Invoice Number Race Condition Under Load
**What goes wrong:** Two concurrent handlers call `GetNextInvoiceNumberAsync` simultaneously, both get the same "next" number, both create invoices with duplicate numbers.
**Why it happens:** No database-level serialization on the read-then-increment operation.
**How to avoid:** Use SQL SEQUENCE (`CREATE SEQUENCE InvoiceNumberSeq`) which is atomic, or use `SELECT MAX() ... WITH (UPDLOCK, HOLDLOCK)` in a transaction. Always add a unique index on InvoiceNumber as a safety net.
**Warning signs:** Unique constraint violations in production under concurrent billing events.

### Pitfall 3: EF Core Migration for Nullable PatientId
**What goes wrong:** Changing `Invoice.PatientId` from `Guid` to `Guid?` requires an EF Core migration. Existing data has `Guid.Empty` for anonymous patients which needs to be migrated to NULL.
**Why it happens:** Schema change with data migration.
**How to avoid:** Create migration with data fixup: `UPDATE Invoices SET PatientId = NULL WHERE PatientId = '00000000-0000-0000-0000-000000000000'`.
**Warning signs:** Foreign key violations or incorrect query results after migration.

### Pitfall 4: SignalR useEffect Dependency Array
**What goes wrong:** Including callback functions (invalidatePendingInvoices, invalidateInvoice) in the useEffect dependency array causes the connection to be rebuilt on every render where queryClient changes identity.
**Why it happens:** React recreates callback references when dependencies change, triggering the effect.
**How to avoid:** Use useRef for the queryClient and callbacks. The connection effect should have an empty dependency array `[]`. Access current callbacks via refs inside the event handlers.
**Warning signs:** Multiple SignalR connections in browser DevTools, frequent reconnections.

### Pitfall 5: Zod Schema Factory and Translation Timing
**What goes wrong:** If the Zod schema is defined at module scope (outside component), the `t()` function isn't available. If defined inside the component, it recreates on every render.
**How to avoid:** Use a factory function `const createSchema = (t: TFunction) => z.object({...})` called inside the component with `useMemo` keyed on `t`.

### Fix 2: Invoice Number with SQL SEQUENCE
```csharp
// Migration: create sequence
migrationBuilder.Sql("CREATE SEQUENCE InvoiceNumberSeq START WITH 1 INCREMENT BY 1;");
migrationBuilder.CreateIndex(
    name: "IX_Invoices_InvoiceNumber",
    table: "Invoices",
    column: "InvoiceNumber",
    unique: true);

// Repository method
public async Task<string> GetNextInvoiceNumberAsync(int year, CancellationToken ct)
{
    var nextVal = await context.Database
        .SqlQuery<long>($"SELECT NEXT VALUE FOR InvoiceNumberSeq AS Value")
        .FirstAsync(ct);
    return $"HD-{year}-{nextVal:D5}";
}
```
**Note:** SQL SEQUENCE resets annually; consider a year-prefixed sequence or handle year rollover logic. Alternative: use `UPDLOCK, HOLDLOCK` with existing MAX pattern if SEQUENCE lifecycle is too complex.

## Open Questions

1. **SQL SEQUENCE vs UPDLOCK approach for invoice numbers**
   - What we know: Both are valid. SEQUENCE is simpler and lock-free. UPDLOCK works with the existing MAX pattern.
   - What's unclear: Whether the invoice number needs to reset annually. If yes, SEQUENCE requires annual management (or a new sequence per year).
   - Recommendation: Use SEQUENCE for simplicity. If annual reset needed, create year-scoped sequences or use a counter table with UPDLOCK.

2. **Invoice.PatientId nullable migration data fixup**
   - What we know: Changing Guid to Guid? requires migration. Existing rows with Guid.Empty need to become NULL.
   - What's unclear: Whether any queries or UI depend on PatientId being non-null for filtering.
   - Recommendation: Add migration with data fixup. Search codebase for `invoice.PatientId` comparisons that might break.

3. **Authorization policy registration**
   - What we know: The project uses `RequireAuthorization()` on groups but no named policies are registered. The auth module has a permission-based system.
   - What's unclear: Whether to use role claims, permission claims, or a custom policy.
   - Recommendation: Check how the existing auth module handles permissions (likely claim-based). Add a policy that checks for Admin or Manager role/permission claim.
