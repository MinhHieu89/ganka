# Phase 01.1: Change the current code structure of the backend - Context

**Gathered:** 2026-02-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Restructure the backend Application layer from type-based organization (Endpoints/, Validators/, Services/) to vertical slice feature-based organization (Features/). Move business logic from Infrastructure services into Application handlers. Add unit tests for all handlers. Applies to all existing modules (Auth, Audit) and sets the pattern for future modules.

</domain>

<decisions>
## Implementation Decisions

### Feature file layout
- Organize Application layer by feature, not by type
- Each feature is a single `.cs` file under `Features/` folder: e.g., `Auth.Application/Features/Login.cs`
- Flat structure — `Features/Login.cs`, NOT `Features/Login/Login.cs`
- Each feature file contains: endpoint handler class, FluentValidation validator class, and request/response records
- No more separate `Endpoints/`, `Validators/` folders

### Business logic placement
- Handlers (endpoints) in Application layer own the business logic
- Current pattern of endpoints delegating to service interfaces (implemented in Infrastructure) is eliminated
- Handlers inject thin repository interfaces defined in Application, implementations stay in Infrastructure
- Repositories are pure data access — no business logic in them
- Specialized infrastructure services (password hashing, JWT generation) remain as injected services where appropriate

### Contracts layer
- Keep Contracts as a separate project per module — unchanged
- Contracts hold DTOs for cross-module communication
- Feature files reference Contracts DTOs but don't define cross-module types

### Unit tests
- Add unit tests for every handler/endpoint
- Unit test style: mock repository interfaces, test handler logic directly (no database)
- Integration tests deferred to later — unit tests only for this phase
- Test project per module pattern (e.g., `Auth.Unit.Tests/`)

### Claude's Discretion
- Exact test file naming convention and folder structure within test projects
- How to handle edge cases where a handler is truly thin (simple CRUD pass-through)
- Whether to keep or remove the `Marker.cs` assembly marker pattern

</decisions>

<specifics>
## Specific Ideas

- Feature file example: `Login.cs` contains `LoginEndpoint` class, `LoginCommandValidator` class, and `LoginCommand`/`LoginResponse` records all in one file
- Pattern inspired by Wolverine vertical slice — colocate everything related to one operation
- Existing Auth module has: Login, Logout, RefreshToken, UserEndpoints, RoleEndpoints, PermissionEndpoints — all should be restructured

</specifics>

<deferred>
## Deferred Ideas

- Integration tests with Testcontainers — add in a future phase
- Applying same restructuring to frontend — separate concern

</deferred>
