# Phase 01.1: Change the current code structure of the backend - Context

**Gathered:** 2026-02-28
**Updated:** 2026-03-01
**Status:** Ready for planning (addendum plan)

<domain>
## Phase Boundary

Restructure the backend from type-based organization (Endpoints/, Validators/, Services/) to vertical slice feature-based organization. Separate concerns into proper layers: Presentation (Minimal API endpoints), Application (handlers + validators + records), Infrastructure (repositories + services). Replace the god repository with repository-per-aggregate-root and add Unit of Work pattern. Add unit tests for all handlers. Re-verify all Phase 1 checks after refactoring. Applies to all existing modules (Auth, Audit) and sets the pattern for future modules.

**Addendum scope:** Clean up Presentation layer duplication, add shared extensions, introduce route groups, centralize DI registration, and adopt Central Package Management.

</domain>

<decisions>
## Implementation Decisions

### Layer separation — Application must not depend on HTTP
- Application layer contains ONLY: handler classes, FluentValidation validators, request/response records
- No HTTP concerns (routing, status codes, Wolverine.Http attributes) in Application layer
- Endpoints move to a new Presentation layer project per module (e.g., `Auth.Presentation/`)
- Presentation layer uses Minimal API — NOT Wolverine.Http for endpoint definitions
- Each module exposes one extension method: e.g., `MapAuthApiEndpoints()` used in Bootstrap's `Program.cs`
- Minimal API endpoints dispatch to Application handlers via Wolverine mediator (not MediatR)

### Feature file layout (Application layer)
- Organize Application layer by feature, not by type
- Each feature is a single `.cs` file under `Features/` folder: e.g., `Auth.Application/Features/Login.cs`
- Flat structure — `Features/Login.cs`, NOT `Features/Login/Login.cs`
- Each feature file contains: handler class, FluentValidation validator class, and request/response records
- No more separate `Endpoints/`, `Validators/` folders in Application

### Repository-per-aggregate-root (no god repository)
- Each aggregate root gets its own repository interface defined in Application layer
- NO single `IAuthRepository` that handles everything — that's a god repository anti-pattern
- Claude determines which entities are aggregate roots based on the domain model
- Repository implementations stay in Infrastructure layer
- Repositories are pure data access — no business logic, no SaveChanges

### Unit of Work pattern
- Explicit `IUnitOfWork` interface injected into handlers
- Handlers call `IUnitOfWork.SaveChangesAsync()` to commit transaction boundaries
- Repositories do NOT have SaveChanges methods — only the Unit of Work owns persistence
- `IUnitOfWork` defined in Application, implemented in Infrastructure (wraps DbContext)

### Business logic placement
- Handlers in Application layer own the business logic
- Current pattern of endpoints delegating to service interfaces (implemented in Infrastructure) is eliminated
- Specialized infrastructure services (password hashing, JWT generation) remain as injected services

### Contracts layer
- Keep Contracts as a separate project per module — unchanged
- Contracts hold DTOs for cross-module communication
- Feature files reference Contracts DTOs but don't define cross-module types

### Unit tests
- Add unit tests for every handler
- Unit test style: mock repository interfaces + IUnitOfWork, test handler logic directly (no database)
- Integration tests deferred to later — unit tests only for this phase
- Test project per module pattern (e.g., `Auth.Unit.Tests/`)

### Phase 1 re-verification
- Dedicated final plan that re-runs ALL Phase 1 verification steps
- Ensures the restructuring didn't break any existing functionality
- Clear pass/fail gate before phase is considered complete

### Presentation layer cleanup (ADDENDUM)
- **Result → IResult mapping:** Create `ResultExtensions` with `ToHttpResult()` and `ToHttpResult<T>()` extension methods to eliminate duplicated error mapping across all endpoints
- **HttpContext helpers:** Create `HttpContextExtensions` with `TryGetUserId()` to eliminate repeated ClaimTypes.NameIdentifier parsing
- **Route groups:** Use `app.MapGroup("/api/auth")` and `app.MapGroup("/api/admin")` with group-level `.RequireAuthorization()` instead of flat route definitions
- **Endpoint pattern:** Each endpoint does ONLY: route definition → call bus → return `result.ToHttpResult()`. No inline error mapping, no inline claims parsing
- **Shared extensions project:** `Shared.Presentation` project referenced by all `*.Presentation` projects — single source of truth for `ResultExtensions`, `HttpContextExtensions`

### DI registration pattern (ADDENDUM)
- Each layer has its own `IoC.cs` file exposing an extension method for DI registration
- `Auth.Application/IoC.cs` → registers handlers, validators
- `Auth.Infrastructure/IoC.cs` → registers repositories, services
- `Auth.Presentation/IoC.cs` → registers presentation-layer services (if any)
- Bootstrapper's `Program.cs` calls each module's extension methods
- Same pattern for Audit module and all future modules

### Central Package Management (ADDENDUM)
- Create `Directory.Packages.props` at solution root
- Migrate ALL existing NuGet package references across ALL .csproj files
- All .csproj files use `<PackageReference Include="..." />` without `Version` attribute
- Version centralized in `Directory.Packages.props` with `<PackageVersion Include="..." Version="..." />`

### Claude's Discretion
- Aggregate root boundaries in Auth module (which entities are aggregate roots)
- Exact test file naming convention and folder structure within test projects
- How to handle edge cases where a handler is truly thin (simple CRUD pass-through)
- Whether to keep or remove the `Marker.cs` assembly marker pattern
- Presentation layer endpoint file organization within each module
- Error code → HTTP status mapping strategy (generic vs extensible) based on current error code usage

</decisions>

<specifics>
## Specific Ideas

- Feature file example: `Login.cs` contains `LoginHandler` class, `LoginCommandValidator` class, and `LoginCommand`/`LoginResponse` records all in one file
- Presentation endpoint example: `Auth.Presentation/AuthApiEndpoints.cs` with `MapAuthApiEndpoints()` extension method that maps Minimal API routes dispatching to Wolverine handlers
- Pattern inspired by vertical slice architecture — colocate everything related to one operation in Application, expose via thin Minimal API endpoints in Presentation
- Existing Auth module has: Login, Logout, RefreshToken, UserEndpoints, RoleEndpoints, PermissionEndpoints — all should be restructured
- Refactored endpoint example (logout): `authGroup.MapPost("/logout", async (IMessageBus bus, HttpContext context) => { if (!context.TryGetUserId(out var userId)) return Results.Unauthorized(); var result = await bus.InvokeAsync<Result>(new LogoutCommand(userId)); return result.ToHttpResult(); }).RequireAuthorization();`

</specifics>
