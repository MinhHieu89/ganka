<objective>
Migrate the 5 authentication-flow features (Login, Logout, RefreshToken, GetCurrentUser, UpdateLanguage) from old Wolverine.Http endpoints + AuthService to vertical slice feature files with Minimal API endpoints. Move all business logic from Infrastructure's AuthService into Application handlers.

Purpose: Eliminate the god service pattern (AuthService) for auth operations. Each handler becomes self-contained with its own command, validator, and logic. This is the core of the vertical slice restructuring for the authentication half of the Auth module.

Output: 5 feature files in Auth.Application/Features/, auth-flow routes in Auth.Presentation, old auth-flow endpoints and AuthService deleted.
</objective>

Read existing endpoint files and AuthService to extract the EXACT business logic for each feature. Each feature file follows the pattern from RESEARCH.md Pattern 1.

**Login.cs:**
- `LoginCommand(string Email, string Password, bool RememberMe = false, string? IpAddress = null)` record
- `LoginCommandValidator : AbstractValidator<LoginCommand>` -- move validation from `Validators/LoginCommandValidator.cs` (Email not empty + valid format, Password not empty + min length)
- `LoginHandler` injecting `IUserRepository`, `IRefreshTokenRepository`, `IPasswordHasher`, `IJwtService`, `IUnitOfWork`, `ILogger<LoginHandler>`
- Move ALL business logic from `AuthService.LoginAsync()` into `LoginHandler.Handle()`:
  1. Look up user by email (with roles and permissions) via `IUserRepository.GetByEmailWithRolesAndPermissionsAsync()`
  2. Check user exists and is active
  3. Verify password via `IPasswordHasher.VerifyPassword()`
  4. Generate access token and refresh token via `IJwtService`
  5. Create `RefreshToken` entity, add via `IRefreshTokenRepository.Add()`
  6. Record login via `user.RecordLogin()`
  7. Call `IUnitOfWork.SaveChangesAsync()`
  8. Map to response using `UserDto` from Contracts
- Return `Result<LoginResponse>` (use `Shared.Domain.Result<T>`)
- Invoke `IValidator<LoginCommand>.ValidateAsync()` at the start of Handle. If invalid, return `Result.Failure` with validation errors.

**Logout.cs:**
- `LogoutCommand(Guid UserId)` record
- `LogoutHandler` injecting `IRefreshTokenRepository`, `IUnitOfWork`
- Logic from `AuthService.LogoutAsync()`: revoke all refresh tokens for user via `IRefreshTokenRepository.RevokeAllByUserIdAsync()`, then `IUnitOfWork.SaveChangesAsync()`
- Return `Result` (no value)

**RefreshToken.cs:**
- `RefreshTokenCommand(string RefreshToken)` record
- `RefreshTokenHandler` injecting `IRefreshTokenRepository`, `IUserRepository`, `IJwtService`, `IUnitOfWork`, `ILogger<RefreshTokenHandler>`
- Logic from `AuthService.RefreshTokenAsync()`:
  1. Look up refresh token via `IRefreshTokenRepository.GetByTokenAsync()`
  2. Validate token (not revoked, not expired)
  3. If reuse detected (token already used), revoke entire family via `IRefreshTokenRepository.RevokeAllByFamilyIdAsync()` -- token theft detection
  4. Mark current token as used
  5. Load user with roles/permissions via `IUserRepository`
  6. Generate new access + refresh token pair
  7. Save via `IUnitOfWork.SaveChangesAsync()`
- Return `Result<RefreshTokenResponse>`

**GetCurrentUser.cs:**
- `GetCurrentUserQuery(Guid UserId)` record
- `GetCurrentUserHandler` injecting `IUserRepository`
- Logic: Load user by ID with roles/permissions, map to `UserDto` from Contracts
- Return `Result<UserDto>` (read-only, no UoW needed)

**UpdateLanguage.cs:**
- `UpdateLanguageCommand(Guid UserId, string Language)` record
- `UpdateLanguageCommandValidator` -- validate Language is "vi" or "en"
- `UpdateLanguageHandler` injecting `IUserRepository`, `IUnitOfWork`
- Logic: Load user, call `user.SetPreferredLanguage(language)`, save via UoW
- Return `Result` (no value)

**IMPORTANT -- Behavior preservation:**
- Every handler must produce the EXACT same outcome as the current code path
- Read the existing `AuthService.cs` line by line -- do not miss any null checks, logging, or edge cases
- Use `Shared.Domain.Result<T>` and `Shared.Domain.Error` consistently -- check how they're currently used in the existing code
- Handlers follow Wolverine naming: class ends with `Handler`, method named `Handle`, first param is the command/query record

**1. Update AuthApiEndpoints.cs in Auth.Presentation:**
Replace the placeholder with auth-flow Minimal API endpoints. Read the OLD endpoint files to determine exact:
- URL paths (e.g., `/api/auth/login`, `/api/auth/logout`, `/api/auth/refresh`)
- HTTP methods (POST/GET/PUT)
- Authorization requirements

For each endpoint:
- Receive HTTP request (use `[FromBody]` for POST, `[AsParameters]` for GET query params)
- Construct command/query record with any enrichment (e.g., `IpAddress` from `HttpContext.Connection.RemoteIpAddress`)
- Extract `UserId` from `HttpContext.User.Claims` for endpoints requiring the authenticated user (Logout, GetCurrentUser, UpdateLanguage)
- Dispatch via `IMessageBus.InvokeAsync<T>(command)`
- Map `Result<T>` to HTTP response:
  - `Result.IsSuccess` -> `Results.Ok(result.Value)` (or `Results.NoContent()` for void results)
  - `Result.IsFailure` -> Map error codes to appropriate HTTP status (401 for Unauthorized, 400 for validation, 404 for NotFound)

Auth flow endpoints:
```
POST /api/auth/login       -> LoginCommand (anonymous)
POST /api/auth/logout      -> LogoutCommand (authenticated)
POST /api/auth/refresh     -> RefreshTokenCommand (anonymous -- uses refresh token)
GET  /api/auth/me          -> GetCurrentUserQuery (authenticated)
PUT  /api/auth/language     -> UpdateLanguageCommand (authenticated)
```

**IMPORTANT:** Check the exact URL paths in the existing endpoints. Do NOT guess. Read the `[WolverinePost("/api/auth/login")]` attributes.

Use `.RequireAuthorization()` on individual endpoints or route groups matching the old `[Authorize]` attributes. Login and RefreshToken are anonymous (no auth required).
