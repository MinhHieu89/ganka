  <action>
    For each feature file, follow this pattern:
    - Namespace: `Auth.Application.Features`
    - Each file contains: static endpoint class + FluentValidation validator (if endpoint has request body) + request/response records (feature-local only, NOT cross-module DTOs)
    - Response DTOs that are in Auth.Contracts (LoginResponse, UserDto) stay in Contracts -- reference them, don't duplicate
    - Command/request records that were in Contracts (LoginRequest, RefreshTokenRequest) are MOVED here as feature-local records

    **Login.cs:**
    1. Read `LoginEndpoint.cs` (current endpoint) and `AuthService.cs` (business logic for login).
    2. Create `Login.cs` containing:
       - `LoginCommand` record (replaces `LoginRequest` from Contracts -- same fields: Email, Password, RememberMe)
       - `LoginCommandValidator` class (move from Validators/LoginCommandValidator.cs -- same rules)
       - `LoginEndpoint` static class with `[WolverinePost("/api/auth/login")]`
       - INLINE the business logic from AuthService.LoginAsync: user lookup via IAuthRepository, IsActive check, password verification via IPasswordHasher, permission extraction, JWT generation via IJwtService, refresh token storage via IAuthRepository, DTO mapping
       - See RESEARCH.md Pattern 1 for the exact code structure
       - Handler parameters: LoginCommand, IAuthRepository, IJwtService, IPasswordHasher, HttpContext, ILogger<LoginEndpoint>
       - Return: Results.Ok(LoginResponse) on success, Results.Problem 401 on failure

    **Logout.cs:**
    1. Read `LogoutEndpoint.cs` and the logout method in `AuthService.cs`.
    2. Create `Logout.cs` containing:
       - `LogoutRequest` record if needed (or extract from claims)
       - `LogoutEndpoint` static class with appropriate route and [Authorize]
       - INLINE logout logic: extract refresh token from request, revoke via IAuthRepository
       - Handler parameters: IAuthRepository, HttpContext (for claims), ILogger

    **RefreshToken.cs:**
    1. Read `RefreshTokenEndpoint.cs` and the refresh method in `AuthService.cs`.
    2. Create `RefreshToken.cs` containing:
       - `RefreshTokenCommand` record (replaces RefreshTokenRequest from Contracts)
       - `RefreshTokenEndpoint` static class with appropriate route
       - INLINE refresh logic: token lookup via IAuthRepository, expiry validation, token rotation, family revocation on reuse, new JWT + refresh token generation
       - This is the MOST complex handler -- carefully read AuthService to capture all edge cases (token reuse detection, family revocation)

    **GetCurrentUser.cs:**
    1. Read how the current user profile endpoint works (may be in UserEndpoints.cs or LoginEndpoint.cs -- check which endpoint returns user profile).
    2. Create `GetCurrentUser.cs` containing:
       - `GetCurrentUserEndpoint` static class with [Authorize]
       - Extract user ID from claims, load user with permissions via IAuthRepository, map to UserDto
       - No validator needed (GET endpoint, no body)

    **UpdateLanguage.cs:**
    1. Read `LanguagePreferenceEndpoint.cs`.
    2. Create `UpdateLanguage.cs` containing:
       - `UpdateLanguageRequest` record
       - `UpdateLanguageRequestValidator` (validate language is "vi" or "en")
       - `UpdateLanguageEndpoint` static class with [Authorize]
       - INLINE: extract user ID from claims, update preference via IAuthRepository, save

    CRITICAL: Do NOT create new service interfaces. All business logic goes directly in the handler. Only inject IAuthRepository, IJwtService, IPasswordHasher as needed.

    CRITICAL: Do NOT import anything from Auth.Infrastructure namespace. Handlers must only reference Application and Domain types.
  </action>
