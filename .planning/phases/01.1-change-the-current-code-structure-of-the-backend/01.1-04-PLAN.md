Read existing endpoint files (`UserEndpoints.cs`, `RoleEndpoints.cs`, `PermissionEndpoints.cs`) and service implementations (`UserService.cs`, `RoleService.cs`, `PermissionService.cs`) to extract the EXACT business logic.

**GetUsers.cs:**
- `GetUsersQuery(int Page = 1, int PageSize = 10)` record
- `GetUsersHandler` injecting `IUserRepository`
- Logic from `UserService.GetUsersAsync()`: Paginated user list with roles mapped to `UserDto`
- Return `GetUsersResponse` with list of `UserDto` and pagination info (total count, page, page size)

**CreateUser.cs:**
- `CreateUserCommand(string Email, string FullName, string Password, bool IsActive = true, List<Guid>? RoleIds = null)` record
- `CreateUserCommandValidator` -- move from `Validators/CreateUserCommandValidator.cs` (email valid, name not empty, password min length)
- `CreateUserHandler` injecting `IUserRepository`, `IRoleRepository`, `IPasswordHasher`, `IUnitOfWork`, `IValidator<CreateUserCommand>`
- Logic from `UserService.CreateUserAsync()`:
  1. Validate command
  2. Check email uniqueness via `IUserRepository.EmailExistsAsync()`
  3. Hash password via `IPasswordHasher.HashPassword()`
  4. Create `User` entity with hashed password
  5. If RoleIds provided, load roles via `IRoleRepository` and assign
  6. Add user via `IUserRepository.Add()`
  7. `IUnitOfWork.SaveChangesAsync()`
- Return `Result<UserDto>`

**UpdateUser.cs:**
- `UpdateUserCommand(Guid UserId, string? FullName, bool? IsActive)` record
- `UpdateUserHandler` injecting `IUserRepository`, `IUnitOfWork`
- Logic from `UserService.UpdateUserAsync()`: Load user, update fields, save
- Return `Result<UserDto>`

**AssignRoles.cs:**
- `AssignRolesCommand(Guid UserId, List<Guid> RoleIds)` record
- `AssignRolesHandler` injecting `IUserRepository`, `IRoleRepository`, `IUnitOfWork`
- Logic from `UserService.AssignRolesAsync()`: Load user with roles, clear current roles, assign new roles
- Return `Result`

**GetRoles.cs:**
- `GetRolesQuery()` record (no params, or optional filter)
- `GetRolesHandler` injecting `IRoleRepository`
- Logic from `RoleService.GetRolesAsync()`: Load all roles with permissions, map to DTOs
- Return list of `RoleDto`

**CreateRole.cs:**
- `CreateRoleCommand(string Name, string? Description, List<Guid>? PermissionIds)` record
- `CreateRoleCommandValidator` -- move from `Validators/CreateRoleCommandValidator.cs`
- `CreateRoleHandler` injecting `IRoleRepository`, `IPermissionRepository`, `IUnitOfWork`, `IValidator<CreateRoleCommand>`
- Logic from `RoleService.CreateRoleAsync()`: Validate, check name uniqueness, create role, assign permissions, save
- Return `Result<RoleDto>`

**UpdateRolePermissions.cs:**
- `UpdateRolePermissionsCommand(Guid RoleId, List<Guid> PermissionIds)` record
- `UpdateRolePermissionsHandler` injecting `IRoleRepository`, `IPermissionRepository`, `IUnitOfWork`
- Logic from `RoleService.UpdateRolePermissionsAsync()`: Load role with permissions, clear current, assign new, save
- Return `Result`

**GetPermissions.cs:**
- `GetPermissionsQuery()` record
- `GetPermissionsHandler` injecting `IPermissionRepository`
- Logic from `PermissionService.GetPermissionsAsync()`: Load all permissions, map to DTOs
- Return list of `PermissionDto`

**ALL handlers must:**
- Follow Wolverine naming: class `XxxHandler`, method `Handle`/`HandleAsync`, first param is command/query
- Use `Result<T>` from `Shared.Domain` for responses that can fail
- Invoke `IValidator<T>.ValidateAsync()` inside the handler for commands with validators
- NOT reference any HTTP types, IResult, HttpContext, or status codes
- Use Contracts DTOs (`UserDto`, `RoleDto`, `PermissionDto`) for response types where they exist

**1. Complete AuthApiEndpoints.cs:**

Add admin-flow endpoints to the existing AuthApiEndpoints.cs (which already has auth-flow routes from Plan 03, or if Plan 03 hasn't run yet due to parallel execution, add all 13 routes). Read the OLD `UserEndpoints.cs`, `RoleEndpoints.cs`, `PermissionEndpoints.cs` to determine exact URL paths and HTTP methods.

Admin-flow endpoints (all require authorization):
```
GET    /api/auth/users              -> GetUsersQuery
POST   /api/auth/users              -> CreateUserCommand
PUT    /api/auth/users/{id}         -> UpdateUserCommand
PUT    /api/auth/users/{id}/roles   -> AssignRolesCommand
GET    /api/auth/roles              -> GetRolesQuery
POST   /api/auth/roles              -> CreateRoleCommand
PUT    /api/auth/roles/{id}/permissions -> UpdateRolePermissionsCommand
GET    /api/auth/permissions        -> GetPermissionsQuery
```

**CRITICAL:** Check the ACTUAL URL paths from existing endpoints. The above are typical patterns but must match exactly.

Each endpoint follows the thin pattern:
- Construct command/query from HTTP request params/body
- Dispatch via `bus.InvokeAsync<T>(command)`
- Map `Result<T>` to HTTP response (Ok/Created/NoContent on success, Problem on failure)

For endpoints with route parameters (e.g., `{id}`), construct the command by combining route param + request body.

Apply authorization requirements:
- Admin routes: `.RequireAuthorization()` (same as existing `[Authorize]`)
- Permission-based auth: If existing endpoints check specific permissions, replicate that check either via policy or inline permission verification

- The safest approach: Plan 04 reads the current AuthApiEndpoints.cs and adds only admin-flow routes, regardless of Plan 03 state.
