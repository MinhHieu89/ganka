# Phase 01.1: Change the Current Code Structure of the Backend - Research

**Researched:** 2026-02-28
**Domain:** Backend vertical slice restructuring -- Wolverine mediator, Minimal API presentation, repository-per-aggregate, Unit of Work
**Confidence:** HIGH

## Summary

This phase restructures the existing Auth and Audit backend modules from a type-based organization (Endpoints/, Validators/, Services/) to a vertical slice feature-based organization with proper layer separation. The current architecture has HTTP concerns (Wolverine.Http attributes, IResult, HttpContext) embedded in the Application layer, and business logic lives in Infrastructure service classes rather than Application handlers. The restructuring introduces three key changes: (1) Presentation layer with Minimal API endpoints that dispatch commands via Wolverine's `IMessageBus`, (2) Application layer with feature-per-file handlers containing all business logic, validators, and request/response records, (3) Infrastructure layer with thin repository-per-aggregate-root implementations and a Unit of Work pattern wrapping DbContext.

The existing codebase is well-structured with clear module boundaries, DDD entities, and architecture tests. The restructuring is a refactoring exercise -- no new features, no database schema changes, no migration changes. All 16 existing features (13 Auth + 3 Audit) must produce identical API responses after restructuring. The stack is .NET 10, Wolverine 5.x, FluentValidation 12.x, EF Core 10.x, and xUnit 2.x with NSubstitute for mocking.

**Primary recommendation:** Use Wolverine `IMessageBus.InvokeAsync<TResponse>(command)` for request/reply dispatching from Minimal API endpoints to Application handlers. Handlers are plain classes following Wolverine naming conventions (`XxxHandler` with `Handle`/`HandleAsync` methods), injected with repository interfaces and IUnitOfWork. Each feature file is self-contained: command record, response record, validator, handler class.

### Pattern 1: Vertical Slice Feature File

**What:** Each feature is a single file containing the command/query record, response record (if any), FluentValidation validator (if needed), and handler class.
**When to use:** Every feature in the Application layer.

```csharp
// Source: User decision + Wolverine handler conventions (Context7: /jasperfx/wolverine)
// File: Auth.Application/Features/Login.cs

using Auth.Application.Interfaces;
using Auth.Contracts.Dtos;
using Auth.Domain.Entities;
using FluentValidation;
using Microsoft.Extensions.Logging;
using Shared.Domain;

namespace Auth.Application.Features;

// --- Command record (internal to Application) ---
public sealed record LoginCommand(string Email, string Password, bool RememberMe = false, string? IpAddress = null);

// --- Validator ---
public sealed class LoginCommandValidator : AbstractValidator<LoginCommand>
{
    public LoginCommandValidator()
    {
        RuleFor(x => x.Email).NotEmpty().EmailAddress();
        RuleFor(x => x.Password).NotEmpty().MinimumLength(8);
    }
}

// --- Handler ---
public sealed class LoginHandler
{
    private readonly IUserRepository _userRepository;
    private readonly IRefreshTokenRepository _refreshTokenRepository;
    private readonly IPasswordHasher _passwordHasher;
    private readonly IJwtService _jwtService;
    private readonly IUnitOfWork _unitOfWork;
    private readonly ILogger<LoginHandler> _logger;

    public LoginHandler(
        IUserRepository userRepository,
        IRefreshTokenRepository refreshTokenRepository,
        IPasswordHasher passwordHasher,
        IJwtService jwtService,
        IUnitOfWork unitOfWork,
        ILogger<LoginHandler> logger)
    {
        _userRepository = userRepository;
        _refreshTokenRepository = refreshTokenRepository;
        _passwordHasher = passwordHasher;
        _jwtService = jwtService;
        _unitOfWork = unitOfWork;
        _logger = logger;
    }

    // Wolverine discovers this via naming convention: class ends with "Handler", method named "Handle"
    // First argument is the message (command), additional args are injected from DI
    public async Task<Result<LoginResponse>> Handle(LoginCommand command)
    {
        var user = await _userRepository.GetByEmailWithRolesAndPermissionsAsync(command.Email);
        if (user is null)
        {
            _logger.LogWarning("Login attempt for non-existent email: {Email}", command.Email);
            return Result<LoginResponse>.Failure(Error.Unauthorized());
        }

        if (!user.IsActive)
            return Result<LoginResponse>.Failure(Error.Unauthorized());

        if (!_passwordHasher.VerifyPassword(command.Password, user.PasswordHash))
            return Result<LoginResponse>.Failure(Error.Unauthorized());

        // Business logic in handler
        var permissions = user.GetEffectivePermissions().Select(p => p.ToString()).ToList();
        var (accessToken, expiresAt) = _jwtService.GenerateAccessToken(user, permissions);
        var refreshTokenValue = _jwtService.GenerateRefreshToken();

        var familyId = Guid.NewGuid();
        var refreshTokenLifetimeDays = _jwtService.GetRefreshTokenLifetimeDays(command.RememberMe);
        var refreshToken = new RefreshToken(
            refreshTokenValue, user.Id,
            DateTime.UtcNow.AddDays(refreshTokenLifetimeDays), familyId);

        _refreshTokenRepository.Add(refreshToken);
        user.RecordLogin(command.IpAddress);

        await _unitOfWork.SaveChangesAsync();

        var userDto = new UserDto(user.Id, user.Email, user.FullName, user.PreferredLanguage,
            user.IsActive, user.UserRoles.Select(ur => ur.Role.Name).ToList(), permissions);

        return new LoginResponse(accessToken, refreshTokenValue, expiresAt, userDto);
    }
}
```

### Pattern 2: Minimal API Presentation Endpoint

**What:** Thin endpoint that receives HTTP request, constructs command, dispatches via `IMessageBus`, maps result to HTTP response.
**When to use:** Every endpoint in the Presentation layer.

```csharp
// Source: Wolverine docs (Context7: /jasperfx/wolverine) - Minimal API + IMessageBus pattern
// File: Auth.Presentation/AuthApiEndpoints.cs

using Auth.Application.Features;
using Microsoft.AspNetCore.Builder;
using Microsoft.AspNetCore.Http;
using Microsoft.AspNetCore.Routing;
using Wolverine;

namespace Auth.Presentation;

public static class AuthApiEndpoints
{
    public static IEndpointRouteBuilder MapAuthApiEndpoints(this IEndpointRouteBuilder app)
    {
        // Auth flow
        app.MapPost("/api/auth/login", async (LoginCommand command, IMessageBus bus, HttpContext ctx) =>
        {
            var enriched = command with { IpAddress = ctx.Connection.RemoteIpAddress?.ToString() };
            var result = await bus.InvokeAsync<Result<LoginResponse>>(enriched);
            return result.IsSuccess ? Results.Ok(result.Value)
                : result.Error.Code switch
                {
                    "Error.Unauthorized" => Results.Problem(detail: result.Error.Description, title: "Unauthorized", statusCode: 401),
                    _ => Results.Problem(result.Error.Description, statusCode: 400)
                };
        });

        // ... other endpoints

        return app;
    }
}
```

### Pattern 4: Unit of Work Interface

**What:** Simple interface wrapping `DbContext.SaveChangesAsync()`, defined in Application, implemented in Infrastructure.
**When to use:** Every handler that mutates state.

```csharp
// File: Auth.Application/Interfaces/IUnitOfWork.cs
namespace Auth.Application.Interfaces;

public interface IUnitOfWork
{
    Task<int> SaveChangesAsync(CancellationToken cancellationToken = default);
}

// File: Auth.Infrastructure/UnitOfWork.cs
namespace Auth.Infrastructure;

public sealed class UnitOfWork : IUnitOfWork
{
    private readonly AuthDbContext _dbContext;

    public UnitOfWork(AuthDbContext dbContext) => _dbContext = dbContext;

    public Task<int> SaveChangesAsync(CancellationToken cancellationToken = default)
        => _dbContext.SaveChangesAsync(cancellationToken);
}
```

### Pattern 5: Unit Test for Handler

**What:** Tests handler logic with mocked repositories and UoW. No database, no HTTP.
**When to use:** Every handler gets at least one test file.

### Anti-Patterns to Avoid
- **God repository:** Do NOT create a single `IAuthRepository` interface that handles User, Role, Permission, RefreshToken, and SystemSetting operations. Each aggregate root gets its own repository.
- **Business logic in repositories:** Repositories are pure data access. Filtering, validation, and orchestration belong in handlers.
- **HTTP concerns in Application layer:** No `IResult`, `HttpContext`, `[Authorize]`, `[FromBody]`, or status code logic in handler code. Return `Result<T>` from handlers; Presentation maps to HTTP.
- **SaveChanges in repositories:** Only `IUnitOfWork.SaveChangesAsync()` persists changes. Repositories never call SaveChanges.
- **Wolverine.Http attributes in Application:** Remove all `[WolverinePost]`, `[WolverineGet]`, `[WolverinePut]` attributes from Application layer. Those move to Presentation (as Minimal API, not Wolverine.Http).

## Aggregate Root Analysis (Claude's Discretion)

### Auth Module Aggregate Roots

Based on domain model analysis of the existing entities:

| Entity | Aggregate Root? | Reasoning |
|--------|----------------|-----------|
| **User** | YES | Independent lifecycle, referenced by RefreshToken. Has UserRoles collection. Central identity entity. |
| **Role** | YES | Independent lifecycle, can exist without specific users. Has RolePermissions collection. Created/managed independently. |
| **Permission** | NO (reference entity) | Seeded data, never created/deleted via user action. Owned by the system. Read-only lookups. Gets its own read-only repository. |
| **RefreshToken** | NO (child of User) | Belongs to User lifecycle. However, queried independently (by token value) for refresh flow. Gets its own repository for practical data access needs. |
| **UserRole** | NO (join entity) | Managed through User aggregate |
| **RolePermission** | NO (join entity) | Managed through Role aggregate |
| **SystemSetting** | NO (configuration) | Key-value settings. Read-only from handler perspective. Gets a simple read repository. |

**Repository interfaces needed:**
1. `IUserRepository` -- User aggregate (includes navigation to UserRoles -> Roles -> Permissions)
2. `IRoleRepository` -- Role aggregate (includes navigation to RolePermissions -> Permissions)
3. `IPermissionRepository` -- Read-only permission queries (GetAll, GetByIds)
4. `IRefreshTokenRepository` -- Practical repository for token lookup/management
5. `ISystemSettingRepository` -- Simple read-only for JWT configuration values

### Audit Module

Audit has NO aggregate roots (entities are immutable append-only logs). The existing `IAuditReadContext` pattern (exposing `IQueryable<AuditLog>` and `IQueryable<AccessLog>`) is appropriate but should be renamed to `IAuditReadRepository` per user decision. The interface remains in Application, implemented by AuditDbContext.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Command dispatch | Custom mediator | Wolverine `IMessageBus.InvokeAsync<T>()` | Already in stack, handles DI, handler discovery, async dispatch |
| Input validation | Manual if-checks in handlers | FluentValidation `IValidator<T>.ValidateAsync()` | Declarative rules, consistent error format, already in use |
| Unit of Work | Custom transaction manager | EF Core DbContext (wrapped in IUnitOfWork) | DbContext already IS a Unit of Work; just expose SaveChangesAsync via interface |
| Password hashing | Custom crypto code | `PasswordHasher` class (already exists, move behind interface) | Argon2id already implemented correctly with proper salt, timing-safe comparison |
| JWT generation | Custom token code | `JwtService` class (already exists, move behind interface) | Already handles claims, token lifetimes, system settings |
| Test mocking | Manual fakes | NSubstitute | Already used in Audit.Unit.Tests; generates mocks from interfaces |

**Key insight:** This phase is a refactoring, not a greenfield build. All business logic already exists in Infrastructure services. The task is to relocate it to Application handlers and introduce proper abstractions (repository interfaces, IUnitOfWork) without changing behavior.

## Common Pitfalls

### Pitfall 1: Breaking Wolverine Handler Discovery
**What goes wrong:** Handlers are not discovered by Wolverine because naming conventions are violated or assembly is not registered.
**Why it happens:** Wolverine auto-discovers handler classes ending with "Handler" that have methods named "Handle"/"HandleAsync". If the class or method name deviates, it's silently ignored.
**How to avoid:** (1) Name all handlers `XxxHandler` with a public `Handle` or `HandleAsync` method. (2) Keep `Marker.cs` in Application layer. (3) Ensure `opts.Discovery.IncludeAssembly(typeof(Auth.Application.Marker).Assembly)` remains in Bootstrapper. (4) First argument of Handle method must be the command/query record.
**Warning signs:** Endpoint returns 404 or "No handler found" at runtime.

### Pitfall 2: Wolverine Return Type for Request/Reply
**What goes wrong:** `IMessageBus.InvokeAsync<T>()` expects the handler to return exactly `T`, not `Task<T>` at the generic parameter level.
**Why it happens:** Confusion between the method signature return type and the generic parameter. If handler returns `Task<Result<LoginResponse>>`, the Minimal API call should be `bus.InvokeAsync<Result<LoginResponse>>(command)`.
**How to avoid:** Match the generic parameter of `InvokeAsync<T>` to the handler's actual return value type (unwrapped from Task). If handler signature is `public async Task<Result<LoginResponse>> Handle(LoginCommand command)`, then call `await bus.InvokeAsync<Result<LoginResponse>>(command)`.
**Warning signs:** InvalidOperationException at runtime about missing handler or type mismatch.

### Pitfall 3: IUnitOfWork and Repository Sharing the Same DbContext Instance
**What goes wrong:** Repository writes and UoW save operate on different DbContext instances, causing changes to be lost.
**Why it happens:** If repositories and UoW are registered with different DbContext lifetimes or factory patterns.
**How to avoid:** Register `IUnitOfWork` as scoped, wrapping the same `AuthDbContext` instance that repositories use. Both are injected from DI per-request scope. EF Core's `AddDbContext<T>` defaults to Scoped lifetime, so all injections within one request share the same instance.
**Warning signs:** SaveChangesAsync returns 0, entities not persisted.

### Pitfall 4: Circular Project References
**What goes wrong:** Auth.Application references Auth.Infrastructure (to use DbContext), breaking the dependency direction.
**Why it happens:** When moving business logic from Infrastructure services to Application handlers, it's tempting to bring along DbContext dependencies.
**How to avoid:** Application layer defines interfaces only. Infrastructure implements them. The Bootstrapper wires everything together via DI registration. Application NEVER references Infrastructure.
**Warning signs:** Architecture tests fail (`Application_Should_Not_Depend_On_Infrastructure`).

### Pitfall 5: Forgetting to Update Bootstrapper DI Registration
**What goes wrong:** New repository interfaces have no implementation registered, causing runtime DI exceptions.
**Why it happens:** Adding 5 repository interfaces + IUnitOfWork + IPasswordHasher interface requires corresponding DI registrations in Program.cs.
**How to avoid:** For every interface added to Application layer, add a matching `builder.Services.AddScoped<IXxx, Xxx>()` in Bootstrapper's Program.cs. Create a checklist.
**Warning signs:** `InvalidOperationException: Unable to resolve service for type 'IUserRepository'` at runtime.

### Pitfall 6: Auth Contracts DTOs vs Application-Internal Records
**What goes wrong:** Confusion about which records live in Contracts (cross-module) vs Application (internal). Duplicating or misplacing types.
**Why it happens:** Currently, command records like `LoginRequest`, `CreateUserCommand` live in Auth.Contracts. After restructuring, the Application handler needs its own command record (potentially with extra fields like IpAddress) while Contracts DTOs remain for cross-module and API response use.
**How to avoid:** (1) Cross-module DTOs (`UserDto`, `RoleDto`, `PermissionDto`, `LoginResponse`) stay in Contracts. (2) Handler command/query records are defined inside the feature file in Application. (3) Presentation layer maps between HTTP request body and Application command. (4) Contracts records used for API responses can be returned from handlers.
**Warning signs:** Architecture test for Contracts independence fails; or Contracts starts depending on Application/Domain.

### Pitfall 7: Removing Wolverine.Http Without Updating Bootstrapper
**What goes wrong:** After removing `[WolverinePost]`/`[WolverineGet]` attributes from Application endpoints, `app.MapWolverineEndpoints()` in Program.cs no longer discovers any HTTP endpoints, but new Minimal API endpoints are not registered.
**How to avoid:** (1) Add `app.MapAuthApiEndpoints()` and `app.MapAuditApiEndpoints()` in Program.cs. (2) Keep `app.MapWolverineEndpoints()` if other modules still use Wolverine.Http (they do -- scaffold modules). (3) Only remove `WolverineFx.Http` package from Auth.Application, NOT from Bootstrapper.
**Warning signs:** All Auth/Audit API endpoints return 404 after restructuring.

## Code Examples

### IPasswordHasher Interface (Moving Concrete Behind Abstraction)

```csharp
// Source: Existing PasswordHasher implementation in Auth.Infrastructure.Services
// File: Auth.Application/Interfaces/IPasswordHasher.cs
namespace Auth.Application.Interfaces;

public interface IPasswordHasher
{
    string HashPassword(string password);
```

```csharp
// File: Auth.Infrastructure/Repositories/UserRepository.cs
using Auth.Application.Interfaces;
using Auth.Domain.Entities;
using Microsoft.EntityFrameworkCore;

namespace Auth.Infrastructure.Repositories;

public sealed class UserRepository : IUserRepository
{
    private readonly AuthDbContext _dbContext;

    public UserRepository(AuthDbContext dbContext) => _dbContext = dbContext;

    public async Task<User?> GetByIdAsync(Guid id)
        => await _dbContext.Users.FindAsync(id);

    public async Task<User?> GetByIdWithRolesAsync(Guid id)
        => await _dbContext.Users
            .Include(u => u.UserRoles)
            .FirstOrDefaultAsync(u => u.Id == id);

    public async Task<User?> GetByEmailWithRolesAndPermissionsAsync(string email)
        => await _dbContext.Users
            .Include(u => u.UserRoles)
                .ThenInclude(ur => ur.Role)
                    .ThenInclude(r => r.RolePermissions)
                        .ThenInclude(rp => rp.Permission)
            .FirstOrDefaultAsync(u => u.Email == email);

    public async Task<(List<User> Users, int TotalCount)> GetPagedAsync(int page, int pageSize)
    {
        var totalCount = await _dbContext.Users.CountAsync();
        var users = await _dbContext.Users
            .AsNoTracking()
            .Include(u => u.UserRoles)
                .ThenInclude(ur => ur.Role)
                    .ThenInclude(r => r.RolePermissions)
                        .ThenInclude(rp => rp.Permission)
            .OrderBy(u => u.FullName)
            .Skip((page - 1) * pageSize)
            .Take(pageSize)
            .ToListAsync();
        return (users, totalCount);
    }

    public async Task<bool> EmailExistsAsync(string email)
        => await _dbContext.Users.AnyAsync(u => u.Email == email);

    public void Add(User user) => _dbContext.Users.Add(user);
}
```

## Open Questions

1. **FluentValidation invocation in handlers vs Presentation layer**
   - What we know: User decided validators live in Application feature files. FluentValidation can be invoked manually via `IValidator<T>.ValidateAsync()`.
   - What's unclear: Should validation be invoked inside the handler (making the handler self-validating) or in the Minimal API endpoint before dispatching?
   - Recommendation: Invoke validation inside the handler. This keeps the handler self-contained and testable (unit tests verify validation behavior). The Minimal API endpoint stays as thin as possible -- just dispatch and map result. If validation fails, handler returns `Result.Failure` with validation errors.

2. **Wolverine handler return type for void operations**
   - What we know: Wolverine supports `void`/`Task` returns (fire-and-forget) and `T`/`Task<T>` returns (request/reply).
   - What's unclear: For operations like Logout that return `Result` (no value), should the handler return `Result` and use `InvokeAsync<Result>(command)`, or return void and handle differently?
   - Recommendation: Use `Task<Result>` return type consistently. All handlers return `Result` or `Result<T>`. Presentation uses `InvokeAsync<Result>(command)` or `InvokeAsync<Result<T>>(command)`.

3. **Presentation project reference chain**
   - What we know: Auth.Presentation needs to reference Auth.Application (for command/query types) and needs `IMessageBus` from WolverineFx.
   - What's unclear: Should Presentation reference Auth.Contracts too (for response DTOs used in endpoint code)?
   - Recommendation: Auth.Presentation references Auth.Application (which transitively brings Auth.Contracts and Auth.Domain). No direct Contracts reference needed -- it comes through Application.

4. **Marker.cs in new Presentation project**
   - What we know: Wolverine uses `Marker.cs` to discover handlers in Application assemblies.
   - What's unclear: Does the Presentation project need its own Marker? Do Minimal API endpoints need Wolverine discovery?
   - Recommendation: NO Marker needed in Presentation. Minimal API endpoints are registered explicitly via `MapAuthApiEndpoints()`. Only Application keeps Marker for handler discovery.
