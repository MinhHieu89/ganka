## Summary

This phase restructures the backend Application layer of the Auth and Audit modules from a type-based organization (`Endpoints/`, `Validators/`, `Services/`) to a vertical slice feature-based organization (`Features/`). The core change is eliminating the service interface pattern where endpoints delegate all logic to `IAuthService`, `IRoleService`, `IUserService`, `IPermissionService` (defined in Application, implemented in Infrastructure). Instead, each feature file becomes self-contained: endpoint handler + validator + request/response records in a single `.cs` file, with business logic living directly in the handler.

The Auth module currently has 7 endpoint classes across `Endpoints/`, 3 validators in `Validators/`, and 5 service interfaces in `Services/` whose implementations in `Auth.Infrastructure/Services/` contain all business logic (DB queries, password verification, token generation, role management). The Audit module has 2 endpoint classes that already use a thin `IAuditReadContext` interface (essentially a read-only repository), making it closer to the target pattern. After restructuring, the Auth module's `Services/` folder and all service interfaces (`IAuthService`, `IRoleService`, `IUserService`, `IPermissionService`) are removed entirely. The business logic moves into the feature handlers. Only specialized infrastructure service interfaces (`IJwtService`, `IPasswordHasher`) remain — these are genuine infrastructure concerns. New thin repository interfaces (`IAuthRepository` or similar) provide data access. The Audit module needs lighter restructuring since it already uses the thin-interface pattern.

Unit tests must be added for all feature handlers. The project already has `Audit.Unit.Tests` (empty scaffold) and uses xUnit + FluentAssertions + NSubstitute. The Auth module needs a new `Auth.Unit.Tests` project. Tests mock repository interfaces and infrastructure service interfaces, then test handler logic directly.

**Primary recommendation:** Restructure Auth module first (more complex, sets the pattern), then Audit module. For each feature: create the feature file with colocated endpoint + validator + records, inline the business logic from the Infrastructure service, define thin repository interfaces, update DI registration in `Program.cs`, add unit tests, then delete the old files.

### Pattern 1: Vertical Slice Feature File
**What:** One `.cs` file per feature containing all related types — endpoint, validator, request/response records.
**When to use:** Every feature in the Application layer.
**Example:**
```csharp
// Source: Wolverine docs + project CONTEXT.md decisions
// File: Auth.Application/Features/Login.cs

using Auth.Application.Repositories;
using Auth.Application.Services;
using Auth.Contracts.Dtos;
using FluentValidation;
using Microsoft.AspNetCore.Http;
using Microsoft.AspNetCore.Mvc;
using Shared.Domain;
using Wolverine.Http;

namespace Auth.Application.Features;

// --- Request/Response records (feature-local, not in Contracts) ---
public sealed record LoginCommand(string Email, string Password, bool RememberMe = false);

// --- Validator (colocated) ---
public sealed class LoginCommandValidator : AbstractValidator<LoginCommand>
{
    public LoginCommandValidator()
    {
        RuleFor(x => x.Email)
            .NotEmpty().WithMessage("Email is required.")
            .EmailAddress().WithMessage("Invalid email format.");

        RuleFor(x => x.Password)
            .NotEmpty().WithMessage("Password is required.")
            .MinimumLength(8).WithMessage("Password must be at least 8 characters.");
    }
}

// --- Endpoint handler (owns the business logic) ---
public static class LoginEndpoint
{
    [WolverinePost("/api/auth/login")]
    public static async Task<IResult> Post(
        [FromBody] LoginCommand command,
        IAuthRepository repository,
        IJwtService jwtService,
        IPasswordHasher passwordHasher,
        HttpContext httpContext,
        ILogger<LoginEndpoint> logger)
    {
        var user = await repository.GetUserWithRolesAndPermissionsAsync(command.Email);
        if (user is null)
        {
            logger.LogWarning("Login attempt for non-existent email: {Email}", command.Email);
            return Results.Problem(detail: "Unauthorized", title: "Unauthorized", statusCode: 401);
        }

        if (!user.IsActive)
        {
            logger.LogWarning("Login attempt for deactivated user: {Email}", command.Email);
            return Results.Problem(detail: "Unauthorized", title: "Unauthorized", statusCode: 401);
        }

        if (!passwordHasher.VerifyPassword(command.Password, user.PasswordHash))
        {
            logger.LogWarning("Invalid password for user: {Email}", command.Email);
            return Results.Problem(detail: "Unauthorized", title: "Unauthorized", statusCode: 401);
        }

        // Business logic lives HERE, not in a service
        var permissions = user.GetEffectivePermissions()
            .Select(p => p.ToString())
            .ToList();

        var (accessToken, expiresAt) = jwtService.GenerateAccessToken(user, permissions);
        var refreshTokenValue = jwtService.GenerateRefreshToken();

        var ipAddress = httpContext.Connection.RemoteIpAddress?.ToString();
        user.RecordLogin(ipAddress);

        var refreshTokenLifetimeDays = jwtService.GetRefreshTokenLifetimeDays(command.RememberMe);
        await repository.StoreRefreshTokenAsync(user, refreshTokenValue, refreshTokenLifetimeDays);

        var userDto = new UserDto(
            user.Id, user.Email, user.FullName, user.PreferredLanguage,
            user.IsActive,
            user.UserRoles.Select(ur => ur.Role.Name).ToList(),
            permissions);

        // LoginResponse is in Contracts (cross-module DTO)
        return Results.Ok(new LoginResponse(accessToken, refreshTokenValue, expiresAt, userDto));
    }
}
```

### Pattern 2: Thin Repository Interface
**What:** Repository interface in Application layer with only the data-access methods needed by features. No business logic. Implementation in Infrastructure uses `AuthDbContext`.
**When to use:** For all data access from feature handlers.
**Example:**
```csharp
// File: Auth.Application/Repositories/IAuthRepository.cs
using Auth.Domain.Entities;

namespace Auth.Application.Repositories;

public interface IAuthRepository
{
    // User queries
    Task<User?> GetUserWithRolesAndPermissionsAsync(string email);
    Task<User?> GetUserWithRolesAndPermissionsAsync(Guid userId);
    Task<User?> GetUserWithRolesAsync(Guid userId);
    Task<(List<User> Users, int TotalCount)> GetUsersPagedAsync(int page, int pageSize);
    Task<bool> EmailExistsAsync(string email);

    // User mutations
    Task AddUserAsync(User user);
    Task UpdateUserAsync(User user);

    // Role queries and mutations
    Task<List<Role>> GetRolesWithPermissionsAsync();
    Task<Role?> GetRoleWithPermissionsAsync(Guid roleId);
    Task<List<Role>> GetRolesByIdsAsync(List<Guid> roleIds);
    Task<bool> RoleNameExistsAsync(string name);
    Task AddRoleAsync(Role role);

    // Permission queries
    Task<List<Permission>> GetPermissionsAsync();
    Task<List<Permission>> GetPermissionsByIdsAsync(List<Guid> permissionIds);

    // Refresh token operations
    Task<RefreshToken?> GetRefreshTokenWithUserAsync(string token);
    Task StoreRefreshTokenAsync(User user, string tokenValue, int lifetimeDays);
    Task RevokeRefreshTokenAsync(string token, Guid userId);
    Task RevokeTokenFamilyAsync(Guid familyId);

    // Unit of work
    Task SaveChangesAsync(CancellationToken ct = default);
}
```

### Pattern 5: Request/Response Records — Where They Live
**What:** Distinguishing between feature-local records and cross-module Contracts DTOs.
**Decision rule:**
- **Feature-local** (defined in the feature `.cs` file): `LoginCommand`, `LogoutRequest`, `UpdateLanguageRequest` — types used only by the endpoint handler, not consumed by other modules.
- **Contracts** (stay in `Auth.Contracts/Dtos/`): `LoginResponse`, `UserDto`, `RoleDto`, `PermissionDto` — types consumed by other modules or the frontend API contract.

**Current DTOs in Auth.Contracts that should STAY in Contracts:**
| DTO | Reason |
|-----|--------|
| `LoginResponse` | API response contract, contains `UserDto` |
| `UserDto` | Cross-module: other modules may need user info |
| `RoleDto` | API response DTO |
| `PermissionDto` | API response DTO |

**Current DTOs in Auth.Contracts that can MOVE to feature files:**
| DTO | Move to | Reason |
|-----|---------|--------|
| `LoginRequest` | `Features/Login.cs` as `LoginCommand` | Only used by login endpoint |
| `RefreshTokenRequest` | `Features/RefreshToken.cs` | Only used by refresh endpoint |
| `CreateUserCommand` | `Features/CreateUser.cs` | Only used by create user endpoint |
| `CreateRoleCommand` | `Features/CreateRole.cs` | Only used by create role endpoint |
| `UpdateUserCommand` | `Features/UpdateUser.cs` | Only used by update user endpoint |
| `UpdateRolePermissionsCommand` | `Features/UpdateRolePermissions.cs` | Only used by that endpoint |
| `AssignRolesCommand` | `Features/AssignRoles.cs` | Only used by assign roles endpoint |

**Important:** After moving records out of Contracts, verify no other module or project references them. If any do (unlikely for command records), they must stay in Contracts.

### Anti-Patterns to Avoid
- **Fat service layer:** Don't create a new service interface to replace the old ones. The whole point is to eliminate the service layer and put business logic in the handler.
- **God repository:** Don't make `IAuthRepository` a catch-all with 50 methods. If it gets too large, consider splitting into focused interfaces (`IUserRepository`, `IRoleRepository`). However, for Auth module's size (~13 features), a single repository is acceptable.
- **Leaking EF Core into Application:** The repository interface must not expose `IQueryable<T>` (that leaks Infrastructure concerns). Return materialized collections.
- **Moving PermissionGroupDto:** `PermissionGroupDto` is currently defined inside `IPermissionService.cs` in Application/Services. It should move to the feature file or Contracts — NOT stay orphaned in a deleted file.
- **Breaking architecture tests:** The existing `DependencyDirectionTests` and `ModuleBoundaryTests` enforce that Application does NOT depend on Infrastructure. After restructuring, all business logic is in Application (good), but handlers MUST NOT import `AuthDbContext` or anything from Infrastructure namespace.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Password hashing | Custom hash logic in handler | `IPasswordHasher` interface + `PasswordHasher` in Infrastructure | Argon2id has subtle security parameters; keep it isolated |
| JWT generation | Inline JWT code in handlers | `IJwtService` interface + `JwtService` in Infrastructure | Token generation depends on config, signing keys, and system settings table |
| Request validation | Manual if-checks in handlers | FluentValidation + `UseFluentValidationProblemDetailMiddleware()` | Wolverine auto-discovers validators and short-circuits before the handler runs |
| Repository pattern | Hand-writing SQL or raw ADO.NET | EF Core behind `IAuthRepository` | Type safety, migrations, query filters for soft delete and multi-tenant |
| Test mocking | Manual fake classes | NSubstitute `Substitute.For<T>()` | Cleaner syntax, less boilerplate, compile-time safe |

**Key insight:** The restructuring is about **where** business logic lives (Application, not Infrastructure), not about replacing libraries. All existing libraries remain. The change is purely organizational and architectural.

## Common Pitfalls

### Pitfall 1: Breaking Wolverine Endpoint Discovery
**What goes wrong:** After moving endpoints from `Auth.Application.Endpoints` namespace to `Auth.Application.Features`, Wolverine fails to discover them.
**Why it happens:** Wolverine discovers handlers via assembly scanning (configured with `opts.Discovery.IncludeAssembly(typeof(Auth.Application.Marker).Assembly)`). It scans by assembly, not by namespace. As long as the endpoint classes are in the same assembly (`Auth.Application`), they will be found regardless of namespace.
**How to avoid:** Verify `Marker.cs` is still in `Auth.Application` project root. Namespace changes within the same assembly are fine.
**Warning signs:** 404 responses from endpoints after restructuring.

### Pitfall 2: Circular References When Adding Repository Interfaces
**What goes wrong:** Adding `IAuthRepository` to Application that returns domain entities. Application already references Domain. Infrastructure implements the repository and references Application. This is the correct direction. The danger is if the repository interface accidentally references Infrastructure types (e.g., `DbSet<T>`, `IQueryable<T>`).
**Why it happens:** Copy-pasting data access code from Infrastructure service into the interface definition.
**How to avoid:** Repository interface methods must use only Domain types and primitive types. Never `IQueryable`, never `DbSet`, never `DbContext`.
**Warning signs:** Architecture test `Application_Should_Not_Depend_On_Infrastructure` failing.

### Pitfall 3: Forgetting to Update DI Registration in Program.cs
**What goes wrong:** After removing service interfaces (`IAuthService`, `IUserService`, etc.) and adding repository interface (`IAuthRepository`), the DI container fails at runtime.
**Why it happens:** `Program.cs` still registers old services, or new repository is not registered.
**How to avoid:** For each removed service interface, remove its `AddScoped<>` line. For each new interface, add its registration. Also add `IPasswordHasher` registration.
**Warning signs:** `InvalidOperationException: Unable to resolve service for type 'IAuthRepository'` at runtime.

### Pitfall 4: DTOs That Are Actually Cross-Module Contracts
**What goes wrong:** Moving a DTO from `Auth.Contracts` into a feature file, then discovering another module or the frontend expects that type from `Auth.Contracts`.
**Why it happens:** Not checking all references before moving.
**How to avoid:** Before moving any DTO from Contracts to a feature file, check if it's referenced outside `Auth.Application`. Command/request records are almost never cross-module (they're inbound API requests). Response DTOs (`UserDto`, `LoginResponse`, `RoleDto`, `PermissionDto`) are likely cross-module and should stay in Contracts.
**Warning signs:** Compilation errors in other modules after moving DTOs.

### Pitfall 5: Audit Module's IAuditReadContext Pattern
**What goes wrong:** Trying to replace `IAuditReadContext` with a full repository pattern, but the Audit endpoints use `IQueryable<T>` heavily for dynamic query composition (filtering, cursor pagination).
**Why it happens:** The Audit module has a legitimate reason for IQueryable — it builds complex queries dynamically.
**How to avoid:** The Audit module already uses a pattern closer to the target. `IAuditReadContext` is essentially a thin read repository. Consider renaming it to `IAuditReadRepository` for consistency, but DON'T change the `IQueryable<T>` pattern — it's needed for the dynamic query composition. Alternatively, move the query logic into specific repository methods.
**Warning signs:** Overly complex repository interface with dozens of parameter combinations for filtering.

### Pitfall 6: Test Project Missing EF Core / Domain References
**What goes wrong:** Unit tests can't create domain entities (e.g., `User.Create(...)`) because the test project doesn't reference `Auth.Domain`.
**Why it happens:** Test project only references `Auth.Application` but domain entities are in `Auth.Domain`.
**How to avoid:** Test project must reference both `Auth.Application` and `Auth.Domain` (and `Shared.Domain` for `Result<T>`, `BranchId`, etc.).
**Warning signs:** Compilation errors when writing test arrange code.

### Pitfall 7: ILogger Injection in Static Endpoint Methods
**What goes wrong:** `ILogger<LoginEndpoint>` fails to inject because `LoginEndpoint` is a static class and generic logging requires a concrete type.
**Why it happens:** Wolverine HTTP endpoints are static methods. `ILogger<T>` works fine with static classes — the type parameter is just used for log category naming. This is NOT actually a problem with Wolverine, but developers sometimes overthink it.
**How to avoid:** Simply inject `ILogger<LoginEndpoint>` as a parameter — Wolverine resolves it from DI just like any other service.
**Warning signs:** None — this works fine, just noted for clarity.

## Code Examples

### Example 1: Thin Handler for Simple CRUD (Claude's Discretion)

For features that are truly thin pass-through (e.g., `GetRoles` which just queries and returns), keep the handler minimal. Don't add artificial complexity.

```csharp
// File: Auth.Application/Features/GetRoles.cs
using Auth.Application.Repositories;
using Auth.Contracts.Dtos;
using Microsoft.AspNetCore.Authorization;
using Microsoft.AspNetCore.Http;
using Wolverine.Http;

namespace Auth.Application.Features;

public static class GetRolesEndpoint
{
    [WolverineGet("/api/admin/roles")]
    [Authorize]
    public static async Task<IResult> Get(IAuthRepository repository)
    {
        var roles = await repository.GetRolesWithPermissionsAsync();

        var result = roles.Select(r => new RoleDto(
            r.Id,
            r.Name,
            r.Description,
            r.IsSystem,
            r.RolePermissions.Select(rp => new PermissionDto(
                rp.Permission.Id,
                rp.Permission.Module.ToString(),
                rp.Permission.Action.ToString(),
                rp.Permission.Description)).ToList())).ToList();

        return Results.Ok(result);
    }
}
```

**Recommendation for thin handlers:** Even for simple CRUD, having the mapping logic in the handler (not the repository) keeps the repository truly thin. The handler still owns the "shape" of the response. No validator is needed for GET endpoints with no body.

### Example 2: Repository Implementation in Infrastructure

```csharp
// File: Auth.Infrastructure/Repositories/AuthRepository.cs
using Auth.Application.Repositories;
using Auth.Domain.Entities;
using Microsoft.EntityFrameworkCore;

namespace Auth.Infrastructure.Repositories;

public sealed class AuthRepository : IAuthRepository
{
    private readonly AuthDbContext _dbContext;

    public AuthRepository(AuthDbContext dbContext)
    {
        _dbContext = dbContext;
    }

    public async Task<User?> GetUserWithRolesAndPermissionsAsync(string email)
    {
        return await _dbContext.Users
            .Include(u => u.UserRoles)
                .ThenInclude(ur => ur.Role)
                    .ThenInclude(r => r.RolePermissions)
                        .ThenInclude(rp => rp.Permission)
            .FirstOrDefaultAsync(u => u.Email == email);
    }

    public async Task SaveChangesAsync(CancellationToken ct = default)
    {
        await _dbContext.SaveChangesAsync(ct);
    }

    // ... other methods
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| MediatR + Controller + Handler | Wolverine.Http vertical slice (no mediator) | Wolverine docs recommend this | Already using Wolverine; this phase makes it properly vertical |
| Service layer pattern (IService + ServiceImpl) | Feature-based handlers with direct logic | Industry trend since ~2022 (Jimmy Bogard VSA) | Eliminates indirection, improves locality of behavior |
| Repository returning IQueryable | Repository returning materialized data | Best practice for testability | Prevents EF Core leaking into Application |

**Deprecated/outdated:**
- The "thin controller + fat service + repository" pattern the codebase currently uses is the old MVC-era approach. The Wolverine team specifically recommends against it, preferring vertical slice with logic in the endpoint handler.

## Open Questions

1. **IAuthRepository granularity: single interface or split?**
   - What we know: Auth module has ~13 features touching Users, Roles, Permissions, RefreshTokens, and SystemSettings. A single `IAuthRepository` would have ~20+ methods.
   - What's unclear: Whether one large interface or multiple focused interfaces (`IUserRepository`, `IRoleRepository`, `IRefreshTokenRepository`) is preferred.
   - Recommendation: Start with a single `IAuthRepository`. If it grows unwieldy, split later. The user's CONTEXT.md says "thin repository interfaces" (plural), suggesting splitting is acceptable. For this phase, a single interface per module keeps DI registration simpler.

2. **Audit module: rename IAuditReadContext to IAuditReadRepository?**
   - What we know: `IAuditReadContext` already serves as a thin read-only data access interface. It returns `IQueryable<T>` which is unusual for a repository.
   - What's unclear: Whether to keep the current `IQueryable` pattern (better for dynamic query composition in audit endpoints) or switch to materialized query methods.
   - Recommendation: Rename to `IAuditReadRepository` for naming consistency, but KEEP the `IQueryable<T>` pattern. The audit endpoints build complex dynamic queries with multiple optional filters and cursor pagination — materializing all combinations into repository methods would be impractical.

3. **PasswordHasher: singleton vs scoped lifetime?**
   - What we know: Currently registered as `AddSingleton<PasswordHasher>()`. The `PasswordHasher` class is stateless (no injected dependencies, pure CPU computation).
   - What's unclear: Whether to keep as singleton when adding the `IPasswordHasher` interface.
   - Recommendation: Keep as singleton — it's stateless and thread-safe. Register as `AddSingleton<IPasswordHasher, PasswordHasher>()`.

4. **JwtService uses AuthDbContext for SystemSettings — how to handle?**
   - What we know: `JwtService` reads token lifetime configuration from `SystemSettings` table via `AuthDbContext`. This is a legitimate infrastructure concern.
   - What's unclear: Whether `JwtService` should use the repository interface or keep its direct `AuthDbContext` dependency.
   - Recommendation: `JwtService` stays in Infrastructure and can keep its direct `AuthDbContext` dependency. It doesn't need to go through `IAuthRepository` — it's infrastructure code accessing infrastructure data. The `IJwtService` interface in Application is sufficient abstraction for handlers.
