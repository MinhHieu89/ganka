  <behavior>
    - Test: Adding a new Refraction to a Visit via AddRefraction() and saving via UnitOfWork succeeds (existing test Handle_ManifestRefraction_StoresOnVisit already covers domain logic; this is a config fix)
    - The fix itself is an EF Core configuration change, not application logic -- existing unit tests validate the handler behavior. The root cause is EF Core cannot track entities added to private backing fields without explicit PropertyAccessMode.Field configuration.
  </behavior>
  <action>
    In `VisitConfiguration.cs`, add backing field access mode configuration to all three navigation properties that use private backing fields (_refractions, _diagnoses, _amendments).

    For each HasMany navigation, after the existing configuration chain, add:
    ```csharp
    builder.Navigation(v => v.Refractions)
        .UsePropertyAccessMode(PropertyAccessMode.Field);

    builder.Navigation(v => v.Diagnoses)
        .UsePropertyAccessMode(PropertyAccessMode.Field);

    builder.Navigation(v => v.Amendments)
        .UsePropertyAccessMode(PropertyAccessMode.Field);
    ```

    This tells EF Core to use the private `_refractions`, `_diagnoses`, and `_amendments` backing fields directly when materializing and tracking entities, rather than trying to go through the read-only IReadOnlyCollection properties.

    **Why this fixes the 500 error:** Without PropertyAccessMode.Field, EF Core cannot add newly created Refraction entities to the change tracker when they are added via `_refractions.Add()` through the Visit.AddRefraction() domain method. The change tracker sees the entities but cannot persist them, resulting in an InvalidOperationException at SaveChanges time (surfaced as HTTP 500).

    After making the config change, run the existing 44 unit tests to confirm no regressions:
    ```bash
    dotnet test backend/tests/Clinical.Unit.Tests --no-build -x
    ```

    Then rebuild to verify compilation:
    ```bash
    dotnet build backend/Ganka28.slnx
    ```

    No new migration is needed -- this is a runtime configuration change, not a schema change.
