<objective>
Fix the shared DbUpdateConcurrencyException root cause blocking all child-entity mutations (refraction save, diagnosis add, amendment create).

Purpose: All three mutations fail with HTTP 500 because EF Core does not track new child entities added through the Visit aggregate's private backing fields. When `visit.AddRefraction(refraction)` adds to `_refractions`, the change tracker sees the Visit as Modified but the new Refraction entity is never explicitly registered as Added. EF Core then generates UPDATE instead of INSERT, which affects 0 rows and triggers the concurrency check. This is the same issue the Patient module solved by using direct `allergyRepository.Add(allergy)` instead of going through the aggregate's backing field for persistence.

Output: Three handlers fixed, repository extended with child-entity Add methods, all existing + new tests passing.
</objective>
