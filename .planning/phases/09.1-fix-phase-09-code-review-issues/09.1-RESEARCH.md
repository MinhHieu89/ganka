# Phase 9.1: Fix Phase 09 Code Review Issues - Research

**Researched:** 2026-03-14
**Domain:** Bug fixes across domain, application, infrastructure, presentation, and frontend layers of the Treatment module
**Confidence:** HIGH

## Summary

This phase addresses 32 issues (13 critical, 19 important) found during Phase 9 code review across all layers of the Treatment Protocols module. The issues span security gaps (missing permission-based authorization), domain logic errors (wrong session numbering, missing interval enforcement, missing deduction validation), a migration inconsistency, frontend wiring gaps (non-functional buttons, wrong JSON keys, broken QR flow), missing i18n, and test coverage gaps.

The codebase already has all necessary patterns established in other modules. Permission-based authorization uses `RequirePermissions()` extension method with `Permissions.Treatment.*` constants (already defined). Frontend permission checks use `authStore.permissions`. The i18n system uses `react-i18next` with `useTranslation()`. All fixes are pattern-following, not pattern-creating.

**Primary recommendation:** Fix in layer order (domain -> application -> infrastructure -> frontend -> tests) since upper layers depend on lower layers being correct first.

### Anti-Patterns to Avoid
- **Trusting client-provided user IDs:** RecordTreatmentSession currently accepts `PerformedById` from the client. Must use `ICurrentUser.UserId` server-side (issue #7).
- **Hardcoded fallback values:** `RequestCancellation` uses `?? 15m` when protocol not found. Should return error instead (issue #18).
- **Counting all sessions including cancelled:** `_sessions.Count + 1` for session numbering. Must filter to non-cancelled (issue #2).
- **Mutating state before validating preconditions:** `SwitchTreatmentType` calls `MarkAsSwitched()` before verifying the new template exists (issue #21).

## Common Pitfalls

### Pitfall 1: Session Numbering After Cancellation
**What goes wrong:** `_sessions.Count + 1` includes cancelled sessions, so if session 2 is cancelled and a new session is recorded, it gets numbered 4 instead of 3.
**Fix:** `_sessions.Count(s => s.Status != SessionStatus.Cancelled) + 1`
**Warning signs:** Gap in session numbers visible in UI.

### Pitfall 2: Modify() Not Checking Auto-Completion
**What goes wrong:** After `Modify(totalSessions: 2)` on a package with 2 completed sessions, `IsComplete` returns true but `Status` remains `Active`.
**Fix:** Add auto-completion check at the end of `Modify()`:
```csharp
if (IsComplete && Status == PackageStatus.Active)
{
    Status = PackageStatus.Completed;
    AddDomainEvent(new TreatmentPackageCompletedEvent(...));
}
```

### Pitfall 3: Deduction Validation at Domain vs Application
**What goes wrong:** `TreatmentPackage.RequestCancellation()` accepts any `deductionPercent` 0-100 but TRT-09 requires 10-20%.
**Fix:** Tighten domain validation to 10-20 range. The validator on `ApproveCancellationCommand` already enforces 10-20, but the domain method allows 0-100.

### Pitfall 4: Migration Regeneration May Reset Data
**What goes wrong:** Deleting and recreating migration can cause data loss if database already has data.
**Investigation:** The BranchId column IS present in the current migration file (line 68 of InitialTreatment.cs). The CONTEXT.md says it's missing, but my investigation shows it IS there. This needs careful verification at implementation time.
**Recommendation:** Before regenerating, verify the actual database state. If BranchId exists in both migration and snapshot, the issue may have already been fixed or may be about a different column. Check the model snapshot against the actual entity properties.

### Pitfall 5: JSON Key Mismatch Between Form Serialization and Card Display
**What goes wrong:** `TreatmentSessionForm` serializes IPL params as `{ energy, pulseCount, spotSize, treatmentZones }` but `TreatmentSessionCard` reads `{ energy, pulses, spotSize, zones }`.
**Fix:** Align `TreatmentSessionCard` to read `pulseCount` and `treatmentZones` (matching the form serialization keys), OR align both to the domain value object property names (`EnergyJoules`, `PulseCount`, `SpotSize`, `TreatmentZones`).
**Recommendation:** Use camelCase of domain names: `energyJoules`/`pulseCount`/`spotSize`/`treatmentZones` for IPL, `wavelengthNm`/`powerMw`/`durationMinutes`/`treatmentArea` for LLLT. The form already uses shorter keys (`energy`, `pulseCount`) which is acceptable, but the card must match what the form serializes.

### Pitfall 6: QR Token Registration Requires Backend Endpoint
**What goes wrong:** `SessionOsdiCapture` generates a UUID client-side but never registers it with the backend. When a patient scans the QR, the backend has no record of the token.
**Fix:** Need a new backend endpoint to register the OSDI token linked to the session/package, and the QR URL must use the correct route format (`/osdi/:token` not `/osdi/self-fill?token=`).

### Pitfall 7: Entity Inheritance Change Requires Migration
**What goes wrong:** Making `TreatmentSession` and `SessionConsumable` extend `Entity` adds `CreatedAt`, `UpdatedAt`, `IsDeleted` columns. `TreatmentSession` already has `CreatedAt` manually, so only `UpdatedAt` and `IsDeleted` need adding.
**Fix:** After changing entity inheritance, generate a new migration to add the missing columns.

if (MinIntervalDays > 0)
{
    var lastCompleted = _sessions
        .Where(s => s.Status == SessionStatus.Completed && s.CompletedAt.HasValue)
        .OrderByDescending(s => s.CompletedAt)
        .FirstOrDefault();

    if (lastCompleted is not null)

        var referenceDate = scheduledAt ?? DateTime.UtcNow;
        var daysSinceLast = (int)(referenceDate - lastCompleted.CompletedAt!.Value).TotalDays;

    }
}

## State of the Art

| Old Approach | Current Approach | Impact |
|--------------|------------------|--------|
| `RequireAuthorization()` only | `RequirePermissions()` per endpoint | Granular RBAC per TRT-10 |
| Client-provided PerformedById | `ICurrentUser.UserId` from JWT | Prevents impersonation |
| Soft interval warning only | Domain-level enforcement with override | Prevents accidental too-early sessions |
| 0-100% deduction range | 10-20% domain validation | Matches TRT-09 business rule |

## Open Questions

1. **Migration BranchId Status**
   - What we know: BranchId IS present in the current migration file (line 68) and model snapshot (line 231)
   - What's unclear: The CONTEXT.md says "Migration: BranchId missing" but evidence shows it exists. May refer to a different issue or may have been fixed during Phase 9.
   - Recommendation: At implementation time, verify actual database state with `dotnet ef migrations list` and check if the column exists in the running database. If it does, this issue may be a no-op.

2. **QR OSDI Token Backend Endpoint**
   - What we know: Frontend generates UUID but never registers it. Need a new endpoint.
   - What's unclear: Should this be a lightweight key-value store, a new entity, or a temp token table?
   - Recommendation: Create a simple `POST /api/treatments/osdi-tokens` endpoint that stores `{token, packageId, sessionContext}` with TTL expiration. Keep it minimal -- it's a self-fill capture link.

3. **Version History API Endpoint**
   - What we know: `VersionHistoryDialog` exists but there's no backend endpoint to fetch version history.
   - What's unclear: Should it return the `ProtocolVersion` entities directly or a dedicated DTO?
   - Recommendation: Add `GET /api/treatments/packages/{id}/versions` returning `List<ProtocolVersionDto>` from the already-loaded `_versions` collection on the package aggregate.

4. **ApproveCancellation Deduction Override**
   - What we know: Manager provides `DeductionPercent` in command, validator enforces 10-20%, but handler ignores it.
   - What's unclear: Should we recalculate refund amount with the manager's override, or use the original request's deduction?
   - Recommendation: Implement the override -- recalculate refund using manager's `DeductionPercent` and update the `CancellationRequest` entity before approving. This is the whole point of the manager review workflow.
