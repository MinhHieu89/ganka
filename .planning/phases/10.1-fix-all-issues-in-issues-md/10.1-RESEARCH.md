**Researched:** 2026-03-19
**Domain:** Multi-module bug fixes and improvements across a .NET 9 + React/TypeScript clinic management system
**Confidence:** HIGH

## Summary

This phase addresses 24 reported issues (21 bugs, 3 improvements) spanning 12 modules. Research investigated each issue's root cause by examining the actual source files. Most issues have clear, identifiable root causes in the codebase and can be fixed with targeted changes. The bugs fall into several categories: data filtering issues (backend queries), missing i18n translations, UI state bugs, broken navigation/routing, upload infrastructure problems (likely Azure config), stock status logic errors, and missing feature implementations (dashboard stats, remember login persistence).

The most critical finding is that several issues share root causes: TASK-16 and TASK-23 (OTC catalog not loading) both trace to `SearchDrugCatalogHandler` returning empty array when search term is empty; TASK-21 and TASK-25 (stock=0 status/alerts) both trace to the `isLowStock` flag not distinguishing between "low" and "out of stock"; TASK-8 and TASK-10 (upload failures) likely share an Azure Blob Storage configuration issue in development (`UseDevelopmentStorage=true` fallback).

**Primary recommendation:** Group fixes by module with shared root causes merged. Start with high-confidence fixes (data queries, i18n, UI state), then tackle investigation-first issues (uploads, OTC catalog, dashboard stats).

## Root Cause Analysis by Task

### TASK-2: Mouse scroll in dropdown (Patient allergy)
**Root cause:** The `CommandList` component from cmdk/shadcn uses a custom scroll container. Mouse wheel events may not propagate properly if the list is inside a `Popover`.
**Files:** `frontend/src/features/patient/components/PatientRegistrationForm.tsx` (uses `Command`/`CommandGroup`/`CommandList` inside `Popover`)
**Fix:** Ensure `CommandList` has explicit `max-height` and `overflow-y: auto` CSS. The `CommandGroup` in the allergy selector may be missing `CommandList` wrapper -- the allergy dropdown uses `Command > CommandGroup` but the OTC drug combobox pattern (which works) uses `CommandGroup className="max-h-52 overflow-y-auto"`.
**Confidence:** MEDIUM -- need to verify the exact DOM structure at runtime.

### TASK-3: Edit button visible in edit mode (Patient Detail)
**Root cause:** `PatientProfileHeader` always renders the Edit button regardless of `isEditing` state. The `onEdit` callback is passed but there's no `isEditing` prop to conditionally hide it.
**Files:** `frontend/src/features/patient/components/PatientProfileHeader.tsx` (line 197), `PatientProfilePage.tsx` (line 113-116)
**Fix:** Pass `isEditing` prop to `PatientProfileHeader` and conditionally hide the Edit button when `isEditing === true`.
**Confidence:** HIGH

### TASK-5: Book Appointment doesn't auto-fill patient (HIGH PRIORITY)
**Root cause:** `PatientAppointmentTab.tsx` navigates to `/appointments` without passing patient data. The `AppointmentBookingDialog` supports `defaultDoctorId` and `defaultStartTime` but has NO `defaultPatientId` prop. Navigation uses `navigate({ to: "/appointments" })` with no query params.
**Files:** `frontend/src/features/patient/components/PatientAppointmentTab.tsx` (line 112-114), `frontend/src/features/scheduling/components/AppointmentBookingDialog.tsx`
**Fix:** Two options: (1) Navigate with query params `?patientId={id}` and have the appointments page read + pre-fill, or (2) Add `defaultPatientId`/`defaultPatientName` props to `AppointmentBookingDialog` and open it directly from patient detail. Option 2 is cleaner. Need to add `patientId` prop to AppointmentBookingDialog, pre-populate the patient search field.
**Confidence:** HIGH

### TASK-7: Inactive patients hidden when no filter
**Root cause:** `PatientRepository.GetPagedAsync()` has `.Where(p => p.IsActive)` hardcoded at line 61 of `PatientRepository.cs`. There is no `isActive` parameter in the query. The `PatientListPage.tsx` has no status filter dropdown.
**Files:** `backend/src/Modules/Patient/Patient.Infrastructure/Repositories/PatientRepository.cs` (line 61), `backend/src/Modules/Patient/Patient.Application/Features/GetPatientList.cs`, `frontend/src/features/patient/components/PatientListPage.tsx`, `frontend/src/features/patient/api/patient-api.ts`
**Fix:** (1) Add `bool? IsActive` param to `GetPatientListQuery`, (2) Update `GetPagedAsync` to apply filter only when `IsActive` has a value, (3) Add status filter dropdown to `PatientListPage` defaulting to "Active", (4) Pass filter value through API. **Per issue spec:** default should show Active, but "All" option must be available.
**Confidence:** HIGH

### TASK-8: Cannot upload patient photo

**Files:** `backend/src/Shared/Shared.Infrastructure/Services/AzureBlobService.cs`, `backend/src/Modules/Patient/Patient.Application/Features/UploadPatientPhoto.cs`, `frontend/src/features/patient/api/patient-api.ts` (uploadPatientPhoto function)
**Fix:** Investigation-first. Check if Azurite is running, check backend logs for Azure Storage errors. If dev environment, may need to either (a) configure Azurite, (b) add a local file system fallback for dev, or (c) configure real Azure connection string.
**Confidence:** MEDIUM -- root cause is likely Azure config, but need to verify.

### TASK-14: Remember Login not working
**Root cause:** Found the critical issue. `authStore.ts` uses plain `zustand` with NO persistence middleware. The `create<AuthState>()` call on line 19 stores state only in memory. When the page reloads, all auth state (user, accessToken) is lost. The backend properly stores refresh tokens with `RememberMe` flag and extended lifetime, but the frontend never attempts to restore the session on page load.
**Files:** `frontend/src/shared/stores/authStore.ts`, `frontend/src/features/auth/hooks/useAuth.ts`, `frontend/src/features/auth/api/auth-api.ts`
**Fix:** On app initialization, attempt a silent refresh (`POST /api/auth/refresh`) using the HTTP-only cookie that contains the refresh token. If successful, restore auth state. The backend already has the refresh endpoint and stores refresh tokens. The frontend just needs to call `silentRefresh()` (already exists in `auth-api.ts` line 75) during app bootstrap.
**Confidence:** HIGH

### TASK-16 + TASK-23: OTC catalog not loading / cannot search medications

**Files:** `backend/src/Modules/Pharmacy/Pharmacy.Application/Features/SearchDrugCatalog.cs` (line 19-20), `frontend/src/features/pharmacy/api/pharmacy-api.ts` (line 282-288), `frontend/src/features/pharmacy/components/OtcSaleForm.tsx`
**Fix:** Change `SearchDrugCatalogHandler` to return all active drugs when search term is empty (or add a separate `GetAllDrugs` query). Alternatively, change frontend to use the paginated endpoint `/api/pharmacy/drugs` which does return all drugs.
**Confidence:** HIGH

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Dropdown scroll | Custom scroll handler | Ensure shadcn `CommandList` wrapper is used | Already built into cmdk |
| Auth persistence | localStorage token storage | HTTP-only cookie refresh + `silentRefresh()` | Already exists in backend; more secure |
| Stock status display | Complex conditional rendering | Add `IsOutOfStock` boolean to DTO | Single source of truth |
| Dashboard stats | Hardcoded mock data | Real API endpoints with DB queries | Must show actual data |

## Common Pitfalls

### Pitfall 1: Azure Blob Storage in Development
**What goes wrong:** Upload endpoints fail silently because Azurite is not running
**Why it happens:** `AzureBlobService` falls back to `UseDevelopmentStorage=true` which requires Azurite
**How to avoid:** Either run Azurite (`npm install -g azurite && azurite`), or add a local file system fallback for development
**Warning signs:** 500 errors on upload endpoints, no error in frontend (generic "Failed to upload" message)

### Pitfall 2: Query Parameter Changes Breaking API Compatibility
**What goes wrong:** Adding new parameters to backend queries breaks existing callers
**Why it happens:** Wolverine handlers use record types -- adding required params breaks deserialization
**How to avoid:** Always use optional parameters with defaults (e.g., `bool? IsActive = null`)

### Pitfall 3: Zustand Store Persistence
**What goes wrong:** Adding persistence middleware to authStore could break existing login flow
**Why it happens:** Persisted stale tokens could cause auth loops
**How to avoid:** Don't persist the auth store. Instead, use silent refresh on app bootstrap via HTTP-only refresh token cookie.

### Pitfall 4: i18n Key Mismatches
**What goes wrong:** Adding Vietnamese keys but using wrong namespace or key path
**Why it happens:** Multiple translation namespaces (`patient`, `pharmacy`, `billing`, etc.) and nested key structures
**How to avoid:** Always check which `useTranslation("{namespace}")` the component uses before adding keys
