## Summary

This phase refactors the existing Ganka28 frontend to comprehensively use shadcn/ui patterns. The project already uses shadcn/ui primitives in `shared/components/ui/` (16 components), TanStack Table for data grids, React Hook Form + Zod for validation, and TanStack Start for file-based routing. The refactoring involves three major areas: (1) replacing the dashboard shell with shadcn's `dashboard-01` block layout, (2) replacing the login page with shadcn's `login-04` block layout, and (3) establishing a wrapper component pattern for safe long-term maintenance.

A critical discovery is that shadcn/ui has evolved its form pattern. The project currently uses manual `register()`/`setValue()` calls with raw `Label`/`Input` components and inline `<p className="text-sm text-destructive">` error messages. The modern shadcn/ui pattern (as of October 2025) uses `Field`, `FieldLabel`, `FieldDescription`, `FieldError`, and `FieldGroup` components with React Hook Form's `Controller` -- providing better accessibility (`data-invalid`, `aria-invalid`) and consistent error display. The user's decision says "shadcn Form component" which maps to this newer Field-based pattern.

**Primary recommendation:** Install the `field` and `alert-dialog` components via CLI, use `dashboard-01` and `login-04` blocks as reference patterns (adapt to project structure rather than installing raw), build wrapper components for all shadcn primitives, and migrate all forms to the Field + Controller pattern.

### Pattern 1: Wrapper Component Pattern
**What:** Every shadcn/ui primitive gets a thin wrapper in `shared/components/` that re-exports the component(s) from `ui/`.
**When to use:** Always. All feature code imports from wrappers, never from `ui/` directly.
**Example:**
```typescript
// shared/components/Button.tsx
export { Button, buttonVariants } from "@/shared/components/ui/button"
export type { ButtonProps } from "@/shared/components/ui/button"
```

```typescript
// shared/components/Card.tsx
export {
  Card,
  CardHeader,
  CardTitle,
  CardDescription,
  CardContent,
  CardFooter,
} from "@/shared/components/ui/card"
```

```typescript
// shared/components/AlertDialog.tsx
export {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
  AlertDialogTrigger,
} from "@/shared/components/ui/alert-dialog"
```

**Key insight:** Wrappers start as pure re-exports. When the project needs customization (adding a default prop, restricting variants, wrapping with a context), the wrapper absorbs the change in ONE place. Upgrading shadcn/ui later only affects the `ui/` files.

### Pattern 2: shadcn Form Field Pattern (Field + Controller)
**What:** Use shadcn's `Field`, `FieldLabel`, `FieldError`, `FieldGroup` with React Hook Form `Controller` for consistent, accessible form fields.
**When to use:** All forms in the project. Replaces current inline `<Label>` + `<Input>` + `<p className="text-destructive">` pattern.
**Example:**
```typescript
// Source: https://ui.shadcn.com/docs/forms/react-hook-form
import { Controller, useForm } from "react-hook-form"
import { zodResolver } from "@hookform/resolvers/zod"
import { z } from "zod"
import { Field, FieldLabel, FieldError, FieldGroup } from "@/shared/components/Field"
import { Input } from "@/shared/components/Input"
import { Button } from "@/shared/components/Button"

const schema = z.object({
  username: z.string().min(1, "Required"),
  password: z.string().min(8, "Minimum 8 characters"),
})

function LoginForm() {
  const form = useForm({ resolver: zodResolver(schema) })

  return (
    <form onSubmit={form.handleSubmit(onSubmit)}>
      <FieldGroup>
        <Controller
          name="username"
          control={form.control}
          render={({ field, fieldState }) => (
            <Field data-invalid={fieldState.invalid}>
              <FieldLabel htmlFor={field.name}>Username</FieldLabel>
              <Input
                {...field}
                id={field.name}
                aria-invalid={fieldState.invalid}
              />
              {fieldState.invalid && (
                <FieldError errors={[fieldState.error]} />
              )}
            </Field>
          )}
        />
      </FieldGroup>
      <Button type="submit">Submit</Button>
    </form>
  )
}
```

### Pattern 3: shadcn Data Table Pattern
**What:** Generic `DataTable<TData, TValue>` component using TanStack Table + shadcn Table primitives.
**When to use:** All tables (UserTable, RoleTable, AuditLogTable).
**Example:**
```typescript
// Source: https://ui.shadcn.com/docs/components/base/data-table
import {
  ColumnDef,
  flexRender,
  getCoreRowModel,
  useReactTable,
} from "@tanstack/react-table"
import {
  Table,
  TableBody,
  TableCell,
  TableHead,
  TableHeader,
  TableRow,
} from "@/shared/components/Table"

interface DataTableProps<TData, TValue> {
  columns: ColumnDef<TData, TValue>[]
  data: TData[]
}

export function DataTable<TData, TValue>({
  columns,
  data,
}: DataTableProps<TData, TValue>) {
  const table = useReactTable({
    data,
    columns,
    getCoreRowModel: getCoreRowModel(),
  })

  return (
    <div className="overflow-hidden border">
      <Table>
        <TableHeader>
          {table.getHeaderGroups().map((headerGroup) => (
            <TableRow key={headerGroup.id}>
              {headerGroup.headers.map((header) => (
                <TableHead key={header.id}>
                  {header.isPlaceholder
                    ? null
                    : flexRender(header.column.columnDef.header, header.getContext())}
                </TableHead>
              ))}
            </TableRow>
          ))}
        </TableHeader>
        <TableBody>
          {table.getRowModel().rows?.length ? (
            table.getRowModel().rows.map((row) => (
              <TableRow key={row.id}>
                {row.getVisibleCells().map((cell) => (
                  <TableCell key={cell.id}>
                    {flexRender(cell.column.columnDef.cell, cell.getContext())}
                  </TableCell>
                ))}
              </TableRow>
            ))
          ) : (
            <TableRow>
              <TableCell colSpan={columns.length} className="h-24 text-center">
                No results.
              </TableCell>
            </TableRow>
          )}
        </TableBody>
      </Table>
    </div>
  )
}
```

### Anti-Patterns to Avoid
- **Direct `ui/` imports:** Feature code must NEVER import from `@/shared/components/ui/*`. Always go through wrapper in `@/shared/components/*`.
- **Inline error messages:** Do NOT use `<p className="text-sm text-destructive">{error}</p>`. Use `<FieldError errors={[fieldState.error]} />` from the Field component.
- **Manual register() with Field:** Field + Controller is the correct pattern. Do not mix `register()` with `<Field>` -- use `Controller` to get `field` and `fieldState` props.
- **Installing blocks directly into project:** Dashboard-01 and login-04 assume Next.js app directory. TanStack Start uses different file conventions. Study block code, then adapt manually.
- **Keeping Dialog for non-dismissible modals:** AlertDialog is semantically correct and handles ESC prevention by default. Do not use Dialog with `hideCloseButton` + `onEscapeKeyDown={e => e.preventDefault()}`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Form field layout + errors | Custom `<Label>` + `<Input>` + `<p>` error | shadcn `Field` + `FieldLabel` + `FieldError` | Built-in `data-invalid`, `aria-invalid`, consistent error rendering |
| Non-dismissible modal | Dialog with hideCloseButton prop | AlertDialog component | Semantically correct, ESC prevention is default behavior |
| Data table boilerplate | Per-component table rendering | Generic `DataTable<T>` wrapper | Reduces ~50 lines of boilerplate per table |
| Accessible form composition | Manual aria attributes | Field component with Controller | Automatic `aria-invalid`, `data-invalid` state management |
| Sidebar layout shell | Custom flexbox layout | SidebarProvider + SidebarInset pattern | Built-in collapsible, mobile sheet, keyboard shortcuts |

**Key insight:** shadcn/ui components are designed to work together. The Field component + Controller handles accessibility automatically. The SidebarProvider handles responsive behavior. Fighting these patterns creates technical debt.

## Common Pitfalls

### Pitfall 1: Block Installation on Non-Next.js Projects
**What goes wrong:** Running `npx shadcn add dashboard-01` on a TanStack Start project creates files targeting Next.js app directory (`app/dashboard/page.tsx`).
**Why it happens:** Blocks assume a Next.js project structure with `app/` directory routing.
**How to avoid:** Do NOT install blocks directly. Instead: (1) study the block source code on ui.shadcn.com, (2) identify the component files (app-sidebar.tsx, site-header.tsx, etc.), (3) manually recreate them in the project's structure (`shared/components/`), adapting imports and routing.
**Warning signs:** Files appearing in `app/` directory outside of TanStack Start's route tree.

### Pitfall 2: Mixing Old Form Pattern with New Field Pattern
**What goes wrong:** Some forms use `register()` + inline errors, others use `Controller` + `<FieldError>`, creating inconsistency and confusion.
**Why it happens:** Incremental migration without a clean cutover.
**How to avoid:** Migrate ALL forms in a single wave. Define the form pattern once, apply everywhere. The new pattern uses `Controller` for every field.
**Warning signs:** Both `register("fieldName")` and `<Controller name="fieldName">` in the same codebase.

### Pitfall 3: Forgetting to Update Imports After Wrapper Creation
**What goes wrong:** Creating wrappers but leaving existing feature code importing from `ui/` directly.
**Why it happens:** Find-and-replace misses, or new components added without wrappers.
**How to avoid:** (1) Create ALL wrappers first, (2) do a bulk import replacement across the codebase, (3) add an ESLint rule or convention document. Grep for `from "@/shared/components/ui/` in feature code -- should return zero results.
**Warning signs:** `import { Button } from "@/shared/components/ui/button"` in any file outside `shared/components/`.

### Pitfall 4: Losing Existing Functionality During Layout Replacement
**What goes wrong:** Replacing AppShell/TopBar with dashboard-01 pattern but forgetting to preserve: session warning modal, language toggle, user dropdown, permission-based sidebar rendering, active route highlighting.
**Why it happens:** Block code is a clean starting point without project-specific features.
**How to avoid:** Create a checklist of existing behaviors BEFORE replacing: (1) sidebar collapse, (2) active route detection, (3) admin section conditional rendering, (4) user dropdown with logout, (5) language toggle, (6) session warning modal, (7) Ganka28 branding in sidebar header.
**Warning signs:** Missing functionality after visual refactoring "looks good."

### Pitfall 5: shadcn/ui Component Version Mismatch
**What goes wrong:** Existing `ui/` components (installed during Phase 01) may be slightly behind current shadcn/ui version. New components installed via CLI get the latest version, creating style/API inconsistencies.
**Why it happens:** shadcn/ui evolves rapidly. Components installed weeks apart may have different patterns (e.g., `forwardRef` vs direct function components).
**How to avoid:** After installing new components, run `npx shadcn@latest add --overwrite` on ALL existing components to bring them to the same version. The project's existing button.tsx still uses `forwardRef` which newer shadcn/ui components have dropped.
**Warning signs:** Some components using `React.forwardRef` while newer ones use direct function syntax with `data-slot`.

### Pitfall 6: Login Form Field Name Mismatch
**What goes wrong:** Login-04 block uses `email` field. The project requires `username` field (backend auth contract uses username/email interchangeably, but field should be labeled "Username").
**Why it happens:** Login blocks default to email-based authentication.
**How to avoid:** When adapting login-04, change the field name from `email` to `username`, update the input type from `email` to `text`, keep the Zod validation but adjust it (remove `.email()` validator, keep `.min(1)`).
**Warning signs:** Login form sending `email` field when backend expects `username` (or vice versa).

### AlertDialog for Session Warning (replacing Dialog)
```typescript
// Source: https://ui.shadcn.com/docs/components/radix/alert-dialog
import {
  AlertDialog,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
} from "@/shared/components/AlertDialog"
import { Button } from "@/shared/components/Button"

interface SessionWarningModalProps {
  open: boolean
  remainingSeconds: number
  onExtend: () => void
  onLogout: () => void
}

export function SessionWarningModal({
  open,
  remainingSeconds,
  onExtend,
  onLogout,
}: SessionWarningModalProps) {
  const { t } = useTranslation("auth")
  const timeDisplay = formatTime(remainingSeconds)

  return (
    <AlertDialog open={open}>
      <AlertDialogContent>
        <AlertDialogHeader>
          <AlertDialogTitle>{t("session.expiring")}</AlertDialogTitle>
          <AlertDialogDescription>
            {t("session.expiringMessage", { minutes: timeDisplay })}
          </AlertDialogDescription>
        </AlertDialogHeader>
        <div className="flex items-center justify-center py-4">
          <span className="text-4xl font-mono font-bold tabular-nums text-destructive">
            {timeDisplay}
          </span>
        </div>
        <AlertDialogFooter>
          <Button variant="outline" onClick={onLogout}>
            {t("session.logout")}
          </Button>
          <Button onClick={onExtend}>
            {t("session.extend")}
          </Button>
        </AlertDialogFooter>
      </AlertDialogContent>
    </AlertDialog>
  )
}
```

- `Form`, `FormField`, `FormItem`, `FormLabel`, `FormControl`, `FormMessage`: Replaced by Field component family. Still works but not the recommended pattern.
- `tailwindcss-animate`: Replaced by `tw-animate-css` for new Tailwind v4 projects.
- Dialog with `hideCloseButton` for non-dismissible modals: AlertDialog is the correct semantic choice.
