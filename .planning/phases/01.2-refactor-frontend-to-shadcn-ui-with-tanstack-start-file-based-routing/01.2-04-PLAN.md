    **Create generic DataTable<TData, TValue> component** in `shared/components/DataTable.tsx`:

    Based on shadcn Data Table pattern (research Pattern 3):
    ```typescript
    import {
      type ColumnDef,
      type Table as TanStackTable,
      flexRender,
    } from "@tanstack/react-table"
    import {
      Table,
      TableBody,
      TableCell,
      TableHead,
      TableHeader,
      TableRow,
    } from "@/shared/components/Table"

    interface DataTableProps<TData, TValue> {
      table: TanStackTable<TData>
      columns: ColumnDef<TData, TValue>[]
      emptyMessage?: string
      onRowClick?: (row: TData) => void
      rowClassName?: (row: TData) => string
      renderSubRow?: (row: TData) => React.ReactNode
    }
    ```

    The DataTable receives a pre-configured TanStack Table instance (not raw data+columns) so each consumer retains full control over sorting, pagination, expansion state, etc. The component only handles the rendering boilerplate (Table/TableHeader/TableBody/TableRow/TableCell/flexRender).

    Features the DataTable should support:
    - `emptyMessage` prop for custom empty state text
    - `onRowClick` for clickable rows (RoleTable uses this for selection)
    - `rowClassName` for custom row styling (RoleTable highlights selected row)
    - `renderSubRow` for expandable row content (AuditLogTable uses this for detail view)
    - Sortable column header detection (if column.getCanSort(), add cursor-pointer + sort icon)

    **Rebuild UserTable.tsx** to use DataTable:
    - Keep the existing column definitions (email, fullName, roles, isActive, actions)
    - Keep sorting state and useReactTable setup
    - Replace the ~55 lines of Table/TableHeader/TableBody/TableRow/TableCell rendering with `<DataTable table={table} columns={columns} emptyMessage="No users found" />`
    - All imports from wrappers (Badge, Button from shared/components/)

    **Rebuild RoleTable.tsx** to use DataTable:
    - Keep column definitions (name, description, isSystem, permissions count)
    - Replace rendering with `<DataTable table={table} columns={columns} onRowClick={onSelectRole} rowClassName={(row) => row.id === selectedRoleId ? "bg-accent" : "hover:bg-muted/50"} emptyMessage="No roles found" />`

    **Rebuild AuditLogTable.tsx** to use DataTable:
    - Keep column definitions (expander, timestamp, userEmail, action, entityName, entityId)
    - Keep expansion state, pagination controls
    - Replace table rendering with `<DataTable table={table} columns={columns} renderSubRow={(row) => <AuditLogDetail log={row} />} onRowClick={(_, tanstackRow) => tanstackRow.toggleExpanded()} emptyMessage={t("noData")} />`
    - Keep the pagination controls (Previous/Next buttons) below the DataTable -- these are NOT part of DataTable (they're AuditLogTable-specific because of manual pagination)

    NOTE: The DataTable interface may need refinement during implementation. The key constraint is: the TanStack Table instance is passed in (not created inside DataTable), so consumers keep full control. DataTable only reduces rendering boilerplate.

    **UserFormDialog.tsx -- Migrate to Field+Controller pattern:**

    Current pattern uses `register()` + inline `<p className="text-sm text-destructive">` errors. Replace with Controller + Field:

    For each text field (email, fullName, password):
    ```typescript
    <Controller
      name="email"
      control={createForm.control}
      render={({ field, fieldState }) => (
        <Field data-invalid={fieldState.invalid || undefined}>
          <FieldLabel htmlFor={field.name}>{t("admin.email")}</FieldLabel>
          <Input
            {...field}
            id={field.name}
            type="email"
            aria-invalid={fieldState.invalid || undefined}
          />
          {fieldState.error && (
            <FieldError>{getErrorMessage(fieldState.error)}</FieldError>
          )}
        </Field>
      )}
    />
    ```

    For the isActive checkbox (edit mode):
    ```typescript
    <Controller
      name="isActive"
      control={editForm.control}
      render={({ field }) => (
        <div className="flex items-center space-x-2">
          <Checkbox
            id="isActive"
            checked={field.value}
            onCheckedChange={field.onChange}
          />
          <Label htmlFor="isActive" className="font-normal cursor-pointer">
            {t("admin.active")}
          </Label>
        </div>
      )}
    />
    ```

    For the roleIds checkbox group, keep the current `toggleRole` + `setValue` pattern but wrap in a Field for error display:
    ```typescript
    <Field data-invalid={currentErrors.roleIds ? true : undefined}>
      <FieldLabel>{t("admin.selectRoles")}</FieldLabel>
      <div className="border p-3 space-y-2 max-h-48 overflow-y-auto">
        {roles.map(role => ...checkbox...)}
      </div>
      {currentErrors.roleIds && (
        <FieldError>{tCommon("validation.required")}</FieldError>
      )}
    </Field>
    ```

    Keep the dual form pattern (createForm + editForm) as the user explicitly chose this approach in Phase 01-05.

    All imports from wrappers: Field, FieldLabel, FieldError from Field wrapper; Input, Label, Checkbox, Button, Badge, Dialog* from respective wrappers.
