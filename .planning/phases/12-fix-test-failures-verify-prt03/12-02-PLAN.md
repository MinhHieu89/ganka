Per D-01 and D-02: fix the test host setup, not the tests. The tests are valid.

The issue: `WebApplicationFactory.Services` triggers `CreateHost()` which calls `BuildAsync()` but NOT `StartAsync()`. Wolverine requires `StartAsync()` to initialize its message bus. The `InitializeAsync()` method (from `IAsyncLifetime`) accesses `Services` but never explicitly starts the host.

Fix the `InitializeAsync()` method in `CustomWebApplicationFactory`:

1. Before accessing `Services`, ensure the host is fully started by calling:
   ```csharp
   public async Task InitializeAsync()
   {
       // Force the host to start so Wolverine initializes its message bus
       // WebApplicationFactory.Services only builds the host; StartAsync runs hosted services
       var server = Server; // This triggers CreateHost -> BuildAsync
       await server.Host.StartAsync();

       // Now seed test data (host is fully started, Wolverine is initialized)
       using var scope = Services.CreateScope();
       var authDb = scope.ServiceProvider.GetRequiredService<AuthDbContext>();
       // ... rest of seeding code unchanged ...
   }
   ```

2. Update `DisposeAsync` to stop the host gracefully before database cleanup:
   ```csharp
   async Task IAsyncLifetime.DisposeAsync()
   {
       try
       {
           // Stop the host first
           await Server.Host.StopAsync();
       }
       catch { /* Ignore shutdown errors */ }

       // Then clean up test database
       try
       {
           using var ctx = CreateDbContext<AuthDbContext>(TestConnectionString);
           await ctx.Database.EnsureDeletedAsync();
       }
       catch { /* Ignore disposal errors */ }
   }
   ```

Key insight: `Server` property on `WebApplicationFactory` creates and returns the `TestServer`. Accessing `Server.Host` gives the `IHost` instance. Calling `StartAsync()` on it runs all `IHostedService.StartAsync` including Wolverine's initialization.
