Per D-03 and D-10: fix the production handler code to enforce UTC, never relax the tests.

The EF Core global UTC converter (in ModelBuilderExtensions.ApplySharedConventions) only applies when data is read from the database. In unit tests, appointments come from in-memory mocks with DateTimeKind.Unspecified. The handlers must ensure UTC kind when mapping to DTOs.

**GetAppointmentsByPatient.cs** — In the `Select` lambda where `AppointmentDto` is constructed, change:
```csharp
a.StartTime,
a.EndTime,
```
to:
```csharp
DateTime.SpecifyKind(a.StartTime, DateTimeKind.Utc),
DateTime.SpecifyKind(a.EndTime, DateTimeKind.Utc),
```

**GetAppointmentsByDoctor.cs** — Identical change in the `Select` lambda:
```csharp
DateTime.SpecifyKind(a.StartTime, DateTimeKind.Utc),
DateTime.SpecifyKind(a.EndTime, DateTimeKind.Utc),
```

This is safe because all DateTimes stored in the database are UTC (enforced by the shared convention). SpecifyKind is a no-op when the Kind is already Utc.
