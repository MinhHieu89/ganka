<objective>
Build the Audit module (field-level audit trail + access logging), architectural foundation interfaces (ACL adapters, Azure Blob, template engine, data export), and ICD-10 seed data.

Purpose: Audit logging is a legal requirement for medical records (10+ year retention, immutable). This plan delivers the EF Core interceptor that automatically captures field-level changes on IAuditable entities, the access logging middleware for tracking who views what, and the architectural scaffolding (Azure Blob service, ACL adapter pattern, template engine interface, data export) that later phases build on. ICD-10 ophthalmology codes are seeded from Day 1 for So Y te data readiness.

Output: Automatic audit logging on all entity changes, access logging on all HTTP requests, immutable audit storage, ICD-10 codes in database, and all architecture foundation interfaces ready for implementation in later phases.
</objective>

Build audit logging infrastructure and API:

**Audit Domain Entities:**

1. `AuditLog` -- NOT an AggregateRoot, standalone entity (no BranchId filter needed -- audit logs are immutable cross-tenant):
   - Properties: Id (Guid), Timestamp (DateTime), UserId (Guid), UserEmail (string), EntityName (string), EntityId (string), Action (AuditAction enum), Changes (string -- JSON serialized list of { PropertyName, OldValue, NewValue }), BranchId (Guid -- stored for reference but NOT filtered)
   - No setters exposed -- create via factory method only
   - `static AuditLog Create(userId, userEmail, entityName, entityId, action, changes, branchId)`

2. `AccessLog` -- standalone entity:
   - Properties: Id (Guid), Timestamp (DateTime), UserId (Guid?), UserEmail (string?), Action (AccessAction enum), Resource (string -- e.g., "/api/patients/123"), IpAddress (string), UserAgent (string), StatusCode (int), BranchId (Guid?)
   - Factory method: `static AccessLog Create(...)`

3. `AuditAction` enum: Created, Updated, Deleted
4. `AccessAction` enum: Login, LoginFailed, Logout, ViewRecord, ApiRequest

**Audit DTOs (in Audit.Contracts):**

5. `AuditLogDto` -- record: Id, Timestamp, UserEmail, EntityName, EntityId, Action, Changes (as structured list)
6. `AuditLogQuery` -- record: UserId?, ActionType?, EntityName?, DateFrom?, DateTo?, PageNumber (default 1), PageSize (default 50)
7. `AccessLogDto` -- record: Id, Timestamp, UserEmail, Action, Resource, IpAddress, StatusCode

**Audit.Infrastructure/AuditDbContext.cs:**

8. Schema: "audit"
9. DbSets: AuditLogs, AccessLogs
10. NO query filters -- audit logs are immutable and must be queryable across all branches
11. Configure append-only behavior: In AuditLogConfiguration, add a comment documenting that UPDATE/DELETE permissions should be revoked at the SQL Server level for the application user. Create a SQL migration script comment for this.

**EF Configurations:**

12. `AuditLogConfiguration`: Composite index on (Timestamp DESC, UserId, EntityName) for efficient querying. Index on (EntityId, EntityName) for per-record history. Set all properties as required.
13. `AccessLogConfiguration`: Index on (Timestamp DESC, UserId). Index on (Action) for filtering login attempts.

**AuditInterceptor (Shared.Infrastructure):**

14. Inherits `SaveChangesInterceptor`
15. In `SavingChangesAsync`: Iterate ChangeTracker entries where entity is IAuditable and state is Added/Modified/Deleted. For each, capture: entity type name, entity Id, action type, and for Modified entities: iterate OriginalValues vs CurrentValues to build field-level change list (PropertyName, OldValue as string, NewValue as string). Store as List of AuditEntry (internal class).
16. In `SavedChangesAsync`: After successful save, resolve AuditDbContext from the service provider, create AuditLog entities from captured entries, save to audit schema. This ensures audit records only exist for successfully committed changes.
17. Inject ICurrentUser to get userId and email for the audit record.
18. Register interceptor on ALL module DbContexts in Bootstrapper Program.cs: `AddInterceptors(auditInterceptor)`.

**AccessLoggingMiddleware (Shared.Infrastructure):**

19. ASP.NET Core middleware that runs on every request.
20. For authenticated requests: log UserId, UserEmail, resource path, response status code.
21. For login endpoints specifically: log LoginFailed or Login based on response status.
22. Write to AuditDbContext AccessLog table.
23. Use fire-and-forget pattern (don't block the response) -- queue via Wolverine local queue for async write.
24. Register in Bootstrapper: `app.UseMiddleware<AccessLoggingMiddleware>()`

**Wolverine.HTTP Endpoints:**

25. `AuditLogEndpoints`:
   - `[WolverineGet("/api/admin/audit-logs")]` [Authorize] -- query audit logs with AuditLogQuery parameters (user, action, date range), returns paginated list. Use cursor-based pagination (by Timestamp + Id) for performance.
   - `[WolverineGet("/api/admin/audit-logs/export")]` [Authorize] -- export audit logs as CSV with the same filter parameters. Return file download response.

26. `AccessLogEndpoints`:
   - `[WolverineGet("/api/admin/access-logs")]` [Authorize] -- query access logs, paginated

Use Context7 for EF Core SaveChanges interceptor pattern, Wolverine.HTTP file response, and ASP.NET Core middleware.

**Azure Blob Storage Service (ARC-05):**

1. `IAzureBlobService` (interface in Shared.Infrastructure -- or better: interface in Shared.Application for clean dependency):

   - Move interface to `Shared.Application/Services/IAzureBlobService.cs`
   - UploadAsync(string containerName, string blobName, Stream content, string contentType) -> Task<string> (returns blob URL)
   - DownloadAsync(string containerName, string blobName) -> Task<Stream>
   - DeleteAsync(string containerName, string blobName) -> Task<bool>
   - GetSasUrlAsync(string containerName, string blobName, TimeSpan expiry) -> Task<string>
   - ListBlobsAsync(string containerName, string prefix) -> Task<IEnumerable<BlobInfo>>

2. `AzureBlobService` (implementation in Shared.Infrastructure):
   - Uses Azure.Storage.Blobs BlobServiceClient
   - Reads connection string from IConfiguration ("AzureStorage:ConnectionString")
   - Container naming: one container per module (e.g., "clinical-images", "documents")
   - Blob naming: {entityId}/{timestamp}_{filename}
   - Enable soft delete + versioning via BlobContainerClient options
   - Register as scoped service in Bootstrapper

**ACL Adapter Pattern (ARC-01):**

3. `IExternalSystemAdapter` (in Shared.Domain/Ports/):
   - Marker interface for all external system adapters
   - Generic interface: `IExternalSystemAdapter<TRequest, TResponse>` with `Task<Result<TResponse>> ExecuteAsync(TRequest request, CancellationToken ct)`
   - Concrete adapter interfaces will be defined per external system in later phases (e.g., IZaloOaAdapter, IMisaAdapter)
   - This establishes the pattern: domain defines the port (interface with domain-shaped request/response), infrastructure implements the adapter that translates to/from external API models

**Template Engine Interface (ARC-03):**

4. `ITemplateDefinition` (in Shared.Domain):
   - Interface for disease-specific templates that can be added without code changes
   - Properties: string TemplateId, string TemplateName, string Version, IReadOnlyList<TemplateField> Fields
   - `TemplateField`: Name (string), DataType (string), IsRequired (bool), ValidationRules (string -- JSON), DisplayOrder (int)
   - The idea: Dry Eye template, Myopia Control template, etc. are all implementations of this interface, loaded from configuration/database
   - Actual template implementations come in Phase 4 (Dry Eye) and post-launch (Myopia Control)
   - Create `ITemplateRegistry` interface: RegisterTemplate(ITemplateDefinition), GetTemplate(string templateId), GetAllTemplates()

**Data Export Service (ARC-06):**

5. `IDataExportService` (in Shared.Contracts):
   - Interface for full data export capability ensuring data ownership
   - ExportModuleDataAsync(string moduleName, Stream output, ExportFormat format) -> Task
   - ExportFormat enum: Json, Csv
   - Each module will implement this for its own data in later phases
   - This ensures Ganka28 can always export all their data out of the system
