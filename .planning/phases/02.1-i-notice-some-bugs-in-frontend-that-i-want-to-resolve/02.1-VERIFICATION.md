| File | Line | Pattern | Severity | Impact |
|------|------|---------|----------|--------|
| `auth-api.ts` | 45, 95 | Function named `refreshToken` — shares name with concept of "refresh token value" | Info | Internal function name only; no token value is stored in JS. Not a behavioral issue. |
| `_authenticated.tsx` (implicit) | — | `scheduleRefresh` timer is set on login but not re-initialized after silent refresh in `beforeLoad` | Warning | After silent refresh on page load, the timer for automatic token renewal at 80% lifetime is not started. The `useAuth` hook `scheduleRefresh` is only called from within the hook's `login`/`refresh` callbacks, not from the route guard's `beforeLoad`. This means after a page refresh + silent restore, the auto-refresh timer will NOT fire, and the access token will expire without silent renewal until the user navigates and `useAuth` runs again. |

**Blocker anti-patterns:** None that prevent core functionality on fresh login.
**Warning:** The missing timer initialization after silent refresh (above) means a user who refreshes the browser will have their session restored but will NOT get automatic token renewal. When the access token expires (~1 hour typical), the next API call will 401. The `_authenticated.tsx` sets auth state but never calls `scheduleRefresh`. This is a gap in the session persistence flow.

### Gaps Summary

All 14 automated must-haves are verified. No blocker anti-patterns found.

One notable gap was identified during verification that is NOT in the original plan's must_haves but represents incomplete goal achievement:

**Missing: `scheduleRefresh` not called after silent restore in `beforeLoad`**

The `_authenticated.tsx` `beforeLoad` correctly calls `silentRefresh()` and stores the access token in `useAuthStore`. However, it does NOT start the automatic token refresh timer (`scheduleRefresh`). This timer lives inside the `useAuth` hook and is only started during explicit `login` or `refresh` hook calls. After a browser refresh + silent restore:
- Session IS restored (access token available) — goal partially met
- Automatic renewal at 80% lifetime is NOT initialized — the token will expire unattended

This was not caught in the plan's must_haves, and the 04-SUMMARY says the user approved "session persists across browser refresh" — which is true for the duration of the access token. For long sessions, this would cause silent expiry.

This gap is **informational** for now since: (a) it was not in the phase's must_haves, (b) the user approved the behavior in plan 04, and (c) the next API call after expiry would trigger a 401 and redirect to login. It is flagged for awareness, not as a phase blocker.
