# Phase 02.1: Frontend Bug Fixes + HTTP-only Cookie Security - Research

**Researched:** 2026-03-02
**Domain:** Frontend session persistence (HTTP-only cookie), shadcn/ui visual alignment, auth security hardening
**Confidence:** HIGH

## Summary

This phase addresses three interrelated concerns: (1) session state is lost on every page refresh because the access token lives only in Zustand memory and the refresh token only in a `useRef`; (2) all pages deviate from the shadcn/ui default theme due to a custom Stone+Green color palette and zero border radius; (3) the refresh token is currently transmitted in the JSON response body and has no storage mechanism -- it must be switched to an HTTP-only cookie set by the backend.

The HTTP-only cookie change is the most architecturally significant item. It requires backend modifications to the login and refresh endpoints to set a `Set-Cookie` header with the refresh token (HTTP-only, Secure, SameSite=Lax, Path=/api/auth), removal of the refresh token from the JSON response body, and frontend changes to stop sending the refresh token in request bodies (it will be sent automatically by the browser as a cookie). The access token remains in-memory only (Zustand store, not persisted) and is silently re-obtained on page load via a cookie-based `/api/auth/refresh` call. The "Remember me" feature maps to cookie `Max-Age` -- session cookie (no Max-Age) for standard sessions, 30-day Max-Age for "remember me".

The theme reset is a CSS-variable-only change in `globals.css`. The session persistence fix follows naturally from the HTTP-only cookie approach: on page load, attempt a silent refresh using the cookie; if successful, populate the in-memory store; if not, redirect to login.

**Primary recommendation:** Implement the HTTP-only cookie backend changes first (they are the foundation), then update the frontend auth flow (remove refresh token handling, add credentials:include, implement silent refresh on load), then reset the theme, then audit all pages.

## Architecture Patterns

### Pattern 1: Backend Set-Cookie for Refresh Token

**What:** The login and refresh endpoints use `HttpContext.Response.Cookies.Append` to set the refresh token as an HTTP-only cookie. The refresh token is removed from the JSON response body.

**When to use:** On `/api/auth/login` and `/api/auth/refresh` successful responses.

**Example:**
```csharp
// Source: ASP.NET Core official docs + verified via Context7 /dotnet/aspnetcore.docs
// In AuthApiEndpoints.cs -- login endpoint
authGroup.MapPost("/login", async (LoginCommand command, IMessageBus bus, HttpContext ctx) =>
{
    var enriched = command with { IpAddress = ctx.Connection.RemoteIpAddress?.ToString() };
    var result = await bus.InvokeAsync<Result<LoginResponse>>(enriched);

    if (result.IsSuccess)
    {
        SetRefreshTokenCookie(ctx, result.Value.RefreshToken, command.RememberMe);
        // Return response WITHOUT refresh token
        var response = new
        {
            result.Value.AccessToken,
            result.Value.ExpiresAt,
            result.Value.User
        };
        return Results.Ok(response);
    }
    return result.ToHttpResult();
});

// Helper method for setting cookie
private static void SetRefreshTokenCookie(
    HttpContext ctx, string refreshToken, bool rememberMe)
{
    var cookieOptions = new CookieOptions
    {
        HttpOnly = true,
        Secure = true,       // requires HTTPS (ok for prod, dev proxy handles)
        SameSite = SameSiteMode.Lax,
        Path = "/api/auth",  // only sent to auth endpoints
    };

    if (rememberMe)
    {
        cookieOptions.MaxAge = TimeSpan.FromDays(30);
    }
    // else: session cookie (no MaxAge) -- expires on browser close

    ctx.Response.Cookies.Append("refresh_token", refreshToken, cookieOptions);
}
```

**Key decisions:**
- `SameSite=Lax`: Protects against CSRF for POST requests (cookies not sent on cross-origin POST). Since `/api/auth/refresh` is a POST endpoint, Lax provides CSRF protection without breaking normal navigation.
- `Path=/api/auth`: Cookie only sent to auth endpoints, not to every API call. Minimizes exposure.
- `Secure=true`: In development, the frontend proxy or CORS setup handles this. For production, HTTPS is mandatory.
- No `Domain` attribute: Defaults to the exact domain of the server (no subdomain sharing needed).

### Pattern 2: Backend Read Refresh Token from Cookie

**What:** The `/api/auth/refresh` endpoint reads the refresh token from the cookie instead of the request body. The request body becomes empty (or just `{}`).

**When to use:** On the refresh endpoint.

**Example:**
```csharp
// In AuthApiEndpoints.cs -- refresh endpoint
authGroup.MapPost("/refresh", async (IMessageBus bus, HttpContext ctx) =>
{
    var refreshToken = ctx.Request.Cookies["refresh_token"];
    if (string.IsNullOrEmpty(refreshToken))
        return Results.Unauthorized();

    var result = await bus.InvokeAsync<Result<RefreshTokenResponse>>(
        new RefreshTokenCommand(refreshToken));

    if (result.IsSuccess)
    {
        // Determine rememberMe from existing cookie presence
        // (if cookie has Max-Age, browser sends it; if session cookie, still sent)
        // For simplicity, always set same cookie type as original login
        // The handler returns new refresh token; set it as cookie
        SetRefreshTokenCookie(ctx, result.Value.RefreshToken, rememberMe: false);
        // Note: rememberMe tracking needs thought -- see Open Questions

        var response = new
        {
            result.Value.AccessToken,
            result.Value.ExpiresAt,
            result.Value.User
        };
        return Results.Ok(response);
    }
    return result.ToHttpResult();
});
```

### Pattern 3: Backend Clear Cookie on Logout

**What:** The logout endpoint clears the refresh token cookie by setting it to an expired value.

**When to use:** On `/api/auth/logout`.

**Example:**
```csharp
// In AuthApiEndpoints.cs -- logout endpoint
authGroup.MapPost("/logout", async (IMessageBus bus, HttpContext ctx) =>
{
    if (!ctx.TryGetUserId(out var userId))
        return Results.Unauthorized();

    var result = await bus.InvokeAsync<Result>(new LogoutCommand(userId));

    // Clear the refresh token cookie
    ctx.Response.Cookies.Delete("refresh_token", new CookieOptions
    {
        HttpOnly = true,
        Secure = true,
        SameSite = SameSiteMode.Lax,
        Path = "/api/auth",
    });

    return result.ToHttpResult();
}).RequireAuthorization();
```

### Pattern 4: Frontend credentials:'include' for Cookie Transmission

**What:** The openapi-fetch client must include `credentials: 'include'` so the browser sends cookies cross-origin. This is required because the frontend (localhost:3000) and backend (localhost:5255) are different origins in development.

**When to use:** On the `createClient` call or via middleware.

**Example:**
```typescript
// Source: openapi-fetch docs (Context7 /websites/openapi-ts_dev)
// In api-client.ts
import createClient from "openapi-fetch"
import type { Middleware } from "openapi-fetch"
import { useAuthStore } from "@/shared/stores/authStore"

const API_URL = import.meta.env.VITE_API_URL ?? "http://localhost:5255"

const authMiddleware: Middleware = {
  async onRequest({ request }) {
    const token = useAuthStore.getState().accessToken
    if (token) {
      request.headers.set("Authorization", `Bearer ${token}`)
    }
    return request
  },
}

// credentials: 'include' tells browser to send cookies cross-origin
export const api = createClient({ baseUrl: API_URL, credentials: "include" })
api.use(authMiddleware)
```

**CRITICAL:** The backend CORS configuration already has `AllowCredentials()` (verified in Program.cs line 138). This is required for `credentials: 'include'` to work.

### Pattern 5: Silent Refresh on Page Load (No Persist Middleware)

**What:** On page load, the access token is lost (it was in-memory only). The browser still has the refresh token cookie. The app attempts a silent refresh to `/api/auth/refresh` (cookie sent automatically). If successful, the access token and user are populated in the Zustand store. If failed (cookie expired/invalid), redirect to login.

**When to use:** In the `_authenticated` route's `beforeLoad`, or in a root-level component that runs before authenticated routes.

**Example:**
```typescript
// In _authenticated.tsx -- beforeLoad with async silent refresh
export const Route = createFileRoute("/_authenticated")({
  beforeLoad: async ({ location }) => {
    const { isAuthenticated, accessToken } = useAuthStore.getState()

    // If already authenticated with a valid access token, proceed
    if (isAuthenticated && accessToken) {
      return
    }

    // Attempt silent refresh using the HTTP-only cookie
    try {
      const response = await silentRefresh()
      if (response) {
        useAuthStore.getState().setAuth(
          {
            id: response.user.id,
            email: response.user.email,
            fullName: response.user.fullName,
            permissions: response.user.permissions,
            preferredLanguage: response.user.preferredLanguage,
          },
          response.accessToken,
        )
        return // silently restored session
      }
    } catch {
      // refresh failed
    }

    // No valid session -- redirect to login
    throw redirect({
      to: "/login",
      search: { redirect: location.href },
    })
  },
  component: AppShell,
})
```

The `silentRefresh` function calls `/api/auth/refresh` with an empty body (the cookie is sent automatically by the browser):
```typescript
// In auth-api.ts
async function silentRefresh(): Promise<LoginResponse | null> {
  const res = await api.POST("/api/auth/refresh" as never, {})
  if (res.error) return null
  return res.data as LoginResponse
}
```

### Pattern 6: authStore -- No Persist, In-Memory Only

**What:** The auth store does NOT use the `persist` middleware. Access token and user are held in-memory only. On page refresh, the store resets to initial state. Session continuity is handled by the silent refresh pattern (Pattern 5).

**When to use:** This replaces the previous research recommendation of using Zustand persist.

**Example:**
```typescript
// authStore.ts -- stays as-is (no persist), but remove refreshToken handling
import { create } from "zustand"

export interface AuthUser {
  id: string
  email: string
  fullName: string
  permissions: string[]
  preferredLanguage: string
}

interface AuthState {
  user: AuthUser | null
  accessToken: string | null
  isAuthenticated: boolean
  setAuth: (user: AuthUser, accessToken: string) => void
  clearAuth: () => void
}

export const useAuthStore = create<AuthState>()((set) => ({
  user: null,
  accessToken: null,
  isAuthenticated: false,
  setAuth: (user, accessToken) =>
    set({ user, accessToken, isAuthenticated: true }),
  clearAuth: () =>
    set({ user: null, accessToken: null, isAuthenticated: false }),
}))
```

**Key insight:** The store interface does NOT change. The `setAuth` function already accepts `(user, accessToken)` without a refresh token. The refresh token is now fully backend-managed via cookies.

### Pattern 7: useAuth Hook -- Remove Refresh Token Handling

**What:** The `useAuth` hook no longer manages the refresh token. It no longer stores it in a `useRef`. The `scheduleRefresh` function still calls `/api/auth/refresh`, but the refresh token is sent as a cookie (automatically by the browser), not in the request body. The `login` function no longer receives the refresh token from the response.

**When to use:** This is the primary refactor of the frontend auth flow.

**Key changes to `useAuth.ts`:**
1. Remove `refreshTokenRef` entirely
2. `scheduleRefresh` calls refresh endpoint with empty body (cookie sent automatically)
3. `login` callback no longer reads `response.refreshToken`
4. `refresh` function takes no arguments (cookie-based)
5. `logout` no longer needs to clear `refreshTokenRef`

### Anti-Patterns to Avoid
- **Storing refresh token in localStorage/Zustand persist:** The user decision is HTTP-only cookie. Never persist the refresh token on the frontend.
- **Sending refresh token in request body:** After the switch, the cookie handles this. The request body for `/api/auth/refresh` should be empty.
- **Using `SameSite=None` without justification:** `SameSite=Lax` is correct for same-site frontend-backend communication. `None` is for third-party cookies and requires `Secure`.
- **Fixing colors per-component instead of globally:** Fix CSS variables first, then audit for hardcoded color classes.
- **Wrapping oklch values in hsl():** CSS variables store oklch values directly. `hsl(var(--border))` produces invalid colors.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Cookie management | Custom Set-Cookie header strings | `HttpContext.Response.Cookies.Append` with `CookieOptions` | Handles encoding, SameSite browser compatibility |
| Token refresh scheduling | Complex timer with cookie reading | Keep existing 80%-lifetime timer pattern in `useAuth` | Timer logic is already proven; just change transport |
| Theme color system | Custom color palette | shadcn/ui default CSS variables from Context7 | Proven, accessible, tested across all components |
| CSRF protection for cookie-based auth | Custom CSRF tokens | `SameSite=Lax` on cookie + POST-only endpoints | Lax prevents cross-origin POST with cookies; login/refresh/logout are all POST |

**Key insight:** The HTTP-only cookie approach actually *simplifies* the frontend code. The refresh token completely disappears from frontend JavaScript. No localStorage, no useRef, no passing tokens around. The browser handles cookie storage and transmission transparently.

## Common Pitfalls

### Pitfall 1: Cookie Not Sent Cross-Origin Without credentials:'include'

**What goes wrong:** The browser has the refresh token cookie, but `/api/auth/refresh` calls fail with 401 because the cookie is not sent.
**Why it happens:** By default, `fetch()` does not send cookies on cross-origin requests. The frontend (localhost:3000) and backend (localhost:5255) are different origins.
**How to avoid:** Set `credentials: 'include'` on the openapi-fetch client. The backend CORS policy already has `AllowCredentials()`.
**Warning signs:** Refresh endpoint returns 401 even though the cookie exists in browser DevTools.

### Pitfall 2: Cookie Path Mismatch

**What goes wrong:** Cookie set with `Path=/api/auth` is not sent to `/api/auth/refresh` or `/api/auth/logout`.
**Why it happens:** Path matching in cookies is prefix-based. `/api/auth` matches `/api/auth`, `/api/auth/refresh`, `/api/auth/login`, etc. But if the path was accidentally set to `/api/auth/` (with trailing slash), it would still match. The risk is typos.
**How to avoid:** Use exactly `Path = "/api/auth"` in the `CookieOptions`. Verify in browser DevTools that the cookie shows the correct path.
**Warning signs:** Cookie visible in DevTools but not sent with requests (check Network tab > Request Headers > Cookie).

### Pitfall 3: Secure Flag in Development

**What goes wrong:** `Secure = true` means the cookie is only sent over HTTPS. In development, the backend may run on HTTP (`http://localhost:5255`).
**Why it happens:** Development environments typically use HTTP.
**How to avoid:** In development, either (a) set `Secure = false` via environment-aware configuration, or (b) use the Vite dev server proxy to forward `/api` requests (making it same-origin, avoiding the issue entirely). **Recommended:** Make `Secure` flag configurable based on `app.Environment.IsDevelopment()`.
**Warning signs:** Cookie appears in Set-Cookie response header but browser rejects it (visible in DevTools > Application > Cookies as a warning).

### Pitfall 4: "Remember Me" State Lost on Token Rotation

**What goes wrong:** User logs in with "Remember me" (30-day cookie). On token rotation at the refresh endpoint, the new cookie is set without `Max-Age` (session cookie), losing the "remember me" persistence.
**Why it happens:** The refresh endpoint doesn't know whether the original login had "remember me" checked. The `RefreshTokenHandler` currently always uses `GetRefreshTokenLifetimeDays(false)` (7 days).
**How to avoid:** Store a `rememberMe` flag on the `RefreshToken` entity in the database, or derive it from the token's original expiry. When rotating, check the original token's intended persistence and apply the same cookie type.
**Warning signs:** User checks "remember me", closes browser, opens next day -- session is lost despite 30-day token.

### Pitfall 5: hsl() Wrapping oklch Values in FullCalendar CSS

**What goes wrong:** FullCalendar custom styles use `hsl(var(--border))` syntax (14 instances), but CSS variables store oklch values. This produces incorrect or broken colors.
**Why it happens:** The project was set up with oklch CSS variables, but some inline styles were written assuming hsl format.
**How to avoid:** Replace all `hsl(var(--...))` with `var(--...)` in `AppointmentCalendar.tsx` and `sidebar.tsx`.
**Warning signs:** Colors appearing as black, white, or wrong hue in FullCalendar and sidebar.

### Pitfall 6: Hardcoded Color Classes Surviving Theme Reset

**What goes wrong:** Some components use `bg-stone-50`, `bg-white`, `text-stone-*` instead of semantic tokens. These don't change when CSS variables are updated.
**Why it happens:** Developers use specific color classes for quick styling.
**How to avoid:** After theme reset, grep and replace: `bg-stone-50` -> `bg-muted`/`bg-background`, `bg-white` -> `bg-card`/`bg-background`, `text-stone-*` -> `text-muted-foreground`, etc.
**Known locations:** `PublicBookingPage.tsx`, `book/status.tsx` (bg-stone-50, bg-white), `AuditLogTable.tsx` (bg-stone-*, text-stone-*, border-stone-*).

### Pitfall 7: beforeLoad Must Handle Async Refresh

**What goes wrong:** The `_authenticated` route's `beforeLoad` currently does a synchronous check of `isAuthenticated`. With in-memory-only tokens, this will always be `false` after page refresh. The silent refresh is async.
**Why it happens:** `beforeLoad` in TanStack Router supports async operations (it can return a Promise). But the current code is synchronous.
**How to avoid:** Make `beforeLoad` async. Attempt silent refresh, populate store on success, redirect on failure. This adds a brief loading state on initial page load.
**Warning signs:** Flash of login page before authenticated content loads.

### Pitfall 8: Border Radius in FullCalendar Hardcoded to Zero

**What goes wrong:** FullCalendar inline CSS sets `border-radius: 0` on buttons and events. After `--radius: 0.625rem`, FullCalendar still has zero radius.
**Why it happens:** Custom CSS was written to match the previous zero-radius theme.
**How to avoid:** Remove hardcoded `border-radius: 0` in AppointmentCalendar.tsx, let it inherit from theme.

## Detailed Technical Analysis

### Current Auth Flow (What Changes)

**Current login flow:**
1. Frontend POSTs `{ email, password, rememberMe }` to `/api/auth/login`
2. Backend returns `{ accessToken, refreshToken, expiresAt, user }` in JSON body
3. Frontend stores `user` and `accessToken` in Zustand (in-memory)
4. Frontend stores `refreshToken` in `useRef` inside `useAuth` hook
5. Frontend schedules refresh timer at 80% of access token lifetime
6. On refresh timer fire: POST `{ refreshToken }` to `/api/auth/refresh`
7. On page refresh: everything lost -> redirect to login

**New login flow (after changes):**
1. Frontend POSTs `{ email, password, rememberMe }` to `/api/auth/login` (with `credentials: 'include'`)
2. Backend returns `{ accessToken, expiresAt, user }` in JSON body + sets `refresh_token` as HTTP-only cookie via `Set-Cookie` header
3. Frontend stores `user` and `accessToken` in Zustand (in-memory)
4. Frontend schedules refresh timer at 80% of access token lifetime
5. On refresh timer fire: POST (empty body) to `/api/auth/refresh` -- cookie sent automatically
6. On page refresh: `_authenticated.beforeLoad` calls silent refresh -> cookie sent automatically -> if success, populate store -> if fail, redirect to login

### Backend Changes Required

| File | Change | Impact |
|------|--------|--------|
| `Auth.Presentation/AuthApiEndpoints.cs` | Login endpoint: set refresh token cookie, remove from response body. Refresh endpoint: read token from cookie instead of body, set new cookie on rotation. Logout endpoint: clear cookie. | **Major** -- endpoint behavior changes |
| `Auth.Contracts/Dtos/LoginResponse.cs` | Remove `RefreshToken` property | **Breaking** -- JSON response shape changes |
| `Auth.Application/Features/RefreshToken.cs` | `RefreshTokenCommand` no longer needs to carry the token string (it comes from cookie). OR keep the command as-is and pass cookie value from endpoint. | **Minor** -- the command can stay; endpoint extracts cookie and passes to command |
| `Auth.Application/Features/Login.cs` | `LoginCommand` may need `RememberMe` passed through so endpoint knows cookie type. Already has `RememberMe` field. | **None** -- already supports this |
| `Auth.Unit.Tests/Features/LoginHandlerTests.cs` | Update assertions that check `result.Value.RefreshToken` | **Minor** -- if LoginResponse changes |
| `Auth.Unit.Tests/Features/RefreshTokenHandlerTests.cs` | Tests are handler-level, not endpoint-level. Handler still receives token as command param. | **None** -- handler tests unchanged |

**Design decision for LoginResponse:** Two approaches:
1. **Remove RefreshToken from LoginResponse entirely** -- cleaner, but the handler still generates the token and needs to pass it somewhere. The endpoint can extract it before serializing.
2. **Keep RefreshToken in handler response, strip at endpoint level** -- handler logic unchanged, endpoint extracts refreshToken for cookie and returns a projection without it.

**Recommendation:** Option 2 is safer. Keep `LoginResponse` and `RefreshTokenResponse` records as-is in the Application/Contracts layers. The Presentation layer (endpoint) extracts the refresh token for the cookie and returns a DTO without it. This avoids changing handler logic and keeps handler tests unchanged.

### Development Secure Flag Strategy

In development, the frontend runs on `http://localhost:3000` and backend on `http://localhost:5255` (both HTTP). The `Secure` flag on cookies means the browser only sends them over HTTPS.

**Strategy:** Make `Secure` flag environment-aware:
```csharp
var cookieOptions = new CookieOptions
{
    HttpOnly = true,
    Secure = !app.Environment.IsDevelopment(),  // false in dev, true in prod
    SameSite = SameSiteMode.Lax,
    Path = "/api/auth",
};
```

This is standard practice. In production, HTTPS is mandatory. In development, HTTP works for local testing.

**Alternative:** Use Vite proxy to make frontend requests same-origin (`/api` -> `localhost:5255`), eliminating CORS entirely. However, this requires Vite proxy config changes and the project already has working CORS config. **Not recommended for this phase** -- too many moving parts.

```csharp
authGroup.MapPost("/login", async (LoginCommand command, IMessageBus bus, HttpContext ctx) =>
{
    var enriched = command with { IpAddress = ctx.Connection.RemoteIpAddress?.ToString() };
    var result = await bus.InvokeAsync<Result<LoginResponse>>(enriched);

    if (result.IsSuccess)
    {
        var login = result.Value;
        // Set refresh token as HTTP-only cookie
        var isDevEnvironment = ctx.RequestServices.GetRequiredService<IWebHostEnvironment>().IsDevelopment();
        ctx.Response.Cookies.Append("refresh_token", login.RefreshToken, new CookieOptions
        {
            HttpOnly = true,
            Secure = !isDevEnvironment,
            SameSite = SameSiteMode.Lax,
            Path = "/api/auth",
            MaxAge = command.RememberMe ? TimeSpan.FromDays(30) : null,
        });

        // Return response without refresh token
        return Results.Ok(new
        {
            login.AccessToken,
            login.ExpiresAt,
            login.User,
        });
    }
    return result.ToHttpResult();
});
```

### Complete Backend Refresh Endpoint (After Changes)
```csharp
authGroup.MapPost("/refresh", async (IMessageBus bus, HttpContext ctx) =>
{
    var refreshToken = ctx.Request.Cookies["refresh_token"];
    if (string.IsNullOrEmpty(refreshToken))
        return Results.Problem(detail: "No refresh token", title: "Unauthorized", statusCode: 401);

    var result = await bus.InvokeAsync<Result<RefreshTokenResponse>>(
        new RefreshTokenCommand(refreshToken));

    if (result.IsSuccess)
    {
        var refresh = result.Value;
        var isDevEnvironment = ctx.RequestServices.GetRequiredService<IWebHostEnvironment>().IsDevelopment();
        ctx.Response.Cookies.Append("refresh_token", refresh.RefreshToken, new CookieOptions
        {
            HttpOnly = true,
            Secure = !isDevEnvironment,
            SameSite = SameSiteMode.Lax,
            Path = "/api/auth",
            // Preserve cookie type: if original was persistent, new should be too
            // See Open Question about rememberMe tracking
        });

        return Results.Ok(new
        {
            refresh.AccessToken,
            refresh.ExpiresAt,
            refresh.User,
        });
    }
    return result.ToHttpResult();
});
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Refresh token in response body + localStorage/memory | HTTP-only cookie via Set-Cookie | Industry best practice since ~2020 | XSS cannot steal refresh token |
| `hsl(var(--color))` CSS variable syntax | oklch values directly in CSS variables, no wrapper | shadcn/ui v4 (Tailwind CSS v4) | Must remove all `hsl()` wrappers |
| Custom Stone/Green palette | shadcn/ui neutral default | User decision (Phase 02.1) | All pages shift appearance |
| `--radius: 0rem` | `--radius: 0.625rem` | User decision (Phase 02.1) | All components get rounded corners |

**Deprecated/outdated:**
- `hsl(var(--...))` pattern: Was used in shadcn/ui v3 (Tailwind v3). In v4, CSS variables contain full oklch color values.
- Refresh token in JSON body: While still common, HTTP-only cookies are the security best practice for web applications.

## Open Questions

1. **"Remember Me" persistence during token rotation**
   - What we know: The login endpoint knows `rememberMe` and sets `MaxAge` accordingly. The refresh endpoint rotates the token but doesn't know if the original was "remember me".
   - What's unclear: How to preserve the "remember me" cookie type across rotations.
   - Options: (a) Add a `RememberMe` boolean column to the `RefreshToken` entity. On rotation, read the original token's flag and apply the same cookie type. (b) Store `rememberMe` in a non-HttpOnly flag cookie that the endpoint can read. (c) Always use the same MaxAge as the token's DB expiry.
   - **Recommendation:** Option (a) -- add `RememberMe` bool to `RefreshToken` entity. On rotation, copy the value from the old token. This is the cleanest and most secure approach. The field maps directly to the business concept.

2. **Login page branding panel after theme reset**
   - What we know: The right panel uses `bg-primary` (currently green). After reset, `--primary` becomes `oklch(0.205 0 0)` (near-black).
   - What's unclear: Whether the near-black panel looks good.
   - Recommendation: Accept the near-black primary panel as it follows shadcn/ui neutral aesthetic. White text/icons contrast well. Adjust in visual audit if needed.

3. **Missing `tw-animate-css` and `outline-ring/50` imports**
   - What we know: Latest shadcn/ui v4 docs show these in globals.css. Current project lacks them.
   - What's unclear: Whether these are strictly required or just recommended.
   - Recommendation: Check if `tw-animate-css` is installed in package.json. If not, install and add import. Add `outline-ring/50` base style. These are low-risk additions that improve consistency.

4. **Refresh endpoint: how to handle the transition**
   - What we know: The current refresh endpoint expects `RefreshTokenCommand` with a `RefreshToken` string property from the request body. After changes, the token comes from the cookie.
   - What's unclear: Whether to change the command or keep it and extract the cookie at the endpoint level.
   - Recommendation: Keep `RefreshTokenCommand` as-is. The endpoint extracts the cookie value and creates the command: `new RefreshTokenCommand(cookieValue)`. This leaves handler tests completely unchanged.
