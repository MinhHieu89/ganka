# Phase 02.1: Frontend Bug Fixes - Research

**Researched:** 2026-03-02
**Domain:** Frontend session persistence + shadcn/ui visual alignment
**Confidence:** HIGH

## Summary

This phase addresses two distinct frontend bugs: (1) session state is lost on every page refresh because the Zustand auth store lacks the `persist` middleware, and (2) all pages use a custom Stone+Green theme with zero border radius that deviates from the shadcn/ui default look and feel.

The session persistence fix is straightforward -- the `authStore` is the only Zustand store in the project that does NOT use the `persist` middleware, while `recentPatientsStore` and `sidebarStore` already demonstrate the correct pattern. The fix requires adding `persist` with `partialize` to selectively store `user`, `accessToken`, and `refreshToken` (currently only held in a `useRef` inside `useAuth`), plus adding a rehydration-aware bootstrap flow on app startup that attempts a token refresh before the `_authenticated` route guard kicks in.

The theme reset requires replacing the custom Stone+Green oklch color values with the shadcn/ui neutral default theme values (provided by Context7 from official docs), changing `--radius` from `0rem` to `0.625rem`, and auditing all pages for hardcoded color classes (`bg-stone-50`, `bg-white`) and broken `hsl(var(--...))` references (the CSS variables use oklch format, but FullCalendar styles and the sidebar component wrap them in `hsl()`, producing incorrect colors).

**Primary recommendation:** Fix session persistence first (it blocks usability), then reset the theme globally via CSS variables, then audit each page for hardcoded deviations.

## Architecture Patterns

### Pattern 1: Auth Store Persistence with Partialize
**What:** Add `persist` middleware to `authStore` with `partialize` to store only serializable auth data (user, accessToken, refreshToken), excluding functions.
**When to use:** Whenever Zustand state needs to survive page refresh.
**Example:**
```typescript
// Source: Context7 /pmndrs/zustand - persist middleware docs
import { create } from 'zustand'
import { persist, createJSONStorage } from 'zustand/middleware'

interface AuthState {
  user: AuthUser | null
  accessToken: string | null
  refreshToken: string | null  // NEW: must be stored for rehydration
  isAuthenticated: boolean
  setAuth: (user: AuthUser, accessToken: string, refreshToken: string) => void
  clearAuth: () => void
}

export const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({
      user: null,
      accessToken: null,
      refreshToken: null,
      isAuthenticated: false,
      setAuth: (user, accessToken, refreshToken) =>
        set({ user, accessToken, refreshToken, isAuthenticated: true }),
      clearAuth: () =>
        set({ user: null, accessToken: null, refreshToken: null, isAuthenticated: false }),
    }),
    {
      name: 'ganka28-auth',
      partialize: (state) => ({
        user: state.user,
        accessToken: state.accessToken,
        refreshToken: state.refreshToken,
        isAuthenticated: state.isAuthenticated,
      }),
    },
  ),
)
```

### Pattern 2: Bootstrap Rehydration on App Startup
**What:** On app load, check if persisted auth state exists. If so, attempt a silent token refresh to validate the session before rendering authenticated routes. This avoids showing the login page briefly when the user has a valid persisted session.
**When to use:** In the `_authenticated` route's `beforeLoad` or in a root-level initialization effect.
**Example:**
```typescript
// In _authenticated.tsx beforeLoad:
// 1. Check useAuthStore.getState().isAuthenticated
// 2. If authenticated but token might be expired, attempt refresh
// 3. If refresh fails, clearAuth() and redirect to login
```

### Pattern 3: Refresh Token in Store (not useRef)
**What:** The current `useAuth` hook stores `refreshToken` only in a `useRef`, which is lost on unmount/refresh. The refresh token MUST be stored in the persisted Zustand store.
**When to use:** Always -- the refresh token is the key to session continuity.
**Critical change:** `setAuth` must accept and store `refreshToken`. The `login` callback and `refresh` callback must both pass `refreshToken` to `setAuth`.

### Pattern 4: Global Theme Reset via CSS Variables
**What:** Replace all CSS variable values in `globals.css` `:root` and `.dark` blocks with shadcn/ui neutral default values. This is a single-file change that propagates to all components.
**When to use:** When the entire application needs a theme change without touching individual components.
**Why global:** shadcn/ui components consume CSS variables via Tailwind classes (`bg-primary`, `text-muted-foreground`, etc.). Changing the variable values changes all component appearances at once.

### Anti-Patterns to Avoid
- **Storing access tokens in cookies from frontend:** The backend issues JWTs as response body data, not as Set-Cookie headers. Changing this requires backend API changes, which is out of scope.
- **Using sessionStorage for "remember me":** sessionStorage is cleared when the browser tab/window closes, defeating the purpose of "remember me".
- **Fixing colors per-component instead of globally:** The theme is defined via CSS variables. Fixing individual components creates maintenance burden and inconsistency. Fix at the variable level first.
- **Wrapping oklch values in hsl():** The CSS variables store oklch color values. Using `hsl(var(--border))` will produce invalid colors. Use the oklch values directly or use Tailwind utility classes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| State persistence | Custom localStorage read/write | `zustand/middleware` persist | Handles serialization, rehydration, partialize, storage abstraction |
| Token refresh scheduling | Custom setInterval/setTimeout logic | Keep existing `useAuth` pattern, just persist the refreshToken | Existing timer-based refresh at 80% lifetime is correct |
| Theme color system | Custom color palette | shadcn/ui default CSS variables | Proven, accessible, tested across all components |

**Key insight:** The session persistence bug is NOT a missing feature -- it's a configuration gap. The project already uses `persist` in two other stores. The auth store simply needs the same middleware applied.

## Common Pitfalls

### Pitfall 1: Hydration Race Condition
**What goes wrong:** On page refresh, Zustand rehydrates from localStorage asynchronously. The `_authenticated` route guard checks `isAuthenticated` synchronously in `beforeLoad`. If the guard runs before rehydration completes, the user gets redirected to login even though they have valid persisted credentials.
**Why it happens:** Zustand's `persist` middleware rehydrates asynchronously by default.
**How to avoid:** Either (a) use Zustand's `onRehydrateStorage` callback to signal completion and delay route evaluation, or (b) check localStorage directly in the `beforeLoad` synchronous check as a fallback.
**Warning signs:** Intermittent redirect to login on refresh, especially on slow devices.

### Pitfall 2: Stale Access Token After Refresh
**What goes wrong:** The persisted access token may be expired when the page reloads. If the app tries to use it directly, API calls will fail with 401.
**Why it happens:** Access tokens have short lifetimes (15 minutes per backend config).
**How to avoid:** On app startup with persisted state, always attempt a refresh token exchange before making authenticated API calls. The refresh token has a longer lifetime (7 days default, 30 days with "remember me").
**Warning signs:** 401 errors immediately after page refresh despite being "authenticated" in store.

### Pitfall 3: hsl() Wrapping oklch Values
**What goes wrong:** The FullCalendar custom styles and sidebar component use `hsl(var(--border))` syntax, but the CSS variables store oklch values. This produces incorrect or broken colors.
**Why it happens:** The project was set up with oklch CSS variables, but some inline styles were written assuming hsl format (likely copied from older shadcn/ui examples or FullCalendar integration guides).
**How to avoid:** After theme reset, audit ALL `hsl(var(--...))` references and replace with either Tailwind utility classes or direct `var(--...)` references (oklch values don't need a function wrapper).
**Warning signs:** Colors appearing as black, white, or wrong hue in FullCalendar and sidebar outline variant.

### Pitfall 4: Hardcoded Color Classes
**What goes wrong:** Some components use hardcoded Tailwind color classes (`bg-stone-50`, `bg-white`) instead of semantic tokens (`bg-background`, `bg-card`). These won't change when the theme variables are updated.
**Why it happens:** Developers use specific color classes for quick styling without considering theme consistency.
**How to avoid:** Search for and replace all `bg-stone-*`, `bg-white`, `text-stone-*` with semantic equivalents. Public booking page and audit log table are known offenders.
**Warning signs:** Some sections look different from the rest after theme reset.

### Pitfall 5: Remember Me Checkbox Already Exists But Does Nothing
**What goes wrong:** The login form already has a "Remember me" checkbox and passes `rememberMe` to the backend. The backend already handles different refresh token lifetimes. But since the refresh token is never persisted on the frontend, the "remember me" feature has no effect.
**Why it happens:** The auth store doesn't persist, so all tokens are lost on refresh regardless of "remember me" selection.
**How to avoid:** Once `authStore` persistence is added, "remember me" will automatically work because the backend returns a 30-day refresh token (vs 7-day default). The frontend just needs to store and reuse it.

### Pitfall 6: Border Radius in FullCalendar CSS
**What goes wrong:** The FullCalendar custom styles explicitly set `border-radius: 0` on buttons and events. After the global theme reset to `--radius: 0.625rem`, the FullCalendar will still have zero radius due to these hardcoded overrides.
**Why it happens:** The custom CSS was written to match the previous zero-radius theme.
**How to avoid:** Update or remove the hardcoded `border-radius: 0` in AppointmentCalendar.tsx inline styles, let it inherit from the theme.

## Detailed Bug Analysis

### Bug 1: Session Not Persisting Across Refresh

**Root cause chain:**
1. `authStore.ts` uses plain `create()` without `persist` middleware
2. On page refresh, all Zustand state resets to initial values (`user: null`, `isAuthenticated: false`)
3. `_authenticated.tsx` `beforeLoad` checks `useAuthStore.getState().isAuthenticated` -- gets `false`
4. User is redirected to `/login`
5. Meanwhile, `refreshToken` is stored only in `useRef` inside `useAuth` hook -- also lost on refresh

**Files that need changes:**
| File | Change |
|------|--------|
| `frontend/src/shared/stores/authStore.ts` | Add `persist` middleware, add `refreshToken` field |
| `frontend/src/features/auth/hooks/useAuth.ts` | Remove `refreshTokenRef`, read/write `refreshToken` from store |
| `frontend/src/app/routes/_authenticated.tsx` | Add rehydration-aware bootstrap (attempt silent refresh on load) |
| `frontend/src/features/auth/components/LoginForm.tsx` | Already has "Remember me" -- no changes needed |

**Backend already supports this:**
- `LoginResponse` returns `refreshToken` string
- `RefreshTokenHandler` accepts `RefreshTokenRequest { refreshToken }`
- Backend gives 7-day refresh tokens by default, 30-day with `rememberMe: true`
- Access tokens are 15 minutes

### Bug 2: Pages Don't Match shadcn/ui Look and Feel

**Root cause:**
1. `globals.css` uses custom Stone base + Green accent color palette (decision from Phase 01-02)
2. `--radius: 0rem` creates zero border radius on all components
3. Some pages use hardcoded color classes (`bg-stone-50`, `bg-white`) instead of semantic tokens
4. FullCalendar inline CSS uses `hsl(var(--...))` to wrap oklch values -- produces wrong colors
5. Sidebar component's outline variant also uses `hsl(var(--...))` pattern

**Files that need changes:**
| File | Change |
|------|--------|
| `frontend/src/styles/globals.css` | Replace all CSS variable values with shadcn/ui neutral defaults; set `--radius: 0.625rem` |
| `frontend/src/features/scheduling/components/AppointmentCalendar.tsx` | Fix `hsl(var(--...))` to `var(--...)`; remove `border-radius: 0` hardcodes |
| `frontend/src/shared/components/ui/sidebar.tsx` | Fix `hsl(var(--sidebar-border))` and `hsl(var(--sidebar-accent))` in outline variant |
| `frontend/src/features/booking/components/PublicBookingPage.tsx` | Replace `bg-stone-50` with `bg-muted` or `bg-background` |
| `frontend/src/app/routes/book/status.tsx` | Replace `bg-stone-50` with `bg-muted` or `bg-background` |
| `frontend/src/features/audit/components/AuditLogTable.tsx` | Replace `bg-stone-*`, `text-stone-*`, `border-stone-*` with semantic tokens |
| `frontend/src/features/auth/components/LoginPage.tsx` | Audit branding panel (uses `bg-primary` which will change color) |
| All pages | Visual audit after global theme change |

**Scope of visual change:**
- The current theme is Stone (warm gray with yellow undertone) + Green (primary/accent)
- The new theme is Neutral (pure gray, no color undertone) + dark primary (near-black)
- This is a significant visual change -- the green accent disappears entirely
- The branding panel on the login page uses `bg-primary` which will shift from green to near-black
- Custom branding elements (logo "G" box, feature icons) may need color adjustments
