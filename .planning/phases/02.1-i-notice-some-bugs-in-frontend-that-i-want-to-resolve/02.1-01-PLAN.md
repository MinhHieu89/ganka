<objective>
Implement HTTP-only cookie storage for refresh tokens on the backend auth endpoints using TDD.

Purpose: Replace the current insecure pattern of returning refresh tokens in JSON response bodies with HTTP-only cookies that cannot be accessed by JavaScript (XSS-proof). This is the foundation for frontend session persistence -- the browser automatically sends the cookie on page load, enabling silent token refresh.

Output: Modified login/refresh/logout endpoints with cookie handling, RememberMe column on RefreshToken entity, and comprehensive endpoint-level tests.
</objective>

  <behavior>
    Test cases (RED phase -- write these first):

    1. Login_Success_SetsRefreshTokenCookie:
       POST /api/auth/login with valid credentials
       -> Response has Set-Cookie header with "refresh_token"
       -> Cookie has HttpOnly flag
       -> Cookie has SameSite=Lax
       -> Cookie has Path=/api/auth
       -> JSON body contains accessToken, expiresAt, user
       -> JSON body does NOT contain refreshToken

    2. Login_WithRememberMe_SetsPersistentCookie:
       POST /api/auth/login with rememberMe=true
       -> Cookie has Max-Age of ~30 days (2592000 seconds)

    3. Login_WithoutRememberMe_SetsSessionCookie:
       POST /api/auth/login with rememberMe=false
       -> Cookie does NOT have Max-Age (session cookie)

    4. Refresh_WithCookie_ReturnsNewAccessToken:
       POST /api/auth/refresh with refresh_token cookie (no request body)
       -> Returns 200 with new accessToken, expiresAt, user
       -> Sets new refresh_token cookie
       -> JSON body does NOT contain refreshToken

    5. Refresh_WithoutCookie_Returns401:
       POST /api/auth/refresh with no cookie
       -> Returns 401

    6. Logout_ClearsRefreshTokenCookie:
       POST /api/auth/logout (authenticated)
       -> Response clears the refresh_token cookie (expired Set-Cookie)

    7. Refresh_PreservesRememberMe_OnRotation:
       Login with rememberMe=true, then refresh
       -> New cookie still has Max-Age (persistent)

    Implementation approach (GREEN phase):

    1. Add RememberMe bool property to RefreshToken entity
    2. Configure RememberMe column in RefreshTokenConfiguration
    3. Create EF Core migration for new column
    4. Update RefreshTokenHandler to accept and propagate RememberMe
    5. Create private static SetRefreshTokenCookie helper in AuthApiEndpoints
    6. Modify login endpoint: extract refreshToken from handler result, set as cookie, return response without refreshToken
    7. Modify refresh endpoint: read cookie, pass to handler, set new cookie with preserved RememberMe, return response without refreshToken
    8. Modify logout endpoint: clear cookie after handler execution
    9. Make Secure flag environment-aware: false in Development, true in Production
  </behavior>
  <implementation>
    The key architectural insight is: keep handler logic unchanged. The LoginHandler and RefreshTokenHandler still generate and return refresh tokens in their response records. The Presentation layer (AuthApiEndpoints.cs) is where the cookie logic lives -- it extracts the refresh token from the handler result, sets it as a cookie, and returns a projected response without the refresh token field.

    For RememberMe tracking across rotations: add a RememberMe bool to the RefreshToken entity. The LoginHandler sets it from the command. The RefreshTokenHandler copies it from the old token to the new one. The endpoint reads it to decide cookie Max-Age.

    CookieOptions configuration:
    - HttpOnly = true (cannot be read by JavaScript)
    - Secure = !app.Environment.IsDevelopment() (HTTPS-only in prod, HTTP ok in dev)
    - SameSite = SameSiteMode.Lax (CSRF protection for POST endpoints)
    - Path = "/api/auth" (only sent to auth endpoints)
    - MaxAge = rememberMe ? TimeSpan.FromDays(30) : null (session cookie vs persistent)

    For the CustomWebApplicationFactory: use in-memory database (or Testcontainers if already set up) with seeded test user. Tests call real HTTP endpoints via HttpClient and inspect Set-Cookie headers.
  </implementation>
</feature>
