# Phase 05.1: Fix Architecture Test Failures from Prior Phases - Research

**Researched:** 2026-03-06
**Domain:** .NET Clean Architecture / NetArchTest enforcement in modular monolith
**Confidence:** HIGH

## Summary

This phase addresses 5 failing architecture tests across 3 test classes in `backend/tests/Ganka28.ArchitectureTests/`. The failures stem from 4 root causes introduced as conscious shortcuts during Phases 02, 03, and 05. All fixes are mechanical refactoring -- no new features, no behavioral changes, no database migrations.

The fixes involve: (1) duplicating 3 Patient.Domain.Enums into Patient.Contracts with explicit mapper extension methods, (2) extracting an `IReferenceDataRepository` interface into Shared.Application to replace direct ReferenceDbContext usage in Clinical.Application, (3) moving the `FieldChange` record from Clinical.Domain.Entities to Clinical.Domain.ValueObjects namespace, and (4) moving `PatientFieldValidationResult` + `MissingFieldInfo` records from Patient.Domain.Services to Patient.Contracts.

**Primary recommendation:** Fix each root cause in a separate plan (4 plans total, plus a verification plan), keeping changes small and independently testable. The enum duplication in Patient is the largest change (touches 5 Contracts DTOs + Application handlers + mapper creation).

### Pattern 1: Enum Duplication with Explicit Mappers
**What:** Each module's Contracts layer owns its own enum definitions, identical in values but independent of Domain.
**When to use:** When Contracts DTOs need enum types but Contracts must not reference Domain (architecture rule).
**Example:**
```csharp
// Patient.Contracts/Enums/Gender.cs
namespace Patient.Contracts.Enums;

public enum Gender
{
    Male = 0,
    Female = 1,
    Other = 2
}

// Patient.Contracts/Mappers/EnumMappers.cs
namespace Patient.Contracts.Mappers;

public static class EnumMappers
{
    public static Contracts.Enums.Gender ToContractEnum(this Domain.Enums.Gender domainEnum)
        => (Contracts.Enums.Gender)(int)domainEnum;

    public static Domain.Enums.Gender ToDomainEnum(this Contracts.Enums.Gender contractEnum)
        => (Domain.Enums.Gender)(int)contractEnum;

    // ... same for PatientType, AllergySeverity
}
```

**Where mappers live:** The mapper extension methods must live in a project that references BOTH Domain and Contracts. This means they belong in **Patient.Application** (not Patient.Contracts, which cannot reference Domain). The Contracts project only contains the enum definitions.

### Pattern 2: Repository Interface Abstraction for Cross-Module Data
**What:** Define repository interface in Application layer, implementation in Infrastructure.
**When to use:** When Application handlers need data access but must not reference Infrastructure directly.
**Example:**
```csharp
// Shared.Application/Interfaces/IReferenceDataRepository.cs
namespace Shared.Application.Interfaces;

public interface IReferenceDataRepository
{
    Task<List<Icd10Code>> SearchAsync(string term, int limit, CancellationToken ct);
    Task<List<Icd10Code>> GetByCodesAsync(IReadOnlyCollection<string> codes, CancellationToken ct);
}

// Shared.Infrastructure/Repositories/ReferenceDataRepository.cs
namespace Shared.Infrastructure.Repositories;

public class ReferenceDataRepository : IReferenceDataRepository
{
    private readonly ReferenceDbContext _context;
    // ... wraps existing queries
}
```

### Pattern 3: Value Object Namespace Convention
**What:** Immutable data carrier records belong in `*.Domain.ValueObjects`, not `*.Domain.Entities`.
**When to use:** The architecture test scans `*.Domain.Entities` for private-setter enforcement. Value objects (records) have public init properties by nature.
**Why it matters:** The `All_Domain_Entities_Should_Have_Private_Setters` test specifically scans the `*.Domain.Entities` namespace.

### Anti-Patterns to Avoid
- **Relaxing arch tests instead of fixing code:** Tests exist for a reason -- fix the violations, don't weaken the rules
- **Putting mappers in Contracts:** Contracts cannot reference Domain, so mapper extension methods that convert between Domain and Contract types cannot live in Contracts
- **Leaving stale project references:** After removing `using Shared.Infrastructure` from Clinical.Application handlers, also remove the `<ProjectReference>` from the `.csproj` file
- **Forgetting to update unit tests:** `SearchIcd10CodesHandlerTests.cs` directly instantiates ReferenceDbContext -- must update to use IReferenceDataRepository

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Enum conversion | Custom reflection-based converter | Direct int cast `(TargetEnum)(int)sourceEnum` | Enum values are identical by design; int cast is compile-time safe and zero-cost |
| Reference data access | Ad-hoc query wrappers | Repository pattern with interface in Application | Follows established project pattern (IVisitRepository, IPatientRepository, etc.) |

## Common Pitfalls

### Pitfall 1: Mapper Placement Error
**What goes wrong:** Putting enum mapper extension methods in Patient.Contracts, which cannot reference Patient.Domain.
**Why it happens:** Mappers feel like they "belong" with the Contracts enums.
**How to avoid:** Mappers go in Patient.Application (or a separate Patient.Contracts.Mappers assembly, but Application is simpler). Patient.Application already references both Domain and Contracts.
**Warning signs:** Build error: "Patient.Contracts cannot reference Patient.Domain"

### Pitfall 2: Forgetting Downstream Consumers of Moved Types
**What goes wrong:** Moving PatientFieldValidationResult to Contracts but forgetting to update Patient.Application.Features.ValidatePatientFields which uses it.
**Why it happens:** Only checking the Presentation layer (where the arch test failure was reported).
**How to avoid:** Grep for all usages of the type across the entire codebase before moving.
**Affected files:**
- `Patient.Domain.Services.PatientFieldValidator.cs` (returns the type -- keep using it via new namespace)
- `Patient.Application.Features.ValidatePatientFields.cs` (handler return type)
- `Patient.Presentation.PatientApiEndpoints.cs` (bus.InvokeAsync generic type)

### Pitfall 3: Not Updating Unit Tests
**What goes wrong:** Backend compiles and arch tests pass, but unit tests in `Clinical.Unit.Tests` fail because they still reference ReferenceDbContext directly.
**Why it happens:** Unit test file `SearchIcd10CodesHandlerTests.cs` creates an in-memory ReferenceDbContext and passes it to handler.
**How to avoid:** After changing handler signature to accept IReferenceDataRepository, update the test to use a mock/substitute.

### Pitfall 4: Clinical.Contracts Still References Clinical.Domain in csproj
**What goes wrong:** Clinical.Contracts.csproj has `<ProjectReference Include="..\Clinical.Domain\Clinical.Domain.csproj" />` but no code uses it. The arch test currently passes because no Contracts DTO uses a Clinical.Domain type, but the stale reference is a landmine.
**Why it happens:** Added during Phase 03 scaffolding and never cleaned up.
**How to avoid:** Remove the unused project reference from Clinical.Contracts.csproj as part of cleanup.

### Pitfall 5: FieldChange Namespace Affects Serialization
**What goes wrong:** Moving FieldChange from `Clinical.Domain.Entities` to `Clinical.Domain.ValueObjects` could affect JSON serialization if the fully-qualified type name is stored.
**Why it happens:** VisitAmendment.FieldChangesJson stores JSON-serialized FieldChange records.
**How to avoid:** JSON serialization uses property names, not namespace/type names. The `record FieldChange(string FieldName, string? OldValue, string? NewValue)` definition is identical -- only the namespace changes. System.Text.Json and Newtonsoft.Json both serialize by property name, not by assembly-qualified name. This is safe.

### Pitfall 6: Frontend Impact Assessment
**What goes wrong:** Assuming backend enum changes require frontend changes.
**Why it happens:** Concern about API contract changes.
**How to avoid:** The frontend already receives enums as integers (0, 1, 2) and maps them client-side. The Contracts enum duplication changes the C# type used in DTOs but the serialized JSON remains identical integers. Zero frontend impact.

2. **Should Application-layer handlers use Domain or Contracts enum types internally?**
   - What we know: Application layer references both Domain and Contracts. Handlers receive Contracts commands (with Contracts enums) and interact with Domain entities (with Domain enums).
   - What's unclear: Whether validators (e.g., RegisterPatientCommandValidator) should compare against Domain or Contracts enum values.
   - Recommendation: Validators validate Contracts types (since commands use Contracts enums). Domain conversion happens after validation in the handler. The validator's `x.PatientType == PatientType.Medical` should use `Patient.Contracts.Enums.PatientType.Medical`.

   - What's unclear: Whether there are test helpers or shared fixtures that reference ReferenceDbContext.
