## Resolution

root_cause: |
  TWO ISSUES causing DbUpdateConcurrencyException:

  **Primary Issue: `context.Invoices.Update(invoice)` in InvoiceRepository.Update()**

  The repository's Update method (InvoiceRepository.cs line 94-97) calls `context.Invoices.Update(invoice)`.
  When called on an already-tracked entity (loaded via GetByIdAsync in the same DbContext scope),
  `DbContext.Update()` marks EVERY property as modified - including RowVersion.

  For SQL Server's `rowversion`/`timestamp` column type, the value is auto-generated by the database
  on every UPDATE. EF Core's `.IsRowVersion()` configuration tells EF to:
  1. Include RowVersion in the WHERE clause (for optimistic concurrency check)
  2. Read back the new value after UPDATE

  But when `Update()` marks RowVersion as modified, EF also includes it in the SET clause of the
  UPDATE statement. This creates a conflict: EF tries to SET RowVersion to the old value while
  SQL Server expects to auto-generate a new one. The WHERE clause check against the old RowVersion
  may also fail because the row was just inserted with a DB-generated value that differs from the
  empty byte[] default.

  **Secondary Issue: Handlers inconsistently use Update() vs change tracking**

  Some handlers (AddInvoiceLineItem, RemoveInvoiceLineItem, FinalizeInvoice) rely on EF change
  tracking (no explicit Update call). Others (RecordPayment, ApplyDiscount, ApproveDiscount,
  RejectDiscount, RequestRefund, ApproveRefund, ProcessRefund) explicitly call
  invoiceRepository.Update(invoice). The explicit Update() call is both unnecessary (entity is
  already tracked) and harmful (marks RowVersion as modified).
